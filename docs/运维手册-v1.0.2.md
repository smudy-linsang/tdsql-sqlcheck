## 1. 日常运维速查

| 操作 | 命令 |
|---|---|
| 启动 / 停止 / 重启 | `systemctl start|stop|restart tdsql-sqlcheck` |
| 查看状态 | `systemctl status tdsql-sqlcheck` |
| 实时日志 | `journalctl -u tdsql-sqlcheck -f` |
| 最近 200 行日志 | `journalctl -u tdsql-sqlcheck -n 200 --no-pager` |
| 健康检查 | `curl -s http://127.0.0.1:8000/health`（期望 `{"status":"ok","version":"1.0.2"}`） |
| 全面冒烟 | `bash <发布包>/deploy/verify_deploy.sh --port 8000` |
| 修改配置 | `vi /opt/tdsql-sqlcheck/.env` → `systemctl restart tdsql-sqlcheck` |
| 回滚版本 | `bash <发布包>/deploy/rollback.sh` |

安装布局：`/opt/tdsql-sqlcheck/{current -> releases/vX.Y.Z, releases/, logs/, reports/, .env}`；服务运行账号 `sqlcheck`（nologin）。

## 2. 日志管理

- 应用日志走 **journald**（`SyslogIdentifier=tdsql-sqlcheck`）。
- 持久化与容量：确认 `/etc/systemd/journald.conf` 设置 `Storage=persistent`、`SystemMaxUse=2G`（麒麟默认易失，改后 `systemctl restart systemd-journald`）。
- 归档导出（如接行内日志平台）：`journalctl -u tdsql-sqlcheck --since today -o json > /var/log/sqlcheck-$(date +%F).json`，或配置 rsyslog 按 identifier 分流。
- **业务审计日志**（谁在系统里做了什么）在库表 `operation_logs`，经界面「系统管理→操作审计」查询，保留天数由数据保留策略控制。

## 3. 监控告警

### 3.1 Prometheus 接入（推荐）

- 抓取端点：`http://<应用IP>:8000/metrics`（Nginx 场景走 443 `/metrics`，注意 conf 中的网段白名单）。
- 关键指标与建议告警：

| 指标 | 含义 | 告警建议 |
|---|---|---|
| `up{job="sqlcheck"}` | 实例存活 | ≠1 持续 2 分钟 → 严重 |
| `tdsql_scan_tasks_total{status="failed"}` | 扫描失败累计 | 5 分钟增量 >0 → 警告 |
| 进程级 CPU/RSS（node_exporter） | 资源 | CPU>80% 或 RSS>70%内存 持续10分钟 → 警告 |

### 3.2 无 Prometheus 时的最简探活

crontab 每分钟：`curl -sf -m 5 http://127.0.0.1:8000/health || logger -t sqlcheck-probe "health FAILED"`（对接行内短信/告警脚本）。

### 3.3 系统内置监控

界面「平台治理→监控告警」提供业务级告警（指标阈值/确认闭环），顶栏铃铛联动，属应用功能，与基础设施监控互补。

## 4. 备份与恢复

| 对象 | 内容 | 方式 | 频率 |
|---|---|---|---|
| **元数据库** `tdsql_sqlcheck` | 用户/项目/规则集/连接配置(加密)/慢SQL/审核历史/审计日志 | 纳入 TDSQL 实例行内备份（mysqldump 亦可：`mysqldump -h.. --single-transaction tdsql_sqlcheck > bak.sql`） | 每日 |
| **配置文件** `/opt/tdsql-sqlcheck/.env` | 数据库口令、**AUTH_SECRET_KEY** | 随配置管理/离线保管 | 每次变更 |
| 发布包 tar.gz | 可重建一切代码与依赖 | 介质库留存 | 每版本 |
| `reports/` 导出报告 | 可再生成 | 可不备份 | — |

**恢复演练口径**：新机按部署手册装 v1.0.2 → 恢复 .env → 元数据库还原 → 启动 → verify_deploy.sh 全 PASS。RTO≈30 分钟。

> ⚠️ `AUTH_SECRET_KEY` 丢失=全员令牌失效（重新登录即可，无数据损失）；元数据库中**目标实例连接口令为加密存储**，其加密亦依赖应用密钥体系，故 .env 必须与库备份配对保管。

## 5. 数据保留与容量

- 系统内置保留策略（界面「系统管理→数据保留」）+ 每日定时清理（`RETENTION_CRON_HOUR`，默认凌晨3点），支持手工「立即清理」。
- 建议基线：慢SQL记录 90 天 / 审核历史 180 天 / 操作审计 365 天（满足一般行内审计要求，按合规调整）。
- 容量观测：`SELECT table_name, ROUND((data_length+index_length)/1024/1024) AS mb FROM information_schema.tables WHERE table_schema='tdsql_sqlcheck' ORDER BY mb DESC;`

## 6. 安全运维

- 账号治理：admin 仅留 1-2 人；连续 5 次失败锁 15 分钟，解锁在「用户管理→解锁」；离职账号及时禁用。
- 口令策略：≥8 位、含大小写/数字/特殊字符三类；首登强制改密（系统内置强制）。
- 补丁：仅通过**新版本发布包**升级（同一 install.sh 流程，自动落 releases/ 新目录，可随时回滚），禁止在服务器上手改代码。
- 麒麟加固：保持 kysec/SELinux 默认策略即可（应用目录独立、服务 NoNewPrivileges/ProtectSystem 已在 unit 内启用）。

## 7. 常见故障处置

| 现象 | 排查 | 处置 |
|---|---|---|
| 健康检查失败/服务频繁重启 | `journalctl -u tdsql-sqlcheck -n 100` | 多为元数据库不可达：`mysql -h$SQLCHECK_DB_HOST -P$PORT -u$USER -p` 手工验证；网络恢复后服务自动重连 |
| 登录慢/超时 | 元数据库延迟 | 检查 TDSQL 实例负载与网络 RTT |
| 扫描一直"busy" | 并发限流触发 | 属保护机制；确需提高并发调 `.env` `MAX_CONCURRENT_SCANS_*` 后重启 |
| 定时扫描没跑 | 界面「扫描计划」确认 enabled；`journalctl` 搜 scheduler | 多实例部署时仅 leader 执行（租约机制），单机部署确认 `SCHEDULER_ENABLED=true` |
| 磁盘告警 | journald 上限 2G + reports/ 清理 + 第 5 节容量 SQL | 手工清 reports/ 旧文件安全 |
| CPU 突高 | 大文件审核/大扫描窗口 | 属短时峰值；持续高查 `/metrics` 与操作审计定位来源用户 |

## 8. 升级流程（后续版本通用）

1. 介质校验 → 解压新版本包；
2. 复用现网 `.env`（install.sh 自动保留，新模板落 `.env.new` 供比对新增项）；
3. `sudo ./deploy/install.sh`（落新 releases/ 目录并切换，元数据库结构自动增量迁移）；
4. `verify_deploy.sh` 全 PASS；异常即 `rollback.sh`（分钟级）。

变更窗口建议：业务低峰；升级全程服务中断 < 1 分钟（systemd restart）。
