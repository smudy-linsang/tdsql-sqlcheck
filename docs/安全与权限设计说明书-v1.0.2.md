## 一、安全架构总览

```
请求 ─▶ RequestContextMiddleware（请求ID/日志/指标）
      ─▶ AuthMiddleware
           ├── 免认证路径? ──是──▶ 放行（/health /metrics /login /static 前端 webhook*）
           ├── 提取 Authorization: Bearer <token>
           ├── 验签+有效期 ──失败──▶ 401
           ├── 用户状态校验（active?）──失败──▶ 401
           ├── RBAC矩阵 check_permission(role, method, path) ──拒绝──▶ 403 + 指标
           └── 放行 ─▶ 业务处理 ─▶ 变更操作写 operation_logs
```

\* GitLab Webhook 使用独立的 Secret Token 机制鉴权（hmac常量时间比较），不走用户令牌。

## 二、身份认证

### 2.1 口令存储

- 算法：PBKDF2-HMAC-SHA256，**240,000 轮**，每用户独立 16 字节随机盐
- 验证：`hmac.compare_digest` 常量时间比较，防时序攻击
- 强度策略：长度≥8，大写/小写/数字/特殊字符**至少三类**
- 生命周期：新建/重置口令强制下次登录修改（`must_change_password`）

### 2.2 访问令牌

- 格式：`base64url(payload).base64url(HMAC-SHA256签名)`，自包含无状态
- Payload：`{sub, role, iat, exp, jti}`；有效期默认 8 小时（`AUTH_TOKEN_TTL_HOURS`）
- 签名密钥：`AUTH_SECRET_KEY` 环境变量 → `data/auth_secret.key`（自动生成，0600）
- 失效机制：过期自失效；**用户禁用即时生效**（每请求查用户状态，30秒进程内缓存）；
  多副本部署必须注入统一 `AUTH_SECRET_KEY`

### 2.3 登录保护

- 连续失败锁定：默认 5 次锁 15 分钟（`AUTH_MAX_LOGIN_FAILURES` / `AUTH_LOCK_MINUTES`）
- 全部登录成败写审计日志（含来源IP）并计入 `tdsql_login_total` 指标
- 管理员可解锁（`POST /api/v1/auth/users/{u}/unlock`）

### 2.4 初始账户引导

首次启动且用户表为空时创建 `admin`：
口令取 `ADMIN_INITIAL_PASSWORD`；未配置则生成随机口令**打印日志一次**并强制首登修改。

## 三、RBAC 权限模型

### 3.1 角色定义

| 角色 | 定位 | 典型岗位 |
|------|------|----------|
| admin | 系统管理员 | 平台运营，负责用户开通与权限回收 |
| dba | 数据库管理员 | 连接接入、规则集/门禁定制、扫描治理 |
| developer | 开发人员 | 自助SQL审核、EXPLAIN分析 |
| auditor | 审计员 | 合规审计，全局只读 |

### 3.2 权限矩阵（`check_permission(role, method, path)`）

| 资源 | admin | dba | developer | auditor |
|------|:-:|:-:|:-:|:-:|
| 用户管理 `/api/v1/auth/users*` | 读写 | ✗ | ✗ | ✗ |
| 自助（改口令/登出） | ✓ | ✓ | ✓ | ✓ |
| SQL审核 `/api/v1/audit/*`、GitLab diff审核、EXPLAIN分析 | 读写 | 读写 | **读写** | 只读 |
| 连接管理/扫描/扫描计划 `/api/v1/tdsql/*` | 读写 | 读写 | 只读 | 只读 |
| 规则集/门禁/保留策略/项目/治理 | 读写 | 读写 | 只读 | 只读 |
| 操作审计日志 `/api/v1/admin/operation-logs` | 读 | 读 | 读 | **读** |

判定规则实现要点：
1. `admin-only` 前缀优先判定（用户管理）
2. `GET/HEAD/OPTIONS` 视为只读，所有角色放行（除 admin-only）
3. developer 的写白名单为显式前缀列表（审核/分析类自助操作）
4. 拒绝计入 `tdsql_rbac_denied_total{role}` 指标

### 3.3 账户保护规则

- 系统必须保留至少一个 active 的 admin（禁删/禁降级/禁禁用最后一个）
- 不能删除当前登录账户

## 四、数据安全

### 4.1 密钥管理

| 密钥 | 用途 | 来源优先级 |
|------|------|-----------|
| `TDSQL_ENCRYPTION_KEY` | 连接密码 Fernet 加密 | 环境变量(KMS) → data/encryption.key 自动生成 |
| `AUTH_SECRET_KEY` | 令牌签名 | 环境变量(KMS) → data/auth_secret.key 自动生成 |

- 密钥文件权限 0600，`.gitignore` 排除，**丢失加密密钥将无法解密已存连接密码，必须备份**
- V1.0 硬编码种子密钥仅保留解密回退（兼容历史数据），解密命中时日志提示重新保存以完成密钥升级

### 4.2 敏感数据脱敏

- 慢SQL入库脱敏（默认开）：字面量→`?`，防止 WHERE 条件中的身份证/卡号/手机号落地
- 连接密码：加密存储，任何接口只返回 `***`
- 用户接口：永不返回 `password_hash`/`salt`

### 4.3 传输与边界

- CORS 默认不下发跨域头（同源部署）；跨域需显式配置 `CORS_ALLOW_ORIGINS`
- `/metrics` 免认证（供Prometheus抓取），生产应以网络策略限制抓取来源
- API 文档 `/docs` 可通过 `DOCS_PUBLIC=false` 收入认证边界
- TLS 终结建议在行内负载均衡/网关层完成

### 4.4 目标库账号最小权限

平台连接目标 TDSQL 的账号建议仅授予：

```sql
GRANT SELECT ON information_schema.* TO 'sqlcheck_ro'@'%';
GRANT SELECT ON performance_schema.* TO 'sqlcheck_ro'@'%';
GRANT PROCESS ON *.* TO 'sqlcheck_ro'@'%';          -- processlist
GRANT SELECT ON <业务库>.* TO 'sqlcheck_ro'@'%';     -- EXPLAIN/元数据
```

## 五、操作审计

| 事件 | 记录位置 | 内容 |
|------|----------|------|
| 登录成功/失败/锁定 | operation_logs | 用户、IP、失败原因 |
| 全部变更API（POST/PUT/DELETE） | operation_logs | 用户、方法+路径、状态码、IP、UA |
| 用户管理（建号/重置/禁用/删除） | operation_logs | 操作者与目标用户 |
| SQL审核 | audit_history.created_by | 审核发起人 |
| 连接/规则集/保留策略变更 | operation_logs | 操作者与目标对象 |

查询：`GET /api/v1/admin/operation-logs?operator=&operation_type=`（支持分页），
auditor 角色可读，满足"事后可追溯到人"的内审要求。
日志保留期由 retention_policies 管理（默认365天，可按行内规定调整）。

## 六、与行内基础设施的集成预留

- **统一认证（4A/AD/LDAP/OAuth2）**：`auth_service.authenticate()` 为唯一认证入口，
  接入行内认证时在此扩展外部校验并本地映射角色，令牌机制不变
- **KMS**：密钥统一从环境变量注入，由部署平台/配置中心对接 KMS
- **SIEM**：operation_logs 结构化存储，可由采集器同步至行内日志平台
