## 2. C3′ — MetadataFetcher 抽取（F1 使能项）

### 2.1 设计原则

M 稿 C3 要把 71KB 的 `tdsql_connector.py` 拆 5 个模块。**本版只抽出 F1 直接依赖的元数据采集**，其余不动，把风险和工作量压到最小：

- 新建 `backend/services/connector/metadata_fetcher.py`，内部持有一个 `TDSQLConnectionPool` 引用，通过 `pool._execute(sql, params)` 复用现有连接池与超时；
- **不改** `tdsql_connector.py` 任何现有方法签名（`get_tables/get_table_metadata/get_indexes` 原样保留）；
- `connector/__init__.py` 仅做 `from ..tdsql_connector import TDSQLConnectionPool` 兼容重导出，为将来完整拆分留位，但本版不强推。

### 2.2 `connector/metadata_fetcher.py`

```python
# backend/services/connector/metadata_fetcher.py
"""全量对象元数据采集器（只读）。薄封装 pool._execute，供 F1 反向审核使用。"""
from dataclasses import dataclass, field

# 系统库排除口径与 tdsql_connector._BIGTABLE_SYS_DBS / schema_inspector 保持一致
SYSTEM_SCHEMAS = ("__tencentdb__", "information_schema", "mysql",
                  "performance_schema", "query_rewrite", "sys", "sysdb", "test", "xa")

@dataclass
class DbObject:
    schema: str
    obj_type: str          # table | view | trigger | procedure | function | event
    name: str
    create_sql: str = ""   # 反向 DDL（SHOW CREATE 原文，未规范化）
    extra: dict = field(default_factory=dict)   # 依赖/分片键/错误标注等
    error: str = ""        # 采集失败原因（best-effort，不中断整体）

class MetadataFetcher:
    def __init__(self, pool):
        self.pool = pool

    # ── 库枚举 ──
    def list_databases(self, include: list[str] = None) -> list[str]:
        rows = self.pool._execute(
            "SELECT SCHEMA_NAME FROM information_schema.SCHEMATA "
            "WHERE SCHEMA_NAME NOT IN ({}) ORDER BY SCHEMA_NAME".format(
                ",".join(["%s"] * len(SYSTEM_SCHEMAS))), tuple(SYSTEM_SCHEMAS))
        dbs = [r["SCHEMA_NAME"] for r in rows]
        return [d for d in dbs if (not include or d in include)]

    # ── 表 ──（索引/FK/PK 内嵌于 SHOW CREATE TABLE）
    def fetch_tables(self, db: str) -> list[DbObject]:
        names = self.pool._execute(
            "SELECT TABLE_NAME FROM information_schema.TABLES "
            "WHERE TABLE_SCHEMA=%s AND TABLE_TYPE='BASE TABLE' ORDER BY TABLE_NAME", (db,))
        out = []
        for r in names:
            t = r["TABLE_NAME"]; o = DbObject(db, "table", t)
            try:
                res = self.pool._execute(f"SHOW CREATE TABLE `{db}`.`{t}`")
                o.create_sql = res[0].get("Create Table", "") if res else ""
                o.extra["shard_key"] = self._shard_key(o.create_sql)   # 复用现有分片键解析
            except Exception as e:
                o.error = str(e)
            out.append(o)
        return out

    # ── 视图 ──
    def fetch_views(self, db: str) -> list[DbObject]:
        names = self.pool._execute(
            "SELECT TABLE_NAME FROM information_schema.VIEWS WHERE TABLE_SCHEMA=%s "
            "ORDER BY TABLE_NAME", (db,))
        out = []
        for r in names:
            v = r["TABLE_NAME"]; o = DbObject(db, "view", v)
            try:
                res = self.pool._execute(f"SHOW CREATE VIEW `{db}`.`{v}`")
                o.create_sql = res[0].get("Create View", "") if res else ""
            except Exception as e:
                o.error = str(e)
            out.append(o)
        return out

    # ── 触发器 / 存储过程 / 函数 / 事件 —— best-effort（分布式实例可能不支持）──
    def fetch_triggers(self, db: str) -> list[DbObject]: ...   # SHOW CREATE TRIGGER
    def fetch_routines(self, db: str) -> list[DbObject]: ...   # ROUTINES → SHOW CREATE PROCEDURE/FUNCTION
    def fetch_events(self, db: str) -> list[DbObject]:   ...   # EVENTS   → SHOW CREATE EVENT

    def _shard_key(self, create_sql: str) -> str:
        # 复用 tdsql_connector 已有的 shardkey-from-DDL 正则，避免重复实现
        from backend.services.tdsql_connector import parse_shard_key_from_ddl
        try: return parse_shard_key_from_ddl(create_sql)
        except Exception: return ""
```

> 触发器/过程/事件的 `fetch_*` 与表/视图同构：`information_schema` 枚举名 → 逐对象 `SHOW CREATE` → 失败写 `o.error` 不抛。TDSQL 分布式实例对这些对象若返回错误/空，标注"该实例类型不支持或无权限"，整体流程继续。

## 3. F1 — 实况库反向审核

### 3.1 数据结构：`engine/schema_snapshot.py`

```python
# backend/engine/schema_snapshot.py
import hashlib, re
from dataclasses import dataclass, field

# DDL 规范化：剥离易变量，保证同结构 → 同文本 → 同 sha256
_RE_AUTOINC   = re.compile(r"\s*AUTO_INCREMENT=\d+", re.I)
_RE_TRAILING  = re.compile(r"[ \t]+$", re.M)

def normalize_ddl(create_sql: str, obj_type: str) -> str:
    s = create_sql or ""
    if obj_type == "table":
        s = _RE_AUTOINC.sub("", s)          # 去自增计数（易变）
    s = _RE_TRAILING.sub("", s)             # 去行尾空白
    s = s.replace("\r\n", "\n").strip()
    return s + ";\n"

@dataclass
class SchemaSnapshot:
    connection_id: str
    databases: list          # 采集的库
    object_types: list       # table/view/trigger/procedure/function/event
    objects: list = field(default_factory=list)   # list[DbObject]
    masked: bool = False

    def render_ddl(self) -> str:
        """确定性拼装：库 → 类型(固定序) → 名称 字典序。带分节注释。"""
        TYPE_ORDER = {"table":0,"view":1,"trigger":2,"function":3,"procedure":4,"event":5}
        objs = sorted(self.objects, key=lambda o: (o.schema, TYPE_ORDER.get(o.obj_type,9), o.name))
        buf = ["-- TDSQL-SQLCheck 实况元数据快照", f"-- databases: {','.join(self.databases)}",
               f"-- object_types: {','.join(self.object_types)}",
               f"-- masked: {self.masked}", ""]
        cur_db = None
        for o in objs:
            if o.schema != cur_db:
                cur_db = o.schema; buf.append(f"\n-- ===== database: {cur_db} =====")
            if o.error:
                buf.append(f"-- [SKIP] {o.obj_type} {o.name}: {o.error}"); continue
            body = _mask(o.create_sql, o.obj_type) if self.masked else o.create_sql
            buf.append(f"\n-- [{o.obj_type}] {o.schema}.{o.name}"
                       + (f"  (shard_key={o.extra['shard_key']})" if o.extra.get('shard_key') else ""))
            buf.append(normalize_ddl(body, o.obj_type))
        return "\n".join(buf)

    def counts(self) -> dict:
        c = {}
        for o in self.objects: c[o.obj_type] = c.get(o.obj_type, 0) + 1
        return c

    def sha256(self) -> str:
        return hashlib.sha256(self.render_ddl().encode("utf-8")).hexdigest()

def _mask(create_sql: str, obj_type: str) -> str:
    """过程/函数/触发器体可能含业务逻辑与常量 → 可选脱敏（仅保留签名/头部）。"""
    if obj_type in ("procedure","function","trigger"):
        head = create_sql.split("BEGIN")[0]
        return head + "BEGIN /* body masked */ END"
    return create_sql
```

> **确定性要点**：排序固定 + 去 `AUTO_INCREMENT` + 去行尾空白 + 统一换行 ⇒ 库结构不变时 `sha256` 恒定。这使"库有没有变过""两环境结构是否一致"变成一次指纹比较。

### 3.2 编排服务：`services/live_schema_service.py`

```python
# backend/services/live_schema_service.py
import os, json, datetime, threading
from backend.config import SNAPSHOT_DIR, LIVE_SCHEMA_MAX_CONCURRENCY
from backend.services.database import _get_connection, ensure_db, log_operation
from backend.services.connector.metadata_fetcher import MetadataFetcher
from backend.engine.schema_snapshot import SchemaSnapshot
from backend.engine.checker import RuleChecker
from backend.engine.schema_inspector import SchemaInspector

_sema = threading.BoundedSemaphore(LIVE_SCHEMA_MAX_CONCURRENCY)   # 单进程并发保护

class LiveSchemaService:
    # ① 采集 → 落盘 → 台账（供 API 异步任务调用）
    def create_snapshot(self, pool, connection_id, databases, object_types,
                        masked, created_by) -> dict:
        with _sema:
            fetcher = MetadataFetcher(pool)
            dbs = fetcher.list_databases(include=databases or None)
            snap = SchemaSnapshot(connection_id, dbs, object_types, masked=masked)
            for db in dbs:
                if "table" in object_types:     snap.objects += fetcher.fetch_tables(db)
                if "view" in object_types:      snap.objects += fetcher.fetch_views(db)
                if "trigger" in object_types:   snap.objects += fetcher.fetch_triggers(db)
                if "procedure" in object_types or "function" in object_types:
                                                snap.objects += fetcher.fetch_routines(db)
                if "event" in object_types:     snap.objects += fetcher.fetch_events(db)
            ddl = snap.render_ddl(); digest = snap.sha256()
            os.makedirs(SNAPSHOT_DIR, exist_ok=True)
            ts = datetime.datetime.now().strftime("%Y%m%d_%H%M%S")
            path = os.path.join(SNAPSHOT_DIR, f"{connection_id}_{ts}_{digest[:8]}.sql")
            with open(path, "w", encoding="utf-8") as f: f.write(ddl)
            return self._save_row(connection_id, snap, digest, path, len(ddl.encode()), created_by)

    # ③ 审核：把快照 DDL 丢进现有 119 规则引擎 + 12 项 Schema 检查
    def audit_snapshot(self, snapshot_id, ruleset_id="live_schema", pool=None) -> dict:
        row = self._get_row(snapshot_id)
        content = open(row["file_path"], encoding="utf-8").read()
        checker = RuleChecker()
        overrides = self._overrides(ruleset_id)                 # 复用规则集多租户
        results = checker.audit_file(content, file_path="live_snapshot.sql",
                                     rule_overrides=overrides)   # ← 直接复用
        report = self._merge_by_object(results)                 # 违规按 库.对象 归并
        # 元数据级补充（不需 DDL 文本的检查，如整表字符集/行数）
        if pool is not None:
            report["schema_inspector"] = SchemaInspector().inspect(pool)
        self._save_audit_history(snapshot_id, report)           # source='live_schema'
        return report

    # F2 漂移对比：两份快照对象级 added/removed/changed（按 normalize 后文本比对）
    def diff_snapshots(self, base_id, target_id) -> dict: ...

    # 台账 CRUD（schema_snapshots 表）
    def _save_row(self, ...): ...
    def _get_row(self, sid): ...
    def list_snapshots(self, connection_id=""): ...
```

### 3.3 审核阶段为什么"零新引擎"

`checker.audit_file(content)` 已经在"文件审核"里做了"按 `;` 拆分 → 逐条 `audit_sql` → 119 规则"。F1 只是把 `content` 从"人给的文件"换成"我们反向工程出来的快照"——**审核引擎一行不用改**。这就是概要里说的"70% 是编排"。

### 3.4 `live_schema` 规则档（关键校准）

反向 DDL 是 MySQL 规范化输出（必有反引号、显式 ENGINE/charset），一批"作者期"规则会恒过、产生噪声。用现有"规则集多租户"落一个**内置规则集 `live_schema`**：

- 在 `services/database.py` 的 seed 阶段插入 `rule_sets(id='live_schema', name='实况库档', is_builtin=1)`；
- `rule_set_items` 对"仅作者期有意义"的规则置 `enabled=0`（如"未显式指定引擎/字符集"类，反向 DDL 必带故无意义），对结构类规则保持启用并可调级别：命名规范、缺主键、冗余/缺失索引、超大字段(如 `VARCHAR(5000)`/`TEXT` 滥用)、字符集不一致、缺注释、分片键相关、危险类型等；
- `audit_snapshot` 默认用 `ruleset_id='live_schema'`，也允许调用方传项目自有规则集。

> 具体哪几条 `enabled=0` 在实现期用真实反向 DDL 跑一遍"降噪校准"确定，写入 seed；此处给机制，不写死清单。

- RBAC：本组路径映射菜单键 `live-schema`（见 C5）。

## 4. F2 — Schema 漂移对比

### 4.1 目标
两份快照（同库不同时间 / SIT vs UAT / UAT vs 拟上线）对象级差异，回答"结构改了什么""两环境一不一致"。

### 4.2 算法（`diff_snapshots`）
- 以 `(schema, obj_type, name)` 为键建两侧对象字典；
- `added` = target 有 base 无；`removed` = base 有 target 无；
- `changed` = 两侧都有但 `normalize_ddl` 后文本不等 → 输出行级 unified diff（Python `difflib`）；
- 汇总计数 + 变更清单；快照级 `sha256` 相等则直接判"完全一致"短路返回。

### 4.3 接口
`GET /api/v1/live-schema/snapshots/{id}/diff?base={base_id}` → `{identical:bool, added:[], removed:[], changed:[{object, diff}]}`。前端并排展示。

### 4.4 价值
- 上线前："拟上线库 vs 基线"差异复核；
- 跨环境："SIT vs UAT 结构对齐"体检；
- 时间线："这周谁动了表结构"审计。低成本、直接复用 F1 快照。

## 7. 兼容性与风险清单

| 项 | 处置 |
|---|---|
| 现有 API/响应体/规则 ID | 0 改动，仅新增 `/live-schema/*` 与 `live_schema` 规则档 |
| 数据表 | 仅新增 `schema_snapshots`，走 `ensure_db()` 幂等建表 |
| 采集安全 | 全程只读；建议独立最小权限采集账号（SELECT + SHOW VIEW + 对过程/事件的必要权限） |
| 敏感信息落盘 | `masked` 开关脱敏过程/函数/触发器体；快照文件纳入数据保留清理；快照下载受 RBAC |
| TDSQL 分布式限制 | 触发器/过程/事件逐对象 best-effort，失败标注不中断 |
| 大库/数百实例 | 异步任务 + 并发信号量 + 单对象超时；系统库排除 |
| RBAC 新面 | C5 矩阵测试覆盖 `live-schema`，杜绝再现 BUG-01 类越权 |
| 引入迁移框架的风险（M 稿 C1） | **规避**：不引入，沿用 `ensure_db()` |
| 前端构建链风险（M 稿 C4） | **规避**：不引入 Vite/Node，现有单页加页 |
| SSH RCE 风险（M 稿 C6） | **规避**：本版不做远程执行 |
