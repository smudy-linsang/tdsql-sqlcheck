## 5. 缺陷清单（按严重度排序）

### 🔴 严重 BUG（必须修复）

#### BUG-RBAC-01：DBA 可越权查看审计日志（v1.2 复发的 v1.0.0 老问题）

**严重度**: 🔴 严重

**问题描述**:
`dba` 角色可访问 `GET /api/v1/admin/operation-logs`，应返回 403，实际返回 200。

**复现命令**:
```bash
TOKEN=$(curl -s http://127.0.0.1:8000/api/v1/auth/login -d '{"username":"test_dba","password":"Test@1234"}' | python -c "import json,sys;print(json.load(sys.stdin)['token'])")
curl -s http://127.0.0.1:8000/api/v1/admin/operation-logs -H "Authorization: Bearer $TOKEN" | head -c 200
# {"total":1560,"logs":[{"id":1560,"operator":"admin",...}
```

**根因**:
1. `check_permission()` 在 `role == "dba"` 分支无条件返回 `allowed = True`
2. `role_permissions` 表中 dba 角色的 `sys-auditlog` 菜单被错误地设为 `visible=1`
3. 第二级 `role_permissions` 校验时 `sys-auditlog in visible` 通过
4. 设计漏洞：审计日志本应是 admin/auditor 专属，dba 不应能看

**预期修复**:
- 方案 A：在 `check_permission()` 中，对 `/api/v1/admin/operation-logs` 路径**只允许 admin 和 auditor**
- 方案 B：把 dba 的 `sys-auditlog` 权限收回（`visible=0`），但保持代码可改回（给运维灵活性）
- 推荐方案 A（硬约束优先），可在 `auth_service.py` 中加一个 `_ADMIN_AUDITOR_ONLY_PREFIXES` 常量

**关联文件**:
- `backend/services/auth_service.py:264-314` (check_permission 实现)
- `tests/test_v2_rbac_matrix.py` (智能体 G 写的 RBAC 矩阵测试，未覆盖 dba 越权)

**v1.1.0.1 历史**:
- v1.0.0 修复 BUG-01（commit ba574d7）时**只把 developer 排除**，**没处理 dba**
- v1.1.0.1 增强 `check_permission` 路径匹配算法（commit 3fb8644）但**仍未处理 dba**
- v1.2 智能体 G 的复测报告**没有捕获这个 BUG**，因为其 `test_v2_rbac_matrix.py` **完全没有 dba 角色的越权测试**

#### ISSUE-01：`connectors/metadata_fetcher.py` 是死代码

**严重度**: 🟡 中

**问题描述**:
- `backend/connectors/` 目录在 v1.2 新建，5 个文件
- `connectors/metadata_fetcher.py` 只实现了 `fetch_databases()` 和 `fetch_table_schema()` 两个方法

- `fetcher.fetch_table_schema()` **从未被调用**
- F1 实现在 sql_audit.py 178 行直接用 `pool.get_connection()` + 原始 SQL 跑，**绕过了 connectors 层**
- 实际连接的池仍是 `services/connection_registry.py` 里的 `TDSQLConnectionPool`（71KB 的大类）

**根因**:
智能体 G 新建了 `backend/connectors/` 目录作为"tdsql_connector 拆分"工作的一部分，但实际上**没有真正拆分**——原 `services/tdsql_connector.py` 完整保留，`connectors/` 目录只是个未连接的壳。

**建议修复**:
- 选项 A：删掉 `backend/connectors/` 目录（死代码）
- 选项 B：把 `services/tdsql_connector.py` 的 `TDSQLConnectionPool` / `MetadataFetcher` 等真正拆到 `services/connector/` 子包（按融合版设计文档 §3.3 实施）
- 推荐选项 B（但 v1.2 时间盒已紧，可推迟到 v1.3）

**关联代码**:
- `backend/connectors/metadata_fetcher.py` (1589 字节，仅 2 个方法)
- `backend/connectors/connection_pool.py` (1499 字节，**每调用 `get_connection()` 都新建 TCP 连接**，非连接池！)
- `backend/connectors/slow_query_fetcher.py` (533 字节，**仅实现 `fetch_active_processlist` 一个方法**)
- `backend/connectors/monitordb_client.py` (840 字节，**`fetch_monitor_metrics` 方法查询的是 `tdsql_instance_stat` 表，但设计说要查 `proxy_classes_analysis`**——SQL 不匹配)

#### ISSUE-02：`sql_audit.py:166` 死代码 + 方法名误用

**严重度**: 🟡 中

**问题描述**:
```python
# backend/api/sql_audit.py:166
tables = fetcher.fetch_databases()  # 获取包含数据库内所有的表或展示
# ^^^^^^^^^^^^^^ ① 死代码，tables 变量从未使用
```

**建议修复**:

**关联代码**: `backend/api/sql_audit.py:165-186`

#### ISSUE-03：`sql_audit.py:201` 双重分号

**严重度**: 🟢 低

**问题描述**:
```python
# backend/api/sql_audit.py:201
extracted_sqls.append(f"{create_sql};\n")
# ^^^^^^^^^^^  DDL 本身已带分号（SHOW CREATE TABLE 返回值通常带 ; ）
#                          ^^  额外再加一个分号
```

**影响**: 生成的 .sql 文件每条 DDL 后会出现 `;;\n` 双重分号，**审核引擎能容忍**但**不专业**。

**建议修复**: `extracted_sqls.append(f"{create_sql.rstrip(';')}\n")` 或 `f"{create_sql}\n"`

**关联代码**: `backend/api/sql_audit.py:201, 212`

#### ISSUE-04：`/api/v1/auth/visible-menus` 未在 `is_public_path` 白名单

**严重度**: 🟢 低

**问题描述**:
- `/api/v1/auth/visible-menus` 强制要求 token，否则 401
- 实际场景：登录前/登录失败时，前端可能想拉"无 token 情况下的可见菜单"做兜底
- 当前 `is_public_path` 白名单只有：`/health`、`/metrics`、`/`、`/favicon.ico`、`/api/v1/auth/login`
- 实际前端用法（`frontend/static/js/app.js:774`）是登录后才调，无 token 不影响主流程
- **但**作为安全防御，将 `visible-menus` 设为"未登录也允许，返回空列表"更友好

**建议修复**:
- 方案 A：在 `auth_service.py:202-204` 加 `/api/v1/auth/visible-menus` 到 `PUBLIC_PATHS`
- 方案 B：保持现状（已有兜底逻辑 `get_visible_menus()` 对未知 role 返回空）
- 推荐方案 B（保持当前行为更安全）

**关联代码**: `backend/services/auth_service.py:202-204` (PUBLIC_PATHS), `backend/api/auth.py:193-197` (visible-menus 端点)
