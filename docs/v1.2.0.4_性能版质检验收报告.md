## 0. 准出结论

> **⚠️ 暂不建议以"性能达标"发版（有条件 / Hold）。** 功能层面基本正确、无崩溃无数据损坏、pytest 992 通过、F1/RBAC/审计均正常；**但本版是"性能优化版",而其核心目标——解决"用户一多就卡"——既未被 M 验证、也无法在本次独立压测中证实其改善（本机压测反而略有回退）**。另发现 **1 项功能回归（`run_server_daily` 每节点重复执行 N 次）** 与若干低危项。**建议先修回归 + 补一次生产代表性压测证明并发确有改善，再发版。**

不是"系统坏了"级别的阻断，而是"性能版拿不出性能证据 + 一处回归"的准出把关问题。

## 1. 核心问题：性能改善"无证据"，且 M 的测试打偏了靶

### 1.1 M 的"准出"基于错误/不完整的范围
- M 的报告（`docs/v1.2.0.4_测试报告-Mavis.md`）明确写"测试范围: commit **`21806e9`**"——那是**只含日常巡检 `asyncio.to_thread`+线程池+缓存**的早期版本。
- 而真正针对"用户一多就卡"的**全局 P0/P1 改动在其后的 `d2b00ca`**（20 个 API 文件 `async def`→`def`、中间件审计日志后台化、菜单权限缓存），以及 `524adce`（写库线程锁）。**这些高风险、面最广的改动，M 基本没针对"并发提升"做验证。**
- M 的"并发"测试仅为**2 个并发的日常巡检调用**（还是从 5 并发**撞上 MySQL 1213 死锁**后降到 2 的，M 将死锁判为"非 bug"）。**全程没有一处针对"多用户并发访问平台"的压测。**

### 1.2 独立压测：并发未改善，反而略回退（本机，同一接口同一方法）
对 `GET /api/v1/dashboard/summary`（该接口已 `async`→`def`）压测：

| 并发 | 优化前基线（`21806e9`） | 本版（`524adce`, pool=10） | 本版（pool=80） |
| :---: | :---: | :---: | :---: |
| 1 | 198 req/s / p50 5ms | 141 / 7ms | 169 / 6ms |
| 20 | 212 req/s / p50 93ms | 105 / 186ms | 125 / 155ms |
| 50 | 218 req/s / p50 215ms | **102 / 466ms** | 130 / 358ms |

- 优化前后**吞吐都不随并发上升**（都卡在一条串行流水线上）；本版在本机反而**吞吐更低、尾延迟更高**。
- **公正说明**：本机元数据库是**亚毫秒本地连接**，`async→def`（线程池）的真正价值在于**重叠远程 DB 的往返等待**——生产环境元数据库多为独立实例（有网络 RTT），线程池才可能显现收益；本机 DB 太快，线程调度/GIL 开销反而占了上风。**所以本机压测"不能证明改善、也不足以坐实回退"，只能说明：至今没有任何证据表明"用户一多就卡"被解决。**
- 升级手册宣称的 **"巡检 20 倍提速 / 通信耗时降低 95% / 多核高并发定版"** 均为**未经真实集群验证的说法**（M 自述"mock 太快无法体现差异"）。

> **结论**：一个"性能版"，最该拿出的"并发前后对比曲线"至今空缺。这是不能直接盖章"性能达标"的根本原因。

## 2. 功能回归（本版新引入）

### BUG-PERF-01【中】`run_server_daily` 每次巡检重复执行 N 次
- 现象：`524adce`（加写库线程锁）把 `run_server_daily(conn,...)` 与 `conn.commit()` **移进了 `for nr in node_results:` 逐节点循环内**（`daily_inspect_service.py:483-484`）。原本它在循环**外只执行 1 次**。
- 后果：物理主机巡检采集 + 提交，从"每次巡检 1 次"变成"**每节点 1 次**"。N 节点集群即 **N 倍冗余**的服务器巡检 DB 写入与采集——**在"性能优化版"里制造了性能回归**。AST 静态分析已确认 `run_server_daily` 位于 for 循环体内。
- 修复：将 `run_server_daily(conn,...)` 与最终 `conn.commit()` **移回节点循环之外**（每次 `run_daily` 只跑 1 次）；逐节点 upsert 可保留在锁内，服务器巡检与总提交放循环后。

## 3. 低危项（不单独阻断，但建议一并处理）

| 编号 | 级别 | 问题 | 建议 |
| :--- | :--- | :--- | :--- |
| PERF-02 | 低 | 中间件审计日志改 `asyncio.create_task(asyncio.to_thread(log_operation,...))` **未持引用**，Python 可能在其完成前 GC 该 task（官方文档明确告警），高并发下**审计日志可能静默丢失** | 用后台队列/持有 task 引用集合；审计日志建议可靠落库 |
| PERF-03 | 低 | 多 worker 下 `_VISIBLE_MENUS_CACHE`/`_user_cache` **每 worker 一份**，`set_role_permissions` 只清当前 worker 缓存 → 撤权后**另一 worker 最多 30s 仍按旧权限放行**（银行级 RBAC 的最终一致窗口） | 权限变更用进程间失效（信号/共享缓存），或缩短 TTL/关键权限不缓存 |
| PERF-04 | 低 | `async→def` 改了但**未配套调大元数据连接池**（`SQLCHECK_DB_POOL_SIZE` 默认 10）与去掉每次 checkout 的 `ping()` RTT；线程池并发下连接池成新瓶颈（pool=80 时 conc50 从 102→130 req/s 略升即为佐证） | 池大小对齐线程池并发；`ping` 改"仅空闲超阈值才探活" |
| PERF-05 | 低 | 仓库里**提交了 2 个 ~21MB 源码 tar.gz**（`tdsql-sqlcheck-v1.2.0.3/4-source.tar.gz`，共约 42MB）永久膨胀 git 历史 | 从版本控制移除 + 加 `.gitignore`；发布包不入库 |

> 顺带：`daily_inspect.run` 的内部错误码此前建议 400→500，本版已修（`daily_inspect.py:38` 现为 500），✅。

## 5. 准出意见与整改建议

**准出结论：暂缓（有条件通过）。** 系统功能安全、可用，但**不建议以"性能优化达标"对外发版**，理由是"性能证据缺失 + 1 处性能回归"。请 G：

1. **（必须）** 修 BUG-PERF-01：`run_server_daily` + 总 `commit` 移出逐节点循环，恢复"每次巡检 1 次"。
2. **（必须，性能版的核心）** 补一次**生产代表性并发压测**证明"用户一多就卡"确有改善：在**元数据库为独立实例（有网络 RTT）**或人为注入 DB 往返延迟的环境下，给出"并发 1/20/50/100 的吞吐-延迟曲线，优化前 vs 优化后"。看到**吞吐随并发上升、尾延迟不再线性恶化**，才算性能达标。仅有功能通过不足以证明性能目标达成。
3. **（建议）** 一并做 PERF-04（连接池对齐 + ping 优化）——否则 `async→def` 的收益在连接池瓶颈下打折。
4. **（建议）** PERF-02（审计日志可靠化）、PERF-05（移除 42MB tar.gz）。
5. **（可选）** PERF-03（跨 worker 权限缓存失效）。

完成 1、2（并建议 3）后，凭真实压测曲线再来复验，达标即可发版。

## 6. 给 M 的方法论提醒（下次同类版本）
- **测试范围要对齐最终 commit**：本次报告写的是 `21806e9`，而发版代码到了 `524adce`，中间的全局 P0/P1 改动脱测。
- **"性能版"必须有性能基准**：功能 pass/fail 无法替代吞吐-延迟压测；`in-process mock 太快` 时应改用"注入延迟/独立 DB"的方式做真并发压测，而不是把它判为"非 bug"后跳过。
- **死锁不要轻易判"非 bug"**：5 并发触发 MySQL 1213 就降到 2 并发，是把问题绕过去了；G 后来加的写库锁正是为此，但也说明并发写模型需要被认真压测。
