## 第一章 背景与挑战

### 1.1 业务痛点

作为银行信息科技部的DBA，我们在日常运维中面临三大核心痛点：

| 痛点 | 描述 | 影响 |
|------|------|------|
| **SQL质量管控缺失** | 开发提交的SQL未经审核直接上线，导致生产事故频发 | 系统稳定性差、运维成本高 |
| **Oracle迁移适配困难** | 业务系统从Oracle迁移至TDSQL，42类不兼容语法需逐一排查 | 人工审查效率极低、遗漏风险大 |
| **慢SQL治理被动** | 缺乏主动发现和优化建议能力，只能等问题暴露后救火 | 用户体验差、资源浪费 |

### 1.2 传统方案的困境

| 方案 | 问题 |
|------|------|
| 采购商业产品 | 年费数十万，定制化困难，与行内TDSQL环境适配差 |
| 外包开发 | 沟通成本高，交付周期长，后续维护受制于人 |
| 自行开发 | **我是DBA，编程一窍不通** |

> 💡 **核心矛盾**：最懂业务需求的人（DBA），不具备编程能力；有编程能力的人，不理解DBA的专业需求。

### 1.3 破局思路

**如果DBA的专家经验 + AI的编程能力 = ？**

- ✅ 我懂TDSQL的每一个技术细节和运维规范
- ✅ 我能设计出完整的审核规则体系和架构方案
- ✅ AI工具可以将我的设计思想转化为可运行的代码

> **大胆假设**：一个人 + AI工具 = 一个完整的生产力团队

### 3.2 典型案例：Oracle迁移规则的诞生

这是整个项目中**最能体现专家驱动AI协作价值**的案例。

#### 背景

原厂交付了一份《Oracle迁移TDSQL改造适配方案》V1.5.1文档，包含大量技术细节：
- `to_char`/`nvl`/`decode` 等函数不支持
- `rownum`/`MERGE INTO`/`WITH AS` 等语法不兼容
- 分布式场景下窗口函数、游标、临时表的限制
- 分片键六条军规

**如果人工逐条记忆和排查，一个DBA团队也需要数月时间。**

#### 我的做法

**第一步**：我将原厂729行的技术文档交给设计智能体，要求：
> "将这份文档的每一个规范点，无遗漏地映射到我们的审核规则体系中。新增的规则必须覆盖原厂文档的每一条，同时要标注哪些是映射既有规则、哪些需要新增。"

**第二步**：设计智能体输出了一份648行的《Oracle迁移TDSQL规范接入审核规则库设计说明书》，包含：
- 52个规范点的全量追溯矩阵（无遗漏承诺）
- 42条新规则的逐条详细规格（检测逻辑、正则表达式、正反示例、误报防护）
- 6项既有规则增强方案
- 完整的测试设计（184项检查）

**第三步**：我将设计说明书交给编码智能体，要求：
> "严格按照设计说明书的每一个规格进行编码，不得遗漏任何一条规则，不得跳过任何一项测试。"

**第四步**：编码智能体实现了：
- `oracle_compat.py` 978行代码（42个规则类）
- 解析器增强（UniqueColumnConstraint支持）
- R054死代码修复（raw_sql回退机制）
- 前端分类展示更新

**第五步**：测试智能体独立执行184项检查，发现3个缺陷：
- R117将`BIGINT UNSIGNED`误报为非法类型
- R054唯一索引检查是死代码
- 12处测试断言未更新

**第六步**：我审核测试报告后指挥修复，最终：
- ✅ 184/184项检查全部通过
- ✅ 881条全量回归0失败
- ✅ 设计质检准出

#### 成果

| 指标 | 数值 |
|------|------|
| 新增审核规则 | 42条（R078-R119） |
| 规则总数 | 77 → 119 |
| 覆盖原厂规范点 | 52/52（100%无遗漏） |
| 代码量 | 978行规则 + 解析器增强 + 测试 |
| 耗时 | **约1周**（传统方式需数月） |

> **关键洞察**：如果我不理解TDSQL的技术细节和原厂文档的每一个规范点，AI不可能自动完成这些工作。**是我的专家判断力确保了规则的完整性和准确性。**

### 3.3 典型案例：质检验收的闭环

设计智能体不仅是设计者，还是**质检员**——这是我设计的质量保障机制。

**质检智能体的结论**：

> "引擎本体是几轮交付里质量最高的...但仍不能准出，因为：1个军规是死代码、1个高频误报、测试工作完全没做。"

**我的处理**：

1. 逐条审核质检报告的4个缺陷，确认根因分析准确
2. 指挥编码智能体按优先级修复：
   - 缺陷1（R117 UNSIGNED误报）→ 5行代码修复
   - 缺陷2（R054死代码）→ 解析器+规则层双层修复
   - 缺陷3（测试未做）→ 103条新单测 + 28个陈旧测试修复
   - 缺陷4（文档口径）→ 6处文档更新
3. 最终全量回归 **881 passed, 0 failed**

> **关键洞察**：三角色协作的核心是**制衡**——设计者设计、编码者实现、测试者验证，我作为Tech Lead做最终的质量裁决。

## 第四章 提示词工程：如何让AI完全理解人的意图

### 4.1 核心原则

经过数十轮交互，我总结出与AI高效协作的**六条铁律**：

| 原则 | 说明 | 反面案例 |
|------|------|---------|
| **① 专家先行** | 先想清楚再让AI做，不要让AI替你做决策 | "帮我设计一个系统" → 太模糊 |
| **② 设计先行** | 先出设计说明书，再让编码智能体实现 | "直接写代码" → 方向偏差 |
| **③ 精确约束** | 给出明确的验收标准和边界条件 | "优化一下性能" → 无标准 |
| **④ 闭环验证** | 每一轮交付必须经过测试验证 | "写完就提交" → 质量失控 |
| **⑤ 迭代改进** | 接受AI不完美，但要持续推动改进 | "一次就要求完美" → 不现实 |
| **⑥ 角色分工** | 不同AI角色各司其职，避免角色混乱 | "让编码的去做测试" → 缺乏独立性 |

### 4.2 提示词模式

#### 模式一：任务委派（给编码智能体）

```
你从GitHub更新一下最新内容，然后把设计说明书的内容
严格按照要求进行编码开发。设计说明书位置在
docs/xxx.md，你自己取一下。
完成后自行开展一轮冒烟测试，冒烟测试通过后向我
汇报，我会让测试智能体进行sit和uat测试。
```

**要点**：明确输入（设计说明书）、明确流程（编码→冒烟→汇报）、明确后续（测试智能体接手）。

#### 模式二：缺陷修复（给编码智能体）

```
这是测试智能体的测试报告，你照着修复。改好后我让
测试智能体复测。
```

**要点**：直接传递测试报告，让AI自己分析问题并修复，不做过度指导。

#### 模式三：质检验收（给设计智能体）

```
我让设计智能体开展了一轮质检验收，设计质检智能体
结论是不予准出，质检意见在docs/xxx.md，你拿下来
照着改一下。
```

**要点**：引入独立质检角色，用质检报告驱动修复，形成闭环。

#### 模式四：需求定义（给设计智能体）

```
原厂今天给过来一份文档，叫做"Oracle迁移TDSQL改造
适配方案"。这个文档中给出了一些TDSQL的使用规范，
比如to_char函数不支持需要替换改写等，这些内容对于
使用TDSQL进行开发的业务系统来说至关重要，现在急
需要把这个文档中的内容全部无遗漏的添加到审核规则
库中。
```

**要点**：说清楚业务背景、紧迫程度、核心要求（无遗漏），让AI理解优先级。

### 4.3 避免的陷阱

| 陷阱 | 后果 | 正确做法 |
|------|------|---------|
| 让AI自己做架构决策 | 架构混乱、前后矛盾 | 专家定义架构，AI实现细节 |
| 一次提太多需求 | AI理解偏差、遗漏需求 | 分阶段迭代，每阶段聚焦 |
| 不看AI的输出就继续 | 错误累积、返工成本高 | 每轮审核后再生成下一轮指令 |
| 忽略测试 | 上线后才发现缺陷 | 三角色制衡，测试先行 |
| 用专业编程术语沟通 | DBA不熟悉编程术语 | 用业务语言描述需求 |

## 第七章 经验总结与推广建议

### 7.1 三条核心经验

**经验一：专家是灵魂，AI是工具**

> AI不会替你做架构决策，不会判断业务需求的优先级，不会发现设计层面的遗漏。
> **是你的专家能力决定了产品的上限，AI只是帮你更快地达到这个上限。**

**经验二：设计先行，闭环验证**

> 永远不要让AI"先写代码再改"。先出设计说明书，严格按说明书实现，测试独立验证。
> 这个流程看似"慢"，实际上是最快的路径——因为它避免了方向性错误导致的返工。

**经验三：三角色制衡，质量有保障**

> 设计、编码、测试三个AI角色各司其职、相互制衡。
> 编码智能体认为自己写对了不算数，必须通过测试智能体的独立验证，再由我做最终裁决。

### 7.2 推广建议

| 层面 | 建议 |
|------|------|
| **部门层面** | 建立"AI+专家"项目孵化机制，鼓励各领域专家尝试 |
| **培训层面** | 开展AI提示词工程培训，推广五步交互方法论 |
| **工具层面** | 统一AI工具平台，建立部门级AI智能体协作规范 |
| **激励层面** | 将AI协作创新纳入绩效考核，设立创新实践奖 |
| **知识层面** | 建立AI协作案例库，沉淀可复用的方法论和提示词模板 |
