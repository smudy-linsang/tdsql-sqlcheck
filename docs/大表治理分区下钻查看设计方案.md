# 大表治理 —— 分区表下钻查看 设计方案

> **需求**：大表列表中若为分区表，可在**当前页面下钻**查看每个分区的大小/行数/边界，帮助用户判断**数据增长趋势**与**分区创建是否合理**。
> **状态**：设计（不实施）
> **日期**：2026-07-12
> **前置**：分区表采集已上线（commit 5b6196f），`is_partitioned/partition_count/shard_key` 已可用。

---

## 一、需求拆解与价值

| 用户诉求 | 下钻要展示什么 |
|---------|---------------|
| 了解数据增长 | 各分区大小/行数 + 分区**创建顺序**(ordinal) + 创建/更新时间 → 看新分区是否越来越大 |
| 判断分区是否合理 | 分区**边界值**(LESS THAN…) + 分区方法/表达式 + **兜底 MAXVALUE 分区占比** + **数据倾斜** |

核心洞察：**下钻数据与主列表同源**——都来自 `information_schema.PARTITIONS`，主列表按 (库,表) 聚合，下钻则是同一张表的**逐分区明细**（不聚合），已在内网实测该视图对 TDSQL proxy 返回真实值（用户 DBeaver query 1 证实 p0–p7 各 ~1.3GB）。

---

## 二、数据来源：`information_schema.PARTITIONS` 逐分区字段

| 字段 | 用途 |
|------|------|
| `PARTITION_NAME` | 分区名（p0/p1/…） |
| `PARTITION_ORDINAL_POSITION` | 分区序号（创建/排列顺序，看增长趋势） |
| `PARTITION_METHOD` | 分区方法（RANGE/LIST/HASH/KEY） |
| `PARTITION_EXPRESSION` | 分区表达式（如 `YEAR(created)`、`TO_DAYS(dt)`） |
| `PARTITION_DESCRIPTION` | 分区边界（RANGE 的 `LESS THAN` 值，或 `MAXVALUE`） |
| `TABLE_ROWS` | 该分区行数 |
| `DATA_LENGTH` / `INDEX_LENGTH` | 该分区数据/索引大小 |
| `CREATE_TIME` / `UPDATE_TIME` | 分区创建/最近更新时间 |

> 均为标准列，TDSQL proxy 已验证可读 `DATA_LENGTH/INDEX_LENGTH`；其余标准列一并取用。

---

## 三、方案选型：实时下钻 vs 落库

| 方案 | 说明 | 取舍 |
|------|------|------|
| **A. 实时下钻（推荐）** | 展开某表时才查该表分区，数据永远最新 | ✅ 反映真实增长；分区数有限(<100)，单表一次 SELECT 开销小；**不扩表、无快照过期** |
| B. 采集时落库 | 采集大表时把分区明细存入新表 `bigtable_partitions` | 支持离线/历史趋势，但有存储膨胀与快照过期；**列为二期增强**（做"分区月增长曲线"时才需要） |

**决策：本期用方案 A（按需实时查）。** 历史趋势（方案 B）放 §九 未来增强。

## 四、后端设计

### 4.1 新方法 `TDSQLConnectionPool.get_table_partitions(db, table)`（`tdsql_connector.py`）

```
SELECT PARTITION_NAME, PARTITION_ORDINAL_POSITION, PARTITION_METHOD,
       PARTITION_EXPRESSION, PARTITION_DESCRIPTION,
       COALESCE(TABLE_ROWS,0) AS rows_count,
       COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0) AS bytes,
       CREATE_TIME, UPDATE_TIME
FROM information_schema.PARTITIONS
WHERE TABLE_SCHEMA=%s AND TABLE_NAME=%s
  AND PARTITION_NAME IS NOT NULL          -- 排除非分区表的 NULL 行
ORDER BY PARTITION_ORDINAL_POSITION
```
- 参数化 `%s`（防注入，沿用既有口径）。
- 返回逐分区 dict 列表；`bytes` 换算 `size_gb`、算 `占比`。
- `PARTITION_NAME IS NULL` → 该表非分区表，返回空列表（前端据此不展开）。

### 4.2 新端点 `GET /api/v1/tdsql/table-partitions`（`tdsql_manage.py`，与 `/check/large-tables` 同处）

- 入参：`connection_id`（必填）、`schema`、`table`
- 复用 `_get_pool(connection_id)`；实例不存在 → 400（与既有端点一致的异常处理）
- 返回结构：
```json
{
  "schema": "tdsql_check", "table": "big_audit_trail",
  "is_partitioned": true,
  "partition_method": "RANGE", "partition_expression": "YEAR(created)",
  "partition_count": 8, "total_gb": 10.37,
  "partitions": [
    {"name":"p0","ordinal":1,"boundary":"2019","rows":1600000,
     "size_gb":1.30,"pct":12.5,"is_maxvalue":false,"create_time":"...","update_time":"..."},
    ...
  ],
  "analysis": {
    "max_partition": {"name":"p7","size_gb":1.37},
    "avg_gb": 1.30, "skew_ratio": 1.06,
    "flags": []
  }
}
```

### 4.3 派生分析（回答"分区创建是否合理"）

后端基于分区明细计算，前端渲染标记：

| 标记 | 触发条件 | 含义/建议 |
|------|---------|-----------|
| `maxvalue_oversized` | 存在 `MAXVALUE` 兜底分区且其占比 > 30%（或 > 最大常规分区） | 未及时补建未来分区，新数据全堆积在兜底分区 → **建议补建分区**（可接 `PartitionAdvisor` 出 DDL） |
| `data_skew` | 最大分区 / 平均 > 3 倍（阈值可调） | 分区键分布不均，分区大小失衡 |
| `too_many_partitions` | 分区数接近上限（复用 `PartitionMonitor.MAX_PARTITIONS=100` 的 70%/85% 水位） | 分区过多，需评估合并或调整粒度 |
| `empty_partitions` | 空/超小分区数量偏多 | 预建过多空分区（轻微提示） |

## 五、前端设计

### 5.1 交互：`el-table` 可展开行（`type="expand"`）——真正的"当前页面下钻"

在大表 `el-table` 首列加 `<el-table-column type="expand">`：
- **仅分区表可展开**：非分区行不显示展开箭头（用 `row-key` + 自定义，或在展开模板内对 `!row.is_partitioned` 显示"非分区表"占位）。
- **懒加载**：`@expand-change` 触发时才调 `/table-partitions`，避免一次性拉全部；对已加载的按 `schema.table` 缓存，避免重复请求。
- 加载态 `v-loading`；空/失败态友好提示。

### 5.2 展开内容：分区摘要 + 逐分区表 + 占比可视化

```
┌ 分区概览：RANGE( YEAR(created) ) · 8个分区 · 最大 p7 1.37GB · 倾斜 1.06x ┐
│ [若有] ⚠ 兜底 MAXVALUE 分区占比过高，建议补建未来分区                    │
├───────────────────────────────────────────────────────────────────────┤
│ 分区名 │ 边界(LESS THAN) │ 大小(GB) │ 占比条 ▓▓▓▓░ │ 行数 │ 更新时间     │
│ p0     │ 2019            │ 1.30     │ ▓▓▓▓        │ 160万│ ...          │
│ ...    │ ...             │ ...      │             │      │              │
│ pmax   │ MAXVALUE        │ 0.02     │ ▏           │ ...  │ ...(高亮)    │
└───────────────────────────────────────────────────────────────────────┘
```
- **占比条**：横向 bar（`el-progress` 或纯 CSS），直观看各分区大小分布是否均衡。
- **高亮**：最大分区（加粗/标签）、`MAXVALUE` 兜底分区（黄色）、空分区（灰）。
- 按 `ordinal` 排序 → 从左到右即"创建顺序"，一眼看出数据是否随时间增长。

### 5.3 新增前端状态与函数

- 状态：`partitionDetail`（`{[schema.table]: {...}}` 缓存）、`partitionLoading`。
- 函数：`loadTablePartitions(row)` → 调端点填缓存；`@expand-change` 绑定。
- `return{}` 导出新状态/函数（沿用既有模式）。

## 六、边界与兼容

| 场景 | 处理 |
|------|------|
| HASH/KEY 分区（无边界） | `PARTITION_DESCRIPTION` 为空 → 边界列显示 "-"，仅比大小 |
| 非分区表被展开 | 前端不显示展开箭头；即便展开也显示"非分区表" |
| TDSQL 物理子分区噪声 | 按**精确表名** + `PARTITION_NAME IS NOT NULL` 过滤，已验证返回逻辑分区 p0–p7 |
| 实例未连接 | 端点 400 + 提示"请先连接实例"；前端展开区显示提示 |
| 权限 | 端点在 `/api/v1/tdsql` 前缀 → `instances` 菜单权限，与大表列表一致 |
| 注入 | `schema`/`table` 全部 `%s` 参数化 |

## 八、验收口径

1. 分区表在大表列表**可展开**，展开显示 p0–p7 各分区真实大小（与 DBeaver `information_schema.PARTITIONS` 一致）。
2. 非分区表**不可展开**。
3. 摘要（分区方法/表达式/分区数/最大分区/倾斜）正确；`MAXVALUE 过大`/`数据倾斜` 标记按阈值正确触发。
4. HASH 分区无边界时不报错，边界列显 "-"。
5. `pytest` 全量 0 failed；新增单测通过。
6. 内网对 15005/tdsql_check 的 `big_audit_trail` 实测下钻，8 个分区大小/边界正确。

## 九、未来增强（本期不做）

1. **分区历史趋势**：采集时把分区明细快照落库（方案 B 的 `bigtable_partitions`），画"分区月增长曲线"，真正看清增长速度。
2. **一键补建分区 DDL**：对 `maxvalue_oversized` 的表，接 `PartitionAdvisor` 生成 `ALTER TABLE … ADD PARTITION …` 建议。
3. **分区水位红黄绿**：结合已有 `PartitionMonitor` 水位，在下钻摘要给分区数量红黄绿灯。
