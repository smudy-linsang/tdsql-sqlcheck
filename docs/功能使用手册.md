## 0. V2.0 登录与权限（必读）

### 0.1 登录

打开平台首页即登录页。首次部署使用 `admin` 账户（初始口令见部署时的
`ADMIN_INITIAL_PASSWORD` 或启动日志），**首次登录必须修改口令**。
API 调用先 `POST /api/v1/auth/login` 获取令牌，之后请求头携带
`Authorization: Bearer <token>`（默认8小时有效）。

### 0.2 角色与可用功能

| 功能 | admin | dba | developer | auditor |
|------|:-:|:-:|:-:|:-:|
| SQL/文件审核、EXPLAIN分析 | ✓ | ✓ | ✓ | 只读查看 |
| 连接管理、慢SQL扫描、扫描计划 | ✓ | ✓ | 只读 | 只读 |
| 规则集、门禁、保留策略 | ✓ | ✓ | 只读 | 只读 |
| 用户管理 | ✓ | ✗ | ✗ | ✗ |
| 操作审计日志查看 | ✓ | ✓ | ✓ | ✓ |
| 修改自己口令 | ✓ | ✓ | ✓ | ✓ |

口令要求：≥8位且包含大写/小写/数字/特殊字符中至少三类；连续错5次锁定15分钟（管理员可解锁）。

## 1. 系统概述

### 1.1 工具定位

TDSQL SQL审核平台是覆盖**开发、测试、生产**全生命周期的SQL质量管控与慢SQL分析平台。基于《TDSQL数据库开发规范》和《TDSQL-MySQL慢查询发现与优化方案》构建，帮助团队在SQL上线前发现潜在问题，降低生产事故风险。V2.0面向商业银行生产环境完成银行级改造，支持纯内网部署与数百套数据库实例接入。

### 1.2 核心功能模块

| 模块           | 说明                                     | API前缀                |
|---------------|-----------------------------------------|------------------------|
| 认证与用户管理 (V2.0) | 登录、四角色RBAC、用户生命周期管理        | `/api/v1/auth`         |
| SQL审核        | 77条规则自动审核，支持单条/文件/MyBatis XML   | `/api/v1/audit`        |
| 规则集管理 (V2.0) | 按项目/环境差异化规则启停与级别            | `/api/v1/rulesets`     |
| 慢SQL分析       | EXPLAIN分析、问题诊断、优化建议自动生成       | `/api/v1/slow-queries` |
| Dashboard统计  | 审核概览、慢SQL排行、规则分布图表             | `/api/v1/dashboard`    |
| GitLab集成      | MR Webhook自动审核、Diff分析、仓库批量审核   | `/api/v1/gitlab`       |
| TDSQL管理       | 多实例连接、元数据查询、慢SQL抓取、扫描计划    | `/api/v1/tdsql`       |
| 系统管理 (V2.0)  | 数据保留策略、操作审计日志                  | `/api/v1/admin`        |

## 2. 22条审核规则说明

> **V2.0 说明**：平台当前共 **77 条规则 / 8 大分类**（命名5 / DDL22 / DML9 / 索引10 /
> 分布式14 / 安全8 / 性能5 / 事务4）。本章详解的是 V0.x 起的 22 条基础规则；
> 完整清单请在前端「审核规则」页查看，或调用 `GET /api/v1/rules`（含规范来源与修复建议）。
> 各项目可通过规则集（第10章）差异化启停规则或调整级别。

### 2.1 命名规范（2条）

#### R001 — 表名长度与格式检查

| 属性   | 值                                              |
|--------|------------------------------------------------|
| 级别   | ERROR                                           |
| 类别   | 命名规范                                         |
| 规则   | 库名/表名字符限制32以内，必须符合 `^[a-z][a-z0-9_]*$` |

**说明**: 表名和库名必须以小写字母开头，仅包含小写字母、数字和下划线，长度不超过32个字符。

**违规示例**:
```sql
CREATE TABLE UserInfo (...);        -- ❌ 包含大写字母
CREATE TABLE 2024_orders (...);     -- ❌ 以数字开头
CREATE TABLE order-detail (...);    -- ❌ 包含连字符
```

**修正建议**: 系统会自动生成符合规范的名称建议，如 `UserInfo` → `user_info`。

---

#### R002 — 保留关键字检查

| 属性   | 值                            |
|--------|-------------------------------|
| 级别   | ERROR                         |
| 类别   | 命名规范                       |
| 规则   | 禁止使用TDSQL/MySQL保留关键字作表名 |

**说明**: 表名不能使用MySQL/TDSQL的保留关键字（如 `order`、`select`、`table` 等200+个关键字）。

**违规示例**:
```sql
CREATE TABLE order (...);   -- ❌ order 是保留关键字
CREATE TABLE select (...);  -- ❌ select 是保留关键字
```

**修正建议**: 添加前缀或后缀，如 `t_order`、`order_tbl`。

---

### 2.2 DDL规范（9条）

#### R003 — 必须指定主键

| 属性   | 值                                      |
|--------|-----------------------------------------|
| 级别   | ERROR                                   |
| 类别   | DDL规范                                  |
| 规则   | CREATE TABLE 必须显式指定主键              |

**说明**: TDSQL要求每个表必须有主键，用于数据分布和复制。

**修正建议**: 添加自增主键 `id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY`。

---

#### R004 — 必须使用InnoDB引擎

| 属性   | 值                          |
|--------|----------------------------|
| 级别   | ERROR                       |
| 类别   | DDL规范                      |
| 规则   | 存储引擎必须为 InnoDB         |

**说明**: TDSQL仅支持InnoDB引擎，未指定或使用其他引擎将报错。

---

#### R005 — 必须使用utf8mb4字符集

| 属性   | 值                          |
|--------|----------------------------|
| 级别   | ERROR                       |
| 类别   | DDL规范                      |
| 规则   | 字符集必须为 utf8mb4          |

**说明**: 统一使用utf8mb4字符集以支持完整的Unicode字符（包括emoji）。

---

#### R006 — 禁止ENUM/SET类型

| 属性   | 值                          |
|--------|----------------------------|
| 级别   | ERROR                       |
| 类别   | DDL规范                      |
| 规则   | 禁止使用 ENUM 和 SET 数据类型 |

**说明**: ENUM/SET类型在分布式场景下存在兼容性问题，修改枚举值需要DDL变更。

**修正建议**: 改为 `VARCHAR` 或 `TINYINT` 类型。

---

#### R007 — 禁止TIMESTAMP类型

| 属性   | 值                                    |
|--------|---------------------------------------|
| 级别   | ERROR                                 |
| 类别   | DDL规范                                |
| 规则   | 禁止使用 TIMESTAMP 类型，建议使用 DATETIME |

**说明**: TIMESTAMP类型存在2038年溢出问题，且时区转换容易引发bug。

---

#### R008 — 禁止外键约束

| 属性   | 值                                    |
|--------|---------------------------------------|
| 级别   | ERROR                                 |
| 类别   | DDL规范                                |
| 规则   | 禁止使用外键约束，应在应用层保证数据一致性 |

**说明**: TDSQL分布式架构下外键无法跨分片，且外键会降低写入性能。

---

#### R009 — 财务字段禁止FLOAT/DOUBLE

| 属性   | 值                                        |
|--------|------------------------------------------|
| 级别   | ERROR                                    |
| 类别   | DDL规范                                   |
| 规则   | 涉及金额/价格的字段禁止使用 FLOAT/DOUBLE，应使用 DECIMAL |

**说明**: FLOAT/DOUBLE存在精度丢失风险，不适合财务计算。系统通过字段名关键词（如 `amount`、`price`、`cost`、`balance` 等中英文关键词）自动识别财务字段。

**修正建议**: 使用 `DECIMAL(18,2)` 或 `DECIMAL(20,4)` 等精确类型。

---

#### R010 — VARCHAR长度检查

| 属性   | 值                              |
|--------|--------------------------------|
| 级别   | WARNING                         |
| 类别   | DDL规范                          |
| 规则   | VARCHAR 长度建议不超过 2000       |

**说明**: 超长VARCHAR影响内存分配和查询性能。超过2000字符的字段需评估是否合理。

**修正建议**: 超长字段考虑使用 `TEXT` 类型或拆分到扩展表。

---

#### R011 — 禁止TEXT/BLOB类型

| 属性   | 值                                      |
|--------|-----------------------------------------|
| 级别   | WARNING                                 |
| 类别   | DDL规范                                  |
| 规则   | 禁止在活跃表使用 TEXT/BLOB 类型            |

**说明**: TEXT/BLOB类型会导致行溢出存储，影响查询性能。涵盖 TEXT、TINYTEXT、MEDIUMTEXT、LONGTEXT、BLOB、TINYBLOB、MEDIUMBLOB、LONGBLOB、JSON 类型。

**修正建议**: 1) 拆分到独立扩展表；2) 改为VARCHAR限定长度；3) 评估是否可移除。

---

### 2.3 DML规范（8条）

#### R012 — 禁止SELECT *

| 属性   | 值                          |
|--------|----------------------------|
| 级别   | ERROR                       |
| 类别   | DML规范                      |
| 规则   | 禁止使用 SELECT *，应指定具体字段 |

**说明**: `SELECT *` 会查询所有字段，浪费网络带宽和内存，且表结构变更时容易引发兼容性问题。

**修正建议**: 明确列出需要的字段，如 `SELECT col1, col2, col3 FROM ...`。

---

#### R013 — DML必须带WHERE条件

| 属性   | 值                                          |
|--------|--------------------------------------------|
| 级别   | ERROR                                      |
| 类别   | DML规范                                     |
| 规则   | UPDATE/DELETE 必须带 WHERE 条件              |

**说明**: 不带WHERE的UPDATE/DELETE将影响全表数据，属于高危操作。

---

#### R014 — 禁止无WHERE的UPDATE/DELETE

| 属性   | 值                                          |
|--------|--------------------------------------------|
| 级别   | ERROR                                      |
| 类别   | DML规范                                     |
| 规则   | 禁止不带 WHERE 的 UPDATE/DELETE，防止误操作全表 |

**说明**: 与R013互补，进一步强调UPDATE/DELETE的安全约束。审核引擎会自动去重，避免同一问题重复报告。

---

#### R015 — 子查询嵌套限制

| 属性   | 值                                    |
|--------|---------------------------------------|
| 级别   | ERROR                                 |
| 类别   | DML规范                                |
| 规则   | 子查询嵌套深度不超过3层                  |

**说明**: 过深的子查询嵌套严重影响可读性和执行性能。

**修正建议**: 将子查询改写为 JOIN 或临时表。

---

#### R016 — WHERE条件禁止函数计算

| 属性   | 值                                              |
|--------|------------------------------------------------|
| 级别   | WARNING                                         |
| 类别   | DML规范                                         |
| 规则   | WHERE 条件中禁止函数计算、全模糊LIKE、OR条件       |

**说明**: 以下写法会导致索引失效：
- 函数: `WHERE DATE(create_time) >= '2024-01-01'`
- 全模糊LIKE: `WHERE name LIKE '%keyword%'`
- OR条件: `WHERE a = 1 OR b = 2`

**修正建议**:
1. 函数: `WHERE create_time >= '2024-01-01'`
2. 全模糊: 改为前缀匹配 `LIKE 'keyword%'` 或使用全文索引
3. OR条件: 改写为 `UNION ALL`

---

#### R017 — 禁止ORDER BY RAND()

| 属性   | 值                                          |
|--------|--------------------------------------------|
| 级别   | ERROR                                      |
| 类别   | DML规范                                     |
| 规则   | 禁止 ORDER BY RAND()，会导致全表扫描和临时表   |

**说明**: `ORDER BY RAND()` 需要为每一行生成随机数并排序，性能极差。

**修正建议**:
1. 应用层随机: 先获取 `MAX(id)` 再随机取值
2. 子查询: `SELECT * FROM t WHERE id >= (SELECT FLOOR(RAND() * (SELECT MAX(id) FROM t))) LIMIT 1`
3. 预生成随机列

---

#### R018 — 单表索引数量限制

| 属性   | 值                                |
|--------|----------------------------------|
| 级别   | WARNING                           |
| 类别   | DML规范                            |
| 规则   | 单表索引数量不超过5个（含主键）       |

**说明**: 过多索引会降低写入性能，增加存储开销。

**修正建议**: 合并冗余索引或移除不常用索引。

---

#### R019 — 禁止冗余索引

| 属性   | 值                                          |
|--------|--------------------------------------------|
| 级别   | WARNING                                    |
| 类别   | DML规范                                     |
| 规则   | 禁止创建冗余索引（前缀覆盖检查）              |

**说明**: 如果索引A的列是索引B列的前缀，则A是冗余的。例如索引 `(a)` 和索引 `(a, b)` 同时存在时，`(a)` 是冗余的。

### 2.4 分布式规范（3条）

#### R020 — 分布式表查询提醒分片键

| 属性   | 值                                                |
|--------|--------------------------------------------------|
| 级别   | WARNING                                           |
| 类别   | 分布式规范                                         |
| 规则   | 多表关联查询应在WHERE/ON条件中包含分片键字段          |

**说明**: 涉及多表（≥2张表）的SELECT/UPDATE/DELETE语句，如果WHERE条件中不包含分片键，可能导致广播到所有SET，严重影响性能。

**修正建议**: 在WHERE条件中添加分片键字段，如 `WHERE shard_key = ? AND ...`。

---

#### R021 — 禁止更新分片键

| 属性   | 值                                                |
|--------|--------------------------------------------------|
| 级别   | ERROR                                             |
| 类别   | 分布式规范                                         |
| 规则   | 禁止对分片键(shardkey)字段进行UPDATE操作             |

**说明**: 分片键决定数据路由，更新分片键会导致数据跨分片迁移，必须通过DBA审核。系统通过检测SET子句中的 `shard_key`、`shardkey` 等关键词识别。

---

#### R022 — 禁止不带分片键的全局DELETE/UPDATE

| 属性   | 值                                                  |
|--------|----------------------------------------------------|
| 级别   | ERROR                                               |
| 类别   | 分布式规范                                           |
| 规则   | 分布式表禁止不带分片键的全局DELETE/UPDATE              |

**说明**: DELETE语句的WHERE条件中如果未发现等值查询条件（`=`），在分布式场景下可能导致全SET扫描。

**修正建议**: 添加分片键的等值条件，并限制单次操作行数（建议≤1000行）：
```sql
DELETE FROM table_name WHERE shard_key = ? AND other_condition = ? LIMIT 1000
```

## 3. SQL审核功能使用

### 3.1 单条SQL审核

**API端点**: `POST /api/v1/audit/sql`

**请求示例**:

```bash
curl -X POST http://localhost:8000/api/v1/audit/sql \
  -H "Content-Type: application/json" \
  -d '{
    "sql": "SELECT * FROM t_user ORDER BY RAND()",
    "db_type": "tdsql"
  }'
```

**响应示例**:

```json
{
    "passed": false,
    "sql_type": "SELECT",
    "violations": [
        {
            "rule_id": "R012",
            "category": "dml",
            "severity": "ERROR",
            "message": "禁止使用 SELECT *（涉及表: t_user），应指定具体字段",
            "suggestion": "请明确列出需要的字段，如: SELECT col1, col2, col3 FROM ..."
        },
        {
            "rule_id": "R017",
            "category": "dml",
            "severity": "ERROR",
            "message": "禁止使用 ORDER BY RAND()，会导致全表扫描和临时表排序，严重影响性能",
            "suggestion": "替代方案: 1) 使用应用层随机..."
        }
    ]
}
```

**前端使用**: 在前端界面的"SQL审核"标签页中，直接在文本框输入SQL语句，点击"开始审核"按钮即可。

---

### 3.2 文件审核

**API端点**: `POST /api/v1/audit/file`

支持纯SQL文件和MyBatis XML文件的批量审核。

#### 纯SQL文件审核

```bash
curl -X POST http://localhost:8000/api/v1/audit/file \
  -H "Content-Type: application/json" \
  -d '{
    "content": "CREATE TABLE t_order (id INT, amount FLOAT);\nSELECT * FROM t_user;",
    "file_path": "init.sql"
  }'
```

系统会按分号分割SQL语句，逐条审核并返回汇总结果。

#### MyBatis XML文件审核

```bash
curl -X POST http://localhost:8000/api/v1/audit/file \
  -H "Content-Type: application/json" \
  -d '{
    "content": "<?xml version=\"1.0\" encoding=\"UTF-8\"?>\n<!DOCTYPE mapper PUBLIC \"-//mybatis.org//DTD Mapper 3.0//EN\" \"http://mybatis.org/dtd/mybatis-3-mapper.dtd\">\n<mapper namespace=\"com.example.UserMapper\">\n  <select id=\"getAll\" resultType=\"User\">\n    SELECT * FROM t_user\n  </select>\n  <insert id=\"addUser\">\n    INSERT INTO t_user(name) VALUES(#{name})\n  </insert>\n</mapper>",
    "file_path": "mapper/UserMapper.xml"
  }'
```

**MyBatis XML处理特性**:
- 自动识别 `<select>`、`<insert>`、`<update>`、`<delete>` 标签
- 自动清理MyBatis动态标签（`#{}`替换为`?`，移除`<if>`、`<where>`等标签）
- 准确记录SQL在原文件中的行号

#### 文件上传审核

## 4. 慢SQL分析功能使用

### 4.1 添加慢SQL并自动分析

**API端点**: `POST /api/v1/slow-queries`

```bash
curl -X POST http://localhost:8000/api/v1/slow-queries \
  -H "Content-Type: application/json" \
  -d '{
    "fingerprint": "SELECT * FROM t_order WHERE user_id = ?",
    "sql_text": "SELECT * FROM t_order WHERE user_id = 123",
    "db_name": "order_db",
    "exec_count": 5000,
    "avg_time_ms": 200,
    "max_time_ms": 1500,
    "rows_examined": 850000,
    "rows_sent": 100,
    "lock_time_ms": 50
  }'
```

**请求参数说明**:

| 参数           | 类型    | 必填 | 说明                           |
|---------------|--------|-----|-------------------------------|
| fingerprint   | string | 是   | SQL指纹（去参数值后的模板）       |
| sql_text      | string | 是   | 原始SQL文本                    |
| db_name       | string | 否   | 数据库名                       |
| exec_count    | int    | 否   | 执行次数                       |
| total_time_ms | float  | 否   | 总耗时(ms)                     |
| avg_time_ms   | float  | 否   | 平均耗时(ms)                   |
| max_time_ms   | float  | 否   | 最大耗时(ms)                   |
| rows_examined | int    | 否   | 扫描行数                       |
| rows_sent     | int    | 否   | 返回行数                       |
| lock_time_ms  | float  | 否   | 锁等待时间(ms)                  |

系统会自动分析慢SQL并生成优化建议。

---

### 4.2 EXPLAIN执行计划分析

**API端点**: `POST /api/v1/slow-queries/analyze-explain`

使用步骤：
1. 在MySQL/TDSQL中执行 `EXPLAIN SELECT ...`
2. 将结果填入请求体
3. 调用接口获取分析报告

```bash
curl -X POST http://localhost:8000/api/v1/slow-queries/analyze-explain \
  -H "Content-Type: application/json" \
  -d '{
    "explain_data": [
      {
        "id": 1,
        "select_type": "SIMPLE",
        "table": "t_order",
        "type": "ALL",
        "possible_keys": null,
        "key": null,
        "rows": 850000,
        "filtered": 10.0,
        "extra": "Using where"
      }
    ]
  }'
```

**自动诊断的问题类型**:

| 问题类型         | 说明                               |
|-----------------|------------------------------------|
| 全表扫描         | `type` 为 ALL，未使用索引            |
| 索引缺失         | `possible_keys` 为 NULL            |
| 文件排序         | `extra` 包含 Using filesort        |
| 临时表          | `extra` 包含 Using temporary       |
| 深度分页         | 大偏移量的 LIMIT                    |
| 扫描行数过多      | `rows` 远大于实际需要的行数          |
| 回表查询         | 覆盖索引优化机会                     |

### 7.5 从TDSQL抓取慢SQL

**API端点**: `POST /api/v1/tdsql/slow-queries/fetch`

```bash
# 从 performance_schema.digest 抓取（推荐）
curl -X POST http://localhost:8000/api/v1/tdsql/slow-queries/fetch \
  -H "Content-Type: application/json" \
  -d '{"source": "digest", "limit": 50, "min_time": 0.1, "task_name": "手动扫描", "time_window_start": "2026-07-01 00:00:00", "time_window_end": "2026-07-01 23:59:59"}'

# 从 processlist 多次轮询抓取当前慢SQL
curl -X POST http://localhost:8000/api/v1/tdsql/slow-queries/fetch \
  -H "Content-Type: application/json" \
  -d '{"source": "processlist", "min_time": 0, "poll_duration": 10, "poll_interval": 1, "task_name": "进程轮询", "time_window_start": "2026-07-01 00:00:00", "time_window_end": "2026-07-01 23:59:59"}'
```

**数据源说明**:

| 数据源       | 来源                                       | 说明                        |
|-------------|-------------------------------------------|-----------------------------|
| digest      | `performance_schema.events_statements_summary_by_digest` | Proxy层TopN慢SQL摘要（推荐） |
| processlist | `information_schema.processlist`（多次轮询） | 在指定时长内重复采样正在执行的SQL，合并去重 |

> **注意**: `mysql.slow_log` 数据源已移除。TDSQL分布式实例中各SET的slow_log表不记录数据，慢日志由Proxy层统一管理（写入本地文件由赤兔平台收集）。

**Processlist轮询参数**:

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| poll_duration | float | 10.0 | 轮询持续时间(秒)，范围1-60 |
| poll_interval | float | 1.0 | 采样间隔(秒)，范围0.5-5 |

系统会自动保存并分析抓取到的慢SQL记录。

前端侧边栏「用户管理」（仅 admin 可见）：新建用户（选择角色+初始口令）、
重置口令、解锁、禁用/启用、删除。全部操作写入审计日志。
接口：`GET/POST /api/v1/auth/users`、`PUT/DELETE /api/v1/auth/users/{username}`、
`POST .../reset-password`、`POST .../unlock`。

安全规则：新用户首次登录强制改口令；系统保留至少一个可用 admin；不能删除当前登录账户。

### 10.2 多实例连接管理（dba）

- **保存连接**：`POST /api/v1/tdsql/connections`（密码加密存储，列表只显示 `***`）
- **按ID连接**：`POST /api/v1/tdsql/connections/{id}/connect`
- **多实例查询**：所有查询类接口支持 `?connection_id=` 路由到指定实例，
  可同时保持数百实例连接（空闲30分钟自动回收）
- **状态总览**：`GET /api/v1/tdsql/status` 返回全部活跃连接

### 10.3 扫描计划（dba）

为每个连接配置每日定时慢SQL扫描：

```bash
POST /api/v1/tdsql/scan-schedules
{"connection_id": "<id>", "source": "digest", "cron_hour": 2, "cron_minute": 30,
 "limit_rows": 100, "min_time": 1.0}
```

调度器每分钟检查到期计划（多副本部署时仅 leader 执行）；
`GET /api/v1/tdsql/scan-schedules` 查看各计划的 `last_run_at/last_run_status`。
手动扫描受并发限流保护（单库默认2并发/全局8并发，超限返回429）。

### 10.4 规则集（dba）

按项目/环境差异化规则：

```bash
# 创建规则集：开发环境把 SELECT*(R012) 降为 INFO
POST /api/v1/rulesets
{"id": "dev_env", "name": "开发环境规则集",
 "items": [{"rule_id": "R012", "enabled": true, "severity_override": "INFO"}]}
```

项目绑定：项目的 `rule_set_id` 指向规则集后，带 `project_id` 的审核请求自动应用覆盖，
并联动该项目的质量门禁评估（审核响应中返回 `gate_result`）。
内置 `default` 规则集不可修改；被项目引用的规则集不可删除。

### 10.5 数据保留策略（dba）

```bash
GET  /api/v1/admin/retention              # 查看策略（7张表默认90~365天）
PUT  /api/v1/admin/retention              # {"table_name": "slow_queries", "retention_days": 180}
POST /api/v1/admin/retention/run          # 手动触发清理
```

调度器每日自动清理（默认03:30）。保留天数下限7天。

### 10.6 操作审计与指标

- 操作日志：`GET /api/v1/admin/operation-logs?operator=&operation_type=`
  （谁、何时、做了什么、来源IP，auditor 角色可查）
- Prometheus 指标：`GET /metrics`（请求量/审核量/违规分布/登录/RBAC拒绝/活跃连接）

### 10.7 数据脱敏说明

`DATA_MASKING_ENABLED=true`（默认）时，慢SQL入库前字面量替换为 `?`
（如 `WHERE id_card='110...'` → `WHERE id_card = ?`），分析诊断在原文上执行不受影响。
因此慢SQL列表中看到的SQL文本为脱敏后模板，属预期行为。
