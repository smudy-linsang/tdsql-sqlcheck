# TDSQL SQL审核平台 性能优化评估与全局改进方案

> **文档类型**: 性能评估 + 全局优化设计（供智能体 G 落地实施）
> **评估对象**: v1.2.0.4（G 的日常巡检性能优化）+ 全平台并发性能
> **评估方**: Fable（质检/性能）
> **日期**: 2026-07-23
> **用户反馈**: ①"用户一多就卡"（多用户并发变慢）；②"日常巡检采集慢"

---

## 0. 结论摘要

1. **G 的 v1.2.0.4 是"对的方向、但只解决了一半"**：它把日常巡检 `run_daily` 用 `asyncio.to_thread` 移出事件循环（**这是最有价值的一处，真能缓解"一个巡检卡死所有人"**），并加了 30 秒缓存 + 多节点线程池并发。但它**只针对日常巡检一个接口**，没触及"用户一多就卡"的**系统性根因**；而且巡检"采集慢"的深层成本（每个指标一条重聚合查询、每条查询新建一次 monitordb 连接）只是被**并行**了、没有被**削减**。

2. **"用户一多就卡"的系统性根因已定位并实测复现**：全平台的鉴权中间件与 **122 个 `async def` 接口在事件循环里直接跑阻塞式 pymysql 调用**，导致 uvicorn 单事件循环把并发请求**串行化**。实测吞吐在并发 1→50 时**恒定卡在 ~200 req/s 不涨**、延迟从 5ms **线性涨到 215ms**——这是事件循环被阻塞串行化的典型指纹。

3. **本文给出分级（P0/P1/P2）的全局优化方案**，P0 直击"用户一多就卡"，预期并发吞吐可提升数倍且延迟不再随并发线性恶化；P1 把巡检"采集慢"从"并行"升级为"少查 + 复用连接"，预期采集耗时进一步下降一个量级量级区间；P2 为平台级基础设施加固。

## 一、G 的 v1.2.0.4 优化评估

### 1.1 G 做了什么（`backend/services/daily_inspect_service.py` + `api/daily_inspect.py`）

| 手段 | 说明 | 我的评价 |
| :--- | :--- | :--- |
| **`asyncio.to_thread(run_daily,...)`** | 把阻塞的巡检采集从事件循环调度到 Worker 线程池 | ✅ **最有价值**。避免一次慢巡检把整个事件循环占满、连累所有在线用户。方向完全正确 |
| **30 秒内存缓存 `_DAILY_CACHE`** | key=`conn:date:nodes`，命中直接返回 | 🟡 有用但有隐患（见 1.2） |
| **`ThreadPoolExecutor(max_workers=min(8,节点数))` 多节点并发** | 把"逐节点算指标"并行化 | 🟡 对多 SET 集群有效；但**没削减**单节点内 ~20 条重查询、也没复用连接（见 1.2） |

### 1.2 局限与新引入的隐患

- **只覆盖一个接口**：`run_daily` 之外，慢 SQL 列表、审核历史、大表、监控告警等一堆 `async def` 接口仍在事件循环里跑阻塞 DB——"用户一多就卡"根因未动（见第二节）。
- **采集"并行"≠"变快"**：单节点内仍是 ~20 条"对 60 列做 `GREATEST`/`SUM` 的全表聚合"查询，且 `_monitor_execute` **每条查询新建一次 monitordb 连接**（`pymysql.connect(...)` 即用即关）。N 节点 × ~20 指标 = **~20N 次新建连接 + ~20N 条重聚合**。线程池只是让它们并行，连接建立与聚合成本仍在。
- **连接放大风险**：`ThreadPoolExecutor` 每次 `run_daily` 新建（非复用），8 线程 × 每线程每指标新建 monitordb 连接，**并发多个巡检时可能瞬间打爆 monitordb 连接数**。
- **缓存无界增长**：`_DAILY_CACHE` 是模块级 dict，**只写不清理**，key 随 `conn:date:nodes` 组合无限增长 → 长期运行内存泄漏。
- **多 worker 缓存不一致**：部署用 `--workers 2`，`_DAILY_CACHE`/`_user_cache`/元数据连接池都是**每 worker 一份**，缓存命中率打折、30 秒快照在两个 worker 间不一致。
- **错误码回归（顺带）**：`api/daily_inspect.py` 把 `run` 的异常从 `500` 改成了 `400`——巡检内部错误（服务端问题）用 400（客户端错误）语义不当，建议保留 500（校验类才用 400）。

### 1.3 小结
G 的优化**保留 `asyncio.to_thread` 这一条是对的**，其余两条（缓存/线程池）需按第三节 P1 加固；但真正的大头——"用户一多就卡"——在别处，见下。

## 二、"用户一多就卡"根因分析（含实测证据）

### 2.1 实测证据（本机 Python 3.11 + MariaDB，单 worker）

对一个普通读接口 `GET /api/v1/dashboard/summary` 做并发压测：

| 并发 | 吞吐 | p50 延迟 | p95 延迟 |
| :---: | :---: | :---: | :---: |
| 1 | 198 req/s | 5 ms | 6 ms |
| 5 | 200 req/s | 23 ms | 54 ms |
| 20 | 212 req/s | 93 ms | 101 ms |
| 50 | 218 req/s | 215 ms | 233 ms |

**读法**：并发从 1 涨到 50，**吞吐几乎不动（恒在 ~200 req/s）**，而**延迟随并发近似线性上涨**（5→215ms）。这是"请求在一条串行流水线上排队"的典型特征——**加并发不增吞吐、只堆延迟**。一个正常的异步系统应当是"吞吐随并发上升、延迟相对平稳"。

> 生产环境更糟：银河麒麟节点上元数据库多为独立实例（每次查询有网络 RTT），本机 5ms 的 DB 往返在生产可能是 十几~几十 ms，那条 ~200 req/s 的天花板会**成比例下探**，"卡"会更明显。

### 2.2 根因：阻塞 I/O 跑在异步事件循环上

1. **鉴权中间件 `backend/middleware.py::AuthMiddleware.dispatch` 是 `async def`，却在事件循环里直接调用阻塞式 pymysql**：
   - `auth_service.get_user()`（已有 30s 缓存，✅ 影响已缓解）；
   - `check_permission()`（内置角色走内存；自定义角色会查 DB）；
   - `log_operation()`（**每个 POST/PUT/DELETE 都同步写一条 `operation_logs` 并 commit**，直接压在响应路径上、阻塞事件循环）。
   中间件对**每个请求**都执行，是全局串行点。

2. **122 个 `async def` 接口在事件循环里跑阻塞 DB**（`slow_query.py`/`dashboard.py`/… 全是 `async def` 直接调同步 service）。FastAPI 对 **`def`（同步）接口会自动丢线程池**执行、不占事件循环；而 **`async def` 接口跑在事件循环上**，里面一旦有阻塞 pymysql，就把整个循环卡住，其他请求只能排队。当前 153 个接口里 **122 个是 `async def`**，绝大多数又是纯 DB 操作——正好是最该用线程池、却偏偏占着事件循环的写法。

3. **叠加因素**：
   - `--workers 2` 只是把天花板从 ~200 抬到 ~400 req/s（两条串行流水线），治标不治本，且加剧 per-worker 缓存/连接池不一致。
   - 元数据连接池每次 checkout 都 `ping()` 一次（一个额外 RTT/次）。
   - `log_operation` 同步写放在关键路径。

> **一句话根因**：系统"长着异步的壳、跑着同步的芯"——`async def` 全家桶 + 中间件里的阻塞 DB，把 uvicorn 的并发能力锁死在"单线程串行"。这与巡检无关，是全局问题。

## 三、全局性能优化方案（P0/P1/P2）

### P0 —— 让阻塞 I/O 离开事件循环（直击"用户一多就卡"，收益最大、风险最低）

**P0-1【首选，改动小收益大】把 DB 型 `async def` 接口改为 `def`**
FastAPI 会自动把 `def` 接口丢到 anyio 线程池执行，不再占事件循环。对"只做阻塞 DB、无 `await`"的接口，直接把 `async def` 改成 `def` 即可，**零逻辑改动**。
- 范围：`backend/api/*.py` 中 122 个 `async def` 里，凡是**函数体内没有真正 `await`**（大多数就是调同步 service）的，改成 `def`。
- 反例（保留 async）：确有 `await`（如 `await asyncio.to_thread(...)`、`await file.read()`、流式响应）的接口保持 `async def`。
- 优先改热点：`dashboard`、`slow_query`、`sql_audit`、`rules`、`tdsql_manage`、`auth`（列表/详情类）。
- 预期：这些接口从"串行占循环"变为"线程池并发"，并发吞吐随线程数扩展。

**P0-2 把中间件里的阻塞 DB 移出关键路径**
- `log_operation()`：改为**异步落库/后台队列**——请求响应路径只入队（内存队列），由单独后台线程批量写 `operation_logs`。或至少用 `fastapi.BackgroundTasks`/`run_in_threadpool` 让其不阻塞事件循环。审计日志允许最终一致，不必卡在响应里。
- `check_permission()` 自定义角色分支的 `get_visible_menus()`：加**角色→可见菜单的短 TTL 缓存**（类比现有 `_user_cache`），避免每请求查库。
- `get_user()`：已 30s 缓存，保持。
> 若嫌 `BaseHTTPMiddleware` 改造面大，可把鉴权逻辑改为 FastAPI 依赖（`Depends`）在 `def` 接口里执行（自动进线程池），或用纯 ASGI 中间件 + `run_in_threadpool` 包裹阻塞段。

**P0-3 anyio 线程池与 worker 配置对齐**
- 提高 anyio 默认线程池上限（默认 40）：启动时 `anyio.to_thread.current_default_thread_limiter().total_tokens = N`（如 100），或经 Starlette 配置，确保 `def` 接口有足够线程。
- `--workers` 提到 = CPU 物理核数（如 4~8），并配合下方 P2 的连接池扩容。
- 注意：多 worker 后，内存缓存/连接池按 worker 数倍增，需评估元数据库最大连接数。

> **P0 预期效果**：吞吐不再恒定在 ~200/worker，能随线程池/worker 数扩展；延迟不再随并发线性恶化。这是解决"用户一多就卡"的关键一击。

---

### P1 —— 日常巡检"采集慢"深度优化（在 G 的并行基础上再削成本）

**P1-1【收益最大】monitordb 连接池化，杜绝"每指标新建连接"**
- 现状：`tdsql_connector._monitor_execute` 每次 `pymysql.connect(...)` 即用即关。N 节点 × ~20 指标 = ~20N 次连接握手。
- 改法：为 monitordb 引入**连接池/单次采集内连接复用**——`run_daily` 一次采集**复用同一个（或每节点一个）monitordb 连接**跑完该节点所有指标，采集结束再关。预期把 ~20N 次握手降到 N 次（或 1 次）。

**P1-2【收益最大】把"每指标一条查询"合并为"每节点少数几条"**
- 现状：单节点要跑 `cpu_peak/cpu_avg/mem_peak/...` ~20 条独立的"对 `f_0..f_59` 做 `GREATEST/SUM`"全表聚合。
- 改法：对同一 `query_table` + 同一节点，**一条查询按 `f_key` 分组一次性取回所有指标**（`GROUP BY f_key`，在 SQL 里对 60 列做一次 `GREATEST/SUM` 并按 key 聚合），应用层再分发。把单节点 ~20 次往返降到 1~3 次。
- 收益与 P1-1 叠加：往返数从 ~20N 降到 ~N 甚至 ~常数级。

**P1-3 保留并加固 G 的现有手段**
- `asyncio.to_thread`：✅ 保留。
- `ThreadPoolExecutor`：保留，但**与 P1-1 连接池配合**，限制"线程数 × 每线程连接数"上限，避免打爆 monitordb；`max_workers` 做成可配置。
- `_DAILY_CACHE`：改为**有界 LRU（如 `functools.lru_cache` 或带 maxsize 的自实现 + 定期清理）**，避免无界增长；多 worker 下明确"缓存为每 worker 本地、最终一致"或改用共享缓存。
- 把 `run` 接口内部错误码从 400 改回 500（校验类才 400）。

> **P1 预期效果**：把巡检采集从"20N 次新建连接 + 20N 条重查询（并行）"降到"~N 次连接 + ~N 条查询"，首次采集（缓存未命中）耗时显著下降；配合 P0 的线程池，多用户同时巡检也不互相拖垮。

---

### P2 —— 平台级基础设施加固

**P2-1 元数据连接池优化**
- 每次 checkout 的 `ping()` 改为"仅当空闲超过 N 秒才 ping"（记录 last_used 时间戳），省掉热连接的额外 RTT。
- `SQLCHECK_DB_POOL_SIZE`（默认 10）与 P0 线程池并发对齐：`def` 接口进线程池后，单 worker 可能有几十个线程并发要连接，pool=10 会让多数走"新建即用即关"的溢出路径。建议 pool_size ≈ 线程池并发上限，或引入成熟池（DBUtils `PooledDB` / SQLAlchemy 池，自带 pre-ping、回收、上限）。

**P2-2 慢查询/大表等热点列表接口**
- 表已建索引（`operation_logs/audit_history/slow_queries` 均有 `created_at` 等索引，✅ 非瓶颈）。
- 但要确认**分页 + count 分离**、避免 `SELECT *` 拉大字段（如 `results_json`）到列表页；列表页只取摘要列，详情页再取大字段。

**P2-3 可观测性驱动优化**
- 已有 `metrics_service`：补齐**每接口延迟直方图 + 慢请求日志（>Xms 打点）**，上线后用真实数据定位 Top 慢接口，避免"凭感觉优化"。
- 压测纳入准出：用 `wrk`/`locust` 对 Top 接口做并发基线（并发 1/20/100 的吞吐-延迟曲线），回归时对比。

**P2-4 前端侧**
- 大屏/列表避免高频轮询打同一批 DB 接口（如有 `setInterval` 拉取，改为按需/降频/合并请求）；确认没有 N+1 式的逐行再请求。

## 四、落地优先级与预期收益

| 优先级 | 事项 | 触及根因 | 预期收益 | 风险/工作量 |
| :---: | :--- | :--- | :--- | :--- |
| **P0-1** | DB 型 `async def` → `def` | 用户一多就卡 | 并发吞吐随线程扩展、延迟不再线性恶化 | 低（无逻辑改动）/ 中（面广，按热点分批） |
| **P0-2** | `log_operation` 后台化 + 权限缓存 | 用户一多就卡 | 去掉写路径上的同步落库串行点 | 中 |
| **P0-3** | 线程池上限 + workers 对齐 | 用户一多就卡 | 提高并发上限 | 低 |
| **P1-1** | monitordb 连接复用/池化 | 巡检采集慢 | 连接握手数 ~20N→~N | 中 |
| **P1-2** | 指标查询合并（GROUP BY f_key） | 巡检采集慢 | 往返数 ~20N→~N | 中 |
| **P1-3** | 缓存有界化 + 保留 to_thread | 巡检稳定性 | 防内存泄漏/连接打爆 | 低 |
| **P2** | 元数据池/可观测性/前端轮询 | 全局稳态 | 长期稳定与可诊断 | 中 |

**建议实施顺序**：先 **P0-1 + P0-3**（最快见效、风险最低，直接缓解"一多就卡"）→ **P0-2** → **P1-1 + P1-2**（巡检提速）→ **P1-3/P2** 收尾。每步用第 2.1 节同款并发压测法验证"吞吐随并发上升、延迟平稳"。

## 五、给 G 的落地清单（可直接照做）

1. **P0-1**：遍历 `backend/api/*.py`，将函数体内**无 `await`** 的 `async def <handler>` 改为 `def <handler>`（先改 `dashboard/slow_query/sql_audit/rules/tdsql_manage/auth` 的列表/详情类）。含 `await`（`to_thread`/文件/流式）的保持 `async`。
2. **P0-2**：`backend/middleware.py` 的 `log_operation` 调用改为入内存队列 + 后台线程批量写；`check_permission` 自定义角色的 `get_visible_menus` 加 30s TTL 缓存。
3. **P0-3**：`backend/main.py` 启动时抬高 anyio 线程池上限；部署 `--workers` 设为核数。
4. **P1-1**：`tdsql_connector` 增加 monitordb 连接池（或 `run_daily` 内每节点复用一个连接），`_monitor_execute` 支持传入已有连接。
5. **P1-2**：`daily_inspect_service` 把单节点 ~20 次 `_get_peak/_get_sum/_get_avg` 合并为按 `f_key` 分组的 1~3 条查询。
6. **P1-3**：`_DAILY_CACHE` 改有界 LRU + 定期清理；`api/daily_inspect.py` 的 `run` 异常码 400→500。
7. **P2**：元数据池按需 ping + 扩容；补每接口延迟指标；核查前端轮询频率。

> 每项完成后，用 `GET /api/v1/dashboard/summary` 等热点接口跑并发 1/20/50 压测，确认吞吐曲线"抬头"、延迟不再线性上涨，即为达标。
