## 一、系统概述

### 1.1 系统定位

TDSQL SQL审核工具是专门针对TDSQL分布式数据库的SQL质量管控与慢SQL分析平台。系统覆盖SQL全生命周期（开发→测试→生产），提供规则审核、慢SQL深度分析、大表治理、质量门禁等核心能力，保障生产业务在TDSQL集群上稳定高效运行。

### 1.2 系统目标

| 目标维度 | 量化指标 |
|---------|---------|
| 规则覆盖率 | 70+条审核规则，覆盖架构检查项85%+ |
| 慢SQL分析维度 | 6维分析（EXPLAIN+分布式EXPLAIN+死锁+长事务+字符集+指纹） |
| 大表治理 | 三级分类+保留期管理+分区监控+变更管控 |
| 质量门禁 | GitLab MR发版强制审核 |
| 审核性能 | 单条SQL ≤200ms，1000条SQL文件 ≤10s |

### 1.3 用户角色

| 角色 | 职责 | 主要使用功能 |
|-----|------|------------|
| 应用开发人员 | 编写SQL，提交代码 | 在线SQL审核、文件审核、SQL改写建议 |
| DBA | 数据库运维，审核DDL | 大表治理、慢SQL分析、巡检报告、字符集诊断 |
| 项目经理/架构师 | 项目质量管控 | 质量门禁配置、审核趋势分析、Dashboard |
| DevOps工程师 | CI/CD流水线 | CLI工具、GitLab集成、API调用 |

## 二、系统架构设计

### 2.1 总体架构

系统采用分层架构，自上而下分为：前端展示层、API网关层、业务服务层、核心引擎层、数据存储层。

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端展示层 (Vue 3 SPA)                     │
│  Dashboard │ SQL审核 │ 文件审核 │ 慢SQL分析 │ 大表治理 │ 规则管理  │
├─────────────────────────────────────────────────────────────────┤
│                        API网关层 (FastAPI)                        │
│  认证中间件 │ 请求日志 │ 统一错误处理 │ API版本路由(/api/v2/)      │
├──────────────┬──────────────┬──────────────┬────────────────────┤
│   审核服务    │   慢SQL服务   │   大表治理    │    平台服务         │
│  audit_service│ slow_query_  │ bigtable_    │  scheduler         │
│  quality_gate │ service      │ service      │  monitor_service   │
│  gitlab_hook  │              │              │  project_service   │
├──────────────┴──────────────┴──────────────┴────────────────────┤
│                        核心引擎层                                 │
│  SQLParser │ RuleChecker │ SlowSQLAnalyzer │ BigTableAnalyzer   │
│  RuleEngine(70+条) │ FingerprintEngine │ IndexAdvisor            │
│  SQLRewriter │ CharsetDiagnoser │ DeadlockAnalyzer               │
├─────────────────────────────────────────────────────────────────┤
│                        数据存储层                                 │
│  SQLite(本地) │ TDSQL(远程) │ 文件系统(报告)                      │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 架构设计原则

1. **规则引擎插件化**：所有审核规则实现统一的BaseRule接口，新增规则零侵入
2. **引擎与服务分离**：核心引擎层无状态，可被多个服务层模块复用
3. **连接池线程安全**：TDSQL连接使用threading.local()线程本地存储
4. **配置驱动**：规则启用/禁用、门禁策略、巡检任务均通过配置驱动
5. **渐进式扩展**：单体架构保持，内部按模块严格分层，为未来微服务化预留接口

### 4.2 规则引擎 (M02)

**职责：** 加载全部审核规则，按规则类别分组调度，执行检查并收集违规结果。

**规则分类与数量：**

| 类别 | 规则ID范围 | 数量 | 说明 |
|-----|----------|------|------|
| 命名规范 | R001-R002 | 2 | 表名规范、保留关键字 |
| DDL规范 | R003-R011, R023-R038 | 25 | 建表/修改表结构规范 |
| DML规范 | R012-R019, R039-R052 | 22 | 查询/更新/删除规范 |
| 索引规范 | R053-R058 | 6 | 索引命名与设计规范 |
| 分布式规范 | R020-R022, R059-R065 | 10 | TDSQL分片键与分区规范 |
| 事务规范 | R066-R071 | 6 | 事务大小与锁规范 |
| 安全规范 | R072-R076 | 5 | 连接与权限安全 |
| **合计** | | **76** | |

**规则加载机制：**
- 启动时自动扫描 `backend/engine/rules/` 目录下所有规则模块
- 每个规则模块导出规则类列表，由RuleChecker统一实例化
- 支持通过配置文件 `config/rules_config.json` 控制每条规则的启用/禁用
- 支持规则白名单：指定表名/SQL模式豁免指定规则

**规则基类接口（保持现有BaseRule，扩展元数据）：**

```python
class BaseRule(ABC):
    rule_id: str           # 规则ID，如 "R023"
    category: RuleCategory # 规则类别
    severity: Severity     # 严重级别
    description: str       # 规则描述
    enabled: bool          # 是否启用
    spec_source: str       # 规范来源（新增）
    fix_suggestion: str    # 修复建议模板（新增）

    @abstractmethod
    def check(self, parsed: ParsedSQL, table_metadata: Optional[dict] = None) -> Optional[Violation]:
        """执行规则检查，返回违规记录或None"""
        pass
```

**RuleCategory 枚举扩展：**

```python
class RuleCategory(str, Enum):
    NAMING = "naming"
    DDL = "ddl"
    DML = "dml"
    PERFORMANCE = "performance"
    DISTRIBUTED = "distributed"
    INDEX = "index"        # 新增
    TRANSACTION = "transaction"  # 新增
    SECURITY = "security"        # 新增
```

### 4.3 慢SQL分析引擎 (M03)

**职责：** 对慢SQL进行多维度深度分析，输出诊断报告与优化建议。

**分析流水线：**

```
慢SQL输入
  │
  ├── ① SQL文本静态分析（现有增强）
  │     SELECT* / LIKE% / 子查询 / 深分页 / JOIN / OR / IN列表 / 隐式转换
  │
  ├── ② EXPLAIN执行计划分析（现有增强）
  │     type / key / rows / Extra / key_len / filtered
  │
  ├── ③ 分布式EXPLAIN分析（新增）
  │     分片键检测 / 分区裁剪 / 跨SET JOIN / Proxy下推
  │
  ├── ④ 死锁与锁等待分析（新增）
  │     INNODB STATUS解析 / innodb_lock_waits / metadata_locks
  │
  ├── ⑤ 长事务排查（新增）
  │     innodb_trx / 运行时间 / 锁定行数 / 修改行数
  │
  └── ⑥ 优化建议生成（新增）
        索引推荐 / SQL改写 / 参数建议
```

**核心类：** `SlowSQLAnalyzer`（扩展现有）、`DistributedExplainAnalyzer`（新增）、`DeadlockAnalyzer`（新增）、`LongTransactionAnalyzer`（新增）

### 4.4 大表治理引擎 (M04)

**职责：** 大表三级分类、分区监控、保留期管理、分区改造建议。

**核心类设计：**

| 类名 | 职责 |
|-----|------|
| `BigTableClassifier` | 执行大表盘点SQL，按L1/L2/L3三级分类 |
| `PartitionMonitor` | 分区数水位监控，执行分区盘点SQL |
| `RetentionManager` | 表类型分类与保留期管理 |
| `PartitionAdvisor` | 6条件判定矩阵，生成分区改造建议 |
| `CleanupAdvisor` | 分区清理方式建议（DROP/TRUNCATE/DELETE决策树） |
| `ChangeControlManager` | 大表变更管控流程管理 |

**三级分类阈值常量：**

class BigTableLevel:

    L2_DAILY_INC_GB = 1

class PartitionWatermark:

```python
class GateRule(BaseModel):
    project_id: str
    max_error_count: int = 0        # ERROR级别违规上限（0=不允许任何ERROR）
    max_warning_count: int = -1     # WARNING级别违规上限（-1=不限制）

class GateResult(BaseModel):
    passed: bool
    gate_rule_id: str
    error_count: int
    warning_count: int
    blocked_by: list[str]           # 阻断规则列表
    detail: str                     # 门禁详情描述
```

### 5.7 监控告警服务 (M15) — 新增

**职责：** 定期采集TDSQL实例监控指标，超阈值触发告警。

**监控指标：**

| 指标 | 采集SQL | 告警阈值 |
|-----|---------|---------|
| CPU使用率 | SHOW GLOBAL STATUS LIKE 'Threads_running' | >80%持续5分钟 |
| 主备延迟 | SHOW SLAVE STATUS → Seconds_Behind_Master | ≥60s告警，≥300s紧急 |
| 慢SQL数量 | 查询slow_queries表近1小时增量 | 环比增长>200% |
| 锁等待 | SELECT COUNT(*) FROM sys.innodb_lock_waits | >0告警 |
| 大事务 | SELECT COUNT(*) FROM innodb_trx WHERE ... >60 | >0告警 |

**告警通知：** Webhook（企业微信/钉钉） / 邮件 / 日志

### 5.8 巡检引擎 (M16) — 新增

**巡检任务配置：**

| 每日 | TopN慢SQL巡检、大事务监控 | 日巡检报告 |
| 每周 | 大表盘点、分区水位、慢SQL治理进展 | 周巡检报告 |
| 每月 | 完整巡检报告、索引使用率、表空间增长 | 月巡检报告 |

### 5.9 项目管理服务 (M17) — 新增

**职责：** 多系统/多项目注册与配置管理。

**项目模型：**

### 5.10 配置安全服务 (M19) — 新增

**职责：** TDSQL连接密码加密存储与脱敏。

**加密方案：**

- 脱敏展示：API返回时密码字段显示为 `***`
- 密钥不存在时：首次启动自动生成并提示用户保存

## 八、数据流设计

### 8.1 SQL审核主流程

```
用户输入SQL/文件
  │
  ▼
SQLParser.parse() → ParsedSQL
  │
  ▼
RuleChecker.audit_sql(parsed, table_metadata?)
  │  ├── 加载启用的规则
  │  ├── 按类别分组调度
  │  ├── 每条规则执行check()
  │  └── 收集Violation列表 + 去重
  │
  ▼
AuditResult {sql, sql_type, passed, violations[]}
  │
  ▼ (如果走质量门禁)
GateService.evaluate(audit_result, gate_rule)
  │
  ▼
GateResult {passed, error_count, blocked_by[]}
  │
  ▼
AuditService.save_history(audit_result, gate_result)
  │
  ▼
返回给用户/GitLab MR评论
```

### 8.2 慢SQL分析流程

```
慢SQL来源 (TDSQL慢日志/PROCESSLIST/手动输入)
  │
  ▼
FingerprintEngine.fingerprint(sql) → 指纹归并
  │
  ▼
SlowSQLAnalyzer.analyze(sql, explain_result?)
  │  ├── SQL文本静态分析
  │  ├── EXPLAIN执行计划分析
  │  ├── 分布式EXPLAIN分析（如有分片信息）
  │  └── 综合诊断报告
  │
  ▼ (如有TDSQL连接)
IndexAdvisor.analyze(sql, table_metadata) → 索引建议
SQLRewriter.suggest(sql, parsed) → 改写建议
  │
  ▼
AnalysisReport {problems[], suggestions[], optimized_sql}
  │
  ▼
SlowQueryService.save(record, report)
  │
  ▼
返回分析结果
```

### 8.3 大表巡检流程

```
Scheduler触发 (周度/每日)
  │
  ▼
BigTableService.run_inspection(connection_id)
  │  ├── BigTableClassifier.classify() → 大表清单
  │  ├── PartitionMonitor.check_watermark() → 分区水位
  │  ├── RetentionManager.check_expiry() → 保留期到期
  │  └── PartitionAdvisor.advise() → 分区改造建议
  │
  ▼
巡检结果入库 (inspection_results表)
  │
  ├── 超阈值项 → MonitorService.send_alert()
  │
  ▼
ReportService.generate_inspection_report() → PDF
  │
  ▼
存储至 data/reports/
```

## 九、非功能设计

### 9.1 性能设计

| 设计点 | 方案 |
|-------|------|
| SQL解析缓存 | LRU缓存已解析的ParsedSQL（key=SQL文本hash） |
| 规则并行执行 | 独立规则按类别分组并行执行（concurrent.futures） |
| TDSQL连接复用 | threading.local()线程本地连接池，避免频繁建连 |
| SQLite并发 | WAL模式 + busy_timeout=5000ms |
| 前端按需加载 | Vue Router懒加载，ECharts组件延迟初始化 |
| 报告异步生成 | PDF报告生成放入后台线程，API立即返回任务ID |

### 9.2 安全设计

| 安全项 | 方案 |
|-------|------|
| 密码加密 | AES-256-Fernet加密存储，密钥从环境变量读取 |
| API认证 | API Key头认证（X-API-Key），管理接口需要认证 |
| 输入校验 | pydantic模型校验所有API输入，防止注入 |
| SQL注入防护 | TDSQL查询使用参数化查询（pymysql参数绑定） |
| 敏感信息脱敏 | API响应中密码字段统一替换为 `***` |
| 操作审计 | 关键操作（连接管理、规则配置、门禁配置）记录操作日志 |

### 9.3 可靠性设计

| 可靠性项 | 方案 |
|---------|------|
| TDSQL连接断连 | 自动重连机制（pymysql ping检查 + 重新connect） |
| SQLite备份 | APScheduler每日自动备份SQLite文件至data/backup/ |
| 优雅停机 | FastAPI lifespan shutdown事件等待进行中的审核完成 |
| 任务重试 | 巡检任务失败自动重试3次，间隔5分钟 |
| 异常隔离 | 单条SQL审核异常不影响同文件其他SQL审核 |

### 11.2 关键风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| sqlglot解析复杂SQL不准确 | 规则误报/漏报 | 对无法解析的SQL降级为正则匹配，标记低置信度 |

| SQLite并发写入瓶颈 | 巡检任务与API并发时锁冲突 | WAL模式 + busy_timeout + 写操作排队 |
| 前端单HTML文件过大 | 页面加载慢 | 按需懒加载，ECharts组件延迟初始化 |
