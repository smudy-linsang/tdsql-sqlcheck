## 二、本次更新内容 (v1.0.3.4 → v1.0.4.0)

本次更新为重大能力升级版本，吸收了 TDSQL 原厂诊断能力（G1–G13）：

### 1. 新增功能特性 (G1–G13)
* **G1. monitordb 集群级慢 SQL 数据源**：支持通过 Proxy 层的 `monitordb` 采集集群级慢查询。
* **G2. 慢 SQL 十列增强诊断**：引入安全执行计划分析（`safe-EXPLAIN`）、表结构及索引状态、统计信息过期检测、扫描效率分析。
* **G3. 集群深度巡检**：读取 `m_data_cur` 指标进行阈值判级、健康度评分和严重度归一化。
* **G4. 每日巡检与多日趋势**：包含性能快照落库与 CPU/内存/磁盘资源趋势 ECharts 报告。
* **G5. 实例级索引健康审计**：检测低区分度、重复、冗余、碎片化索引及自增列耗尽。
* **G6. 跨实例表结构比对**：自动比对不同实例（如 UAT 与生产）的对象结构差异并分级展示。
* **G7. 应急诊断一键包**：以安全只读方式提取实时会话、锁等待、长事务与死锁诊断信息。
* **G8/G9. SQL 调用量多维分析与大表增长趋势**：多维度展示调用量分布并记录大表每日大小快照趋势。
* **G10. ZooKeeper 自动发现**：通过 ZK 自动识别并发现 TDSQL 实例。
* **G11. 网关日志 analysis**：离线提取并深度分析 Proxy 的 `interf` 日志生成可视分析报告。
* **G12/G13. 汇报 PPT 与运维工具箱**：支持一键生成大屏 PPT 汇报，集成磁盘性能测试与 SSH 密钥分发工具。

### 2. 权限控制升级 (RBAC 二级菜单矩阵)
* 支持在角色管理中自定义配置角色，并在权限矩阵中控制 25 个二级菜单（包括新增的 `deep-diag` 深度诊断入口等）的可见性。

## 三、数据库结构变更

本次升级包含元数据库的扩充。元数据表从 v1.0.3.4 的 **27张表** 扩充至 **40张表**。
所有数据库变更将在**系统启动时自动进行增量迁移**（采用 `CREATE TABLE IF NOT EXISTS` 和 `ALTER TABLE ... ADD COLUMN` 等幂等语句）。

### 1. 新增 13 张数据表
1. `roles` (角色定义表)
2. `role_permissions` (角色菜单可见性权限矩阵)
3. `cluster_inspection` (集群深度巡检任务表)
4. `cluster_inspection_issue` (集群深度巡检发现的问题明细)
5. `index_audit` (索引健康审计任务表)
6. `index_audit_finding` (索引健康审计发现明细)
7. `schema_diff` (表结构比对任务表)
8. `schema_diff_item` (表结构比对差异明细)
9. `emergency_report` (应急诊断报告)
10. `daily_inspection` (每日巡检/指标快照)
11. `server_daily_inspection` (每日主机资源指标快照)
12. `bigtable_history` (大表增长历史快照)
13. `gateway_log_reports` (网关日志分析报告)

### 2. 现有表字段扩充
* `slow_queries` 表：新增 `rows_affected` (DML 影响行数) 以及 `explain_plan` 等 10 列增强诊断信息字段。
* `tdsql_connections` 表：新增 `monitor_host`、`monitor_port`、`monitor_user`、`monitor_password_encrypted`、`monitor_db` 以及 `set_list` 字段，以接入 `monitordb` 及 SET 发现。
* `audit_history` 表：新增 `project_id`、`connection_id`、`gate_passed`、`gate_detail`、`top_violations`、`results_summary`、`created_by` 等多维度门禁及审计信息。

## 四、升级步骤（严格按顺序执行）

### 第 1 步：上传并校验发布包
将发布包 `tdsql-sqlcheck-v1.0.4.0-linux-x86_64.tar.gz` 拷贝至目标服务器的 `/tmp/dist/` 目录下：
```bash
cd /tmp/dist/
sha256sum -c tdsql-sqlcheck-v1.0.4.0-linux-x86_64.tar.gz.sha256
```
**期望输出**：`tdsql-sqlcheck-v1.0.4.0-linux-x86_64.tar.gz: OK`
*若校验失败，请立即停止部署并重新传输文件。*

### 第 2 步：解压发布包
```bash
tar -xzf tdsql-sqlcheck-v1.0.4.0-linux-x86_64.tar.gz
cd tdsql-sqlcheck-v1.0.4.0-linux-x86_64
ls -la
```
确认存在 `backend/`、`frontend/`、`deploy/`、`wheels/`、`requirements.txt`、`VERSION` 等文件和目录。

### 第 3 步：停止运行中的 v1.0.3.4 服务
```bash
sudo systemctl stop tdsql-sqlcheck
sudo systemctl status tdsql-sqlcheck
```
确保服务已停止且状态为 `inactive (dead)`。

### 第 4 步：备份旧版本
```bash
sudo cp -a /opt/tdsql-sqlcheck /opt/tdsql-sqlcheck-v1.0.3.4-backup
ls -la /opt/tdsql-sqlcheck-v1.0.3.4-backup/
```
确保备份完整，便于在升级失败时秒级回滚。

### 第 5 步：覆盖更新文件 (保留配置与临时数据)
```bash
# 复制后端核心代码
sudo cp -r backend/ /opt/tdsql-sqlcheck/

# 复制前端核心静态包
sudo cp -r frontend/ /opt/tdsql-sqlcheck/

# 复制依赖说明文件
sudo cp requirements.txt /opt/tdsql-sqlcheck/

# 复制版本标记文件
sudo cp VERSION /opt/tdsql-sqlcheck/

# 复制新版运维与验证脚本
sudo cp -r deploy/ /opt/tdsql-sqlcheck/
```
> [!IMPORTANT]
> **请勿覆盖** `/opt/tdsql-sqlcheck/.env` 文件以及 `/opt/tdsql-sqlcheck/data/` 目录，以保持原有的数据库连接设置与口令！

### 第 6 步：升级 Python 离线依赖包 (利用内置 wheels)
```bash
cd /opt/tdsql-sqlcheck
sudo ./venv/bin/pip install --no-index --find-links=/tmp/dist/tdsql-sqlcheck-v1.0.4.0-linux-x86_64/wheels/ -r requirements.txt
```
**期望输出**：包含所有包 `Successfully installed` 或 `already satisfied`。

### 第 7 步：启动服务并自动执行数据库升级
```bash
sudo systemctl start tdsql-sqlcheck
sleep 5
sudo systemctl status tdsql-sqlcheck
```
*启动过程中，FastAPI Lifespan 钩子将连接元数据库，自动执行 13 张新表的建表与字段扩充操作，并初始化系统默认的角色和菜单可见性数据。*

### 第 8 步：执行部署验证脚本
```bash
cd /opt/tdsql-sqlcheck
sudo bash deploy/verify_deploy.sh
```
**期望输出**：
```text

  [PASS] 健康检查 /health
  [PASS] 版本号 1.0.4.0
  [PASS] 首页可访问
  ...
════ 验证结果: PASS=7 FAIL=0 ════
部署验证全部通过 ✔
```

```bash
sudo rm -rf /opt/tdsql-sqlcheck
sudo cp -a /opt/tdsql-sqlcheck-v1.0.3.4-backup /opt/tdsql-sqlcheck
sudo systemctl start tdsql-sqlcheck
```
