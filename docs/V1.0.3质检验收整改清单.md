### P0-2　`database_filter` 存在 SQL 注入（information_schema 查询）

**严重级别**：🔴 P0（安全漏洞）

**问题定位**：`backend/engine/schema_inspector.py` `_run_check()` L244–254

**现状**：`database_filter` 直接以 f-string 拼接进 SQL，无任何校验/转义：
```python
if database_filter:
    sql = sql.replace(
        f"table_schema NOT IN ({SYSTEM_DBS})",
        f"table_schema NOT IN ({SYSTEM_DBS}) AND table_schema = '{database_filter}'"
    )
    sql = sql.replace(
        f"TABLE_SCHEMA NOT IN ({SYSTEM_DBS})",
        f"TABLE_SCHEMA NOT IN ({SYSTEM_DBS}) AND TABLE_SCHEMA = '{database_filter}'"
    )
```

**实测证据**：
```
正常 filter=qa_dirty        -> summary.total = 43
注入 filter=qa_dirty' OR '1'='1  -> summary.total = 15887
```
`OR '1'='1` 使库过滤条件恒真，返回了全实例所有库的元数据，注入成立。虽然当前查询均为 `SELECT information_schema`，破坏面主要是越权读取全库元数据 + 潜在 UNION/报错注入，但属明确的注入缺陷，必须封堵。

**期望**：`database_filter` 仅允许合法 MySQL 标识符（库名），非法输入直接拒绝或忽略过滤，不得进入 SQL 拼接。

**照抄级修复**（`schema_inspector.py`，在模块顶部加入正则，并改写 `_run_check` 的过滤分支）：

模块级（文件头部 `import` 之后）新增：
```python
import re
_IDENT_RE = re.compile(r"^[A-Za-z0-9_$]{1,64}$")
```

`_run_check` 内替换 L244–254 的 `if database_filter:` 整段为：
```python
        # 可选：按数据库过滤（仅接受合法库名，防止 SQL 注入）
        if database_filter:
            if not _IDENT_RE.match(database_filter):
                # 非法库名：直接返回错误结果，不拼接进 SQL
                return {
                    "id": check["id"], "name": check["name"],
                    "severity": check["severity"], "suggestion": check["suggestion"],
                    "count": 0, "rows": [], "columns": [],
                    "error": f"非法的库名过滤参数: {database_filter}",
                }
            sql = sql.replace(
                f"table_schema NOT IN ({SYSTEM_DBS})",
                f"table_schema NOT IN ({SYSTEM_DBS}) AND table_schema = '{database_filter}'"
            )
            sql = sql.replace(
                f"TABLE_SCHEMA NOT IN ({SYSTEM_DBS})",
                f"TABLE_SCHEMA NOT IN ({SYSTEM_DBS}) AND TABLE_SCHEMA = '{database_filter}'"
            )
```

> 校验后合法库名只含 `[A-Za-z0-9_$]`，不含引号，拼接安全。若希望更彻底，可将过滤条件改为参数化 `AND table_schema = %s` 并在 `pool._execute(sql, (database_filter,))` 传参，但改动面更大；上面的白名单校验已足以封堵注入且改动最小。

**复验口径**：
```
filter=qa_dirty        -> total = N
filter=qa_dirty' OR '1'='1  -> 每项 error="非法的库名过滤参数..."，total 不再暴涨为全库
```

**严重级别**：🟠 P1（安全）

**问题定位**：`backend/api/inspection.py` 报告生成段，L124（conn_name）、L143（error_note）、L148（ths）、L151（tds）

**现状**：所有单元格值、列名、实例名、错误信息均以 f-string 直接写入 HTML，无 `html.escape`：
```python
tds = "".join(f"<td>{row.get(c, '')}</td>" for c in cols)
ths = "".join(f"<th>{c}</th>" for c in cols)
```

**实测证据**：构造表名 `t_xss<img src=x onerror=alert(1)>`，导出报告中原样出现：
```html
<td>qa_dirty</td><td>t_xss<img src=x onerror=alert(1)></td>...
```
DBA 打开下载的 HTML 报告即触发 `onerror` 脚本。表/库/列名来自被检实例元数据，攻击者若能在目标库建表即可注入脚本。

**期望**：所有来自数据库的动态值经 HTML 转义后再拼接。

**照抄级修复**（`inspection.py` `export_schema_check_report` 内）：

文件内 `from datetime import datetime` 附近新增：
```python
    from html import escape as _esc
```

改写单元格/表头/实例名/错误提示为转义版本：
```python
    conn_name = _esc(f"{pool.config.host}:{pool.config.port}")   # L124 起
    ...
    # 若从 list_saved 取到 name，也转义：
    conn_name = _esc(c.get("name", conn_name))
    ...
    error_note = (f'<div style="color:#f56c6c;font-size:12px;margin-top:4px">执行失败: '
                  f'{_esc(str(check["error"]))}</div>') if check.get("error") else ""   # L143
    ...
    ths = "".join(f"<th>{_esc(str(c))}</th>" for c in cols)                              # L148
    ...
    tds = "".join(f"<td>{_esc(str(row.get(c, '')))}</td>" for c in cols)                # L151
```
`check['name']` / `check['suggestion']` 为内置常量可不转义，但一并转义更稳妥。

**复验口径**：对含 `<img src=x onerror=alert(1)>` 表名的库导出报告，HTML 中应出现 `t_xss&lt;img ...&gt;`（已转义），浏览器不再执行脚本。

**严重级别**：🟢 P3

**问题定位**：`backend/engine/schema_inspector.py` `get_summary()` L282–294

KPI 卡片"通过项"会把实际未跑成的检查混入，误导判断。

**期望**：出错项单独统计，不计入通过。

**照抄级修复**：
```python
    def get_summary(self, results: list[dict]) -> dict:
        summary = {"total": 0, "error": 0, "warning": 0, "info": 0,
                   "checks_passed": 0, "checks_failed": 0, "checks_error": 0}
        for r in results:
            summary["total"] += r["count"]
            if r["severity"] == "ERROR":
                summary["error"] += r["count"]
            elif r["severity"] == "WARNING":
                summary["warning"] += r["count"]
            else:
                summary["info"] += r["count"]
            if r.get("error"):
                summary["checks_error"] += 1
            elif r["count"] > 0:
                summary["checks_failed"] += 1
            else:
                summary["checks_passed"] += 1
        return summary
```
（前端 KPI/报告可选增加"执行异常"计数展示；不加也不影响，但至少通过项不再虚高。）

**复验口径**：制造一项检查报错（如断开权限），summary `checks_error>=1` 且该项不再计入 `checks_passed`。

### P3-2　C07 无主键检查：跨库同名表漏报

**严重级别**：🟢 P3（检查准确性）

**问题定位**：`backend/engine/schema_inspector.py` C07 SQL，L131–134

**现状**：
```sql
AND table_name NOT IN (
  SELECT table_name FROM information_schema.TABLE_CONSTRAINTS
  WHERE CONSTRAINT_TYPE = 'PRIMARY KEY')
```
子查询仅按 `table_name` 匹配、未按 `table_schema` 关联。若 A 库有一张无主键表，其表名恰与 B 库某张**有主键**的表同名，A 库的无主键表会被误判为"有主键"而漏报。

**实测证据**：`qa_dirty2.t_same`（无主键）与 `qa_dirty.t_same`（有主键）同名，C07 漏报了 `qa_dirty2.t_same`。

**期望**：按 (schema, table) 联合判断主键存在性。

**照抄级修复**（改为 schema 关联的 NOT EXISTS）：
```python
            "sql": (
                "SELECT t.table_schema AS `数据库`, t.table_name AS `表名` "
                "FROM information_schema.TABLES t "
                "WHERE t.table_type = 'BASE TABLE' "
                "AND NOT EXISTS ("
                "  SELECT 1 FROM information_schema.TABLE_CONSTRAINTS c "
                "  WHERE c.CONSTRAINT_TYPE = 'PRIMARY KEY' "
                "  AND c.TABLE_SCHEMA = t.table_schema "
                "  AND c.TABLE_NAME = t.table_name"
                ") "
                "AND t.table_schema NOT IN ({sys})"
            ),
```

> 注意：此 SQL 含 `t.table_schema NOT IN ({sys})`，P0-2 的过滤 `replace` 目标串为 `table_schema NOT IN ({SYSTEM_DBS})`，改写后应保持 `t.table_schema NOT IN ({sys})` 中的 `table_schema NOT IN (...)` 子串可被 replace 命中（当前写法 `t.table_schema NOT IN (...)` 中 `table_schema NOT IN (...)` 仍是子串，能命中，database_filter 追加的 `AND table_schema = '...'` 亦合法）。整改后请用 `database_filter=qa_dirty` 复测一次确保过滤仍生效。

**复验口径**：构造跨库同名（一有主键一无主键）表，C07 应检出无主键的那张。
