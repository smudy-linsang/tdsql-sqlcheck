## 二、本次更新内容 (v1.0.4.9 → v1.0.5.0)

本次更新主要针对「系统管理」中「权限矩阵」的管理颗粒度进行优化和升级：

### 1. 废弃菜单项彻底清理与下线
* **说明**：系统下线了“数据库体检”（`health-check`）模块，但其历史权限键仍在权限矩阵页面展示。
* **效果**：本次升级在后端初始化与存量库订正逻辑中，彻底删除了 `role_permissions` 表中包含 `health-check` 在内的所有过期/废弃菜单项，使权限矩阵更加整洁、清晰。

### 2. 深度诊断子模块权限颗粒化细分
* **说明**：原本「深度诊断」作为一整个大模块进行权限授权，用户无法单独控制其内部子功能。
* **改进**：
  - 将「深度诊断」下的 9 个子模块（集群巡检、日常巡检与对比报告、索引体检、结构比对、应急诊断、SQL分析、网关日志分析、PDF报告与大屏、运维工具箱）作为独立的权限项引入“权限矩阵”。
  - 升级后，存量数据库将自动为所有已有角色完成新权限的初始化默认值填充（默认可见），不影响现有使用。
  - 管理员现在可以在「权限矩阵」里自由为不同角色单独配置这 9 个子功能的可见性。
  - 前端界面与后端 API 二级校验均已完美适配。

### 3. 日常巡检单次手动采集范围限制
* **说明**：为防止内网超大规模集群或监控库数据量过大时，用户因选择过长的采集日期范围导致后台拉取任务堆积卡死，影响整个服务响应。
* **改进**：
  - 前端点击“手动采集巡检数据”时，强制限制单次手动采集的时间跨度最长为 3 天。
  - 超过 3 天会弹出友好提示，建议分批采集，规避大集群采集超时的潜在风险。

### 4. SQL分析模块平均耗时单位修正
* **说明**：在“深度诊断”>“SQL分析”中的“SQL调用量多维分析”表格中，原“平均耗时(s)”显示单位为秒，但由于大多数高频SQL的平均执行时长在几毫秒至几十毫秒级别，使用秒级浮点数显示（如 `0.005`）直观度较低，且与“慢SQL记录”等其他功能模块的单位不统一。
* **改进**：
  - 将表格的“平均耗时(s)”列名修改为“平均耗时(ms)”。
  - 前端渲染时自动将原秒级数值乘 1000 转换为毫秒单位显示并保留两位小数，与其他统计模块的展示习惯完美对齐。

## 三、数据库结构变更

本次升级包含**元数据库结构自动增量订正**。
在服务启动时，后端会自动执行以下 SQL 订正，无需运维人员手动执行：
1. 清除废弃菜单项：
   `DELETE FROM role_permissions WHERE menu_key NOT IN (...)`
2. 自动检查已有角色并补齐 9 个深度诊断子菜单的默认权限项：
   `INSERT INTO role_permissions(role_id, menu_key, visible) VALUES (...)`

## 四、升级步骤（严格按顺序执行）

### 第 1 步：上传并校验发布包
将发布包 `tdsql-sqlcheck-v1.0.5.0-linux-x86_64.tar.gz` 拷贝至目标服务器的 `/tmp/dist/` 目录下：
```bash
cd /tmp/dist/
sha256sum -c tdsql-sqlcheck-v1.0.5.0-linux-x86_64.tar.gz.sha256
```
**期望输出**：`tdsql-sqlcheck-v1.0.5.0-linux-x86_64.tar.gz: OK`
*若校验失败，请立即停止部署并重新传输文件。*

### 第 2 步：解压发布包
```bash
tar -xzf tdsql-sqlcheck-v1.0.5.0-linux-x86_64.tar.gz
cd tdsql-sqlcheck-v1.0.5.0-linux-x86_64
ls -la
```
确认存在 `backend/`、`frontend/`、`deploy/`、`wheels/`、`requirements.txt`、`VERSION` 等文件 and 目录。

### 第 3 步：停止运行中的旧服务
```bash
sudo systemctl stop tdsql-sqlcheck
sudo systemctl status tdsql-sqlcheck
```
确保服务已停止且状态为 `inactive (dead)`。

### 第 4 步：备份旧版本
```bash
sudo cp -a /opt/tdsql-sqlcheck /opt/tdsql-sqlcheck-v1.0.4.9-backup
ls -la /opt/tdsql-sqlcheck-v1.0.4.9-backup/
```
确保备份完整，便于在升级失败时秒级回滚。

### 第 5 步：覆盖更新文件 (保留配置与临时数据)
```bash
# 复制后端核心代码
sudo cp -r backend/ /opt/tdsql-sqlcheck/

# 复制前端核心静态包
sudo cp -r frontend/ /opt/tdsql-sqlcheck/

# 复制依赖说明文件
sudo cp requirements.txt /opt/tdsql-sqlcheck/

# 复制版本标记文件
sudo cp VERSION /opt/tdsql-sqlcheck/

# 复制新版运维与验证脚本
sudo cp -r deploy/ /opt/tdsql-sqlcheck/
```
> [!IMPORTANT]
> **请勿覆盖** `/opt/tdsql-sqlcheck/.env` 文件以及 `/opt/tdsql-sqlcheck/data/` 目录，以保持原有的数据库连接设置与口令！

### 第 6 步：升级 Python 离线依赖包 (利用内置 wheels)
```bash
cd /opt/tdsql-sqlcheck
sudo ./venv/bin/pip install --no-index --find-links=/tmp/dist/tdsql-sqlcheck-v1.0.5.0-linux-x86_64/wheels/ -r requirements.txt
```
**期望输出**：`Successfully installed` 或 `already satisfied`。

### 第 7 步：启动服务并检查状态
```bash
sudo systemctl start tdsql-sqlcheck
sudo systemctl status tdsql-sqlcheck
```
确保服务状态为 `active (running)` 且无错误日志。
