## 2. BUG-V12-01【阻断】main.py 缺 Optional 导入 —— 生产环境无法启动

### 2.1 现象
- 在本验收环境（Python 3.11）直接启动服务：
  ```
  File "/home/user/tdsql-sqlcheck/backend/main.py", line 181, in <module>
      async def serve_frontend(ui: Optional[str] = None):
                                   ^^^^^^^^
  NameError: name 'Optional' is not defined
  ```
  **进程启动即崩溃，`/health` 无响应，服务完全不可用。**
- 运行测试套件（Python 3.11）：
  ```
  ERROR tests/test_v2_rbac_matrix.py - NameError: name 'Optional' is not defined
  !!!!!! Interrupted: 1 error during collection !!!!!!
  ```
  **凡是通过 TestClient 导入 `backend.main` 的用例，在收集阶段即整体报错——"995 passed"在 Python 3.11 上无法复现。**

### 2.2 根因
- `backend/main.py:181` 的函数签名 `async def serve_frontend(ui: Optional[str] = None)` 使用了 `Optional`，但**该文件从未 `from typing import Optional`**，也**没有 `from __future__ import annotations`**。
- Python **< 3.10 ~ 3.13**：函数注解在 `def` 执行（模块导入）时立即求值 → `Optional` 未定义 → `NameError` → 模块导入失败 → 整个应用无法加载。
- Python **3.14**（M 的环境）：PEP 649 起注解默认**延迟求值**，`def` 时不再对注解求值，因此该错误被掩盖，导入成功、服务能跑、测试能过。

### 2.3 为什么两轮复测没发现
M 两轮复测均在 **Windows / Python 3.14.6** 下进行（见 `v1.2_复测报告2-Mavis.md` 环境行）。3.14 的延迟注解恰好绕过了该 Bug，故 M 得到"995 passed、服务 PID 51104 正常"的假象。**独立验收换用生产代表性的 Python 3.11 后，缺陷立即暴露。** 目标生产环境为银河麒麟 V10 SP3，其自带 Python 通常为 3.7~3.9，**该缺陷在生产上 100% 触发、服务无法拉起**——属发布级阻断。

### 2.4 修复方案（一行）
在 `backend/main.py` 顶部补充导入：
```python
from typing import Optional
```
（或将 `ui: Optional[str]` 改为 `ui: str | None` 并加 `from __future__ import annotations`；推荐直接补导入，最小改动。）

### 2.5 修复验证（本验收已实测）
本验收对 `main.py` 打入 `from typing import Optional` 的本地测试补丁后（已回退，未提交）：
- 服务在 Python 3.11 正常启动，`/health` 返回 `{"status":"ok","version":"1.2.0.0"}`；
- pytest 全量 **986 passed**（8 例失败均为环境/夹具数据相关，见 §6，非回归）；
- F1、RBAC、迁移等功能实测通过（见下）。
- **建议 G 应用该导入后，务必在 Python 3.9/3.11（非 3.14）环境重跑一次启动 + pytest 收集，作为准出前置。**

## 3. BUG-V12-02【中】F1 报告重复落库

### 3.1 现象
每调用一次 `POST /api/v1/audit/extract-and-audit`，`audit_history` 表新增 **2 条**记录（`audit_type='file'` 与 `audit_type='extracted_schema'`，`source` 同为 `extracted_*.sql`）：
```
id  audit_type        source
12  file              extracted_tdsql_test_20260722_114527.sql
13  extracted_schema  extracted_tdsql_test_20260722_114527.sql
```
`GET /api/v1/audit/extracted-reports` 的过滤条件为 `audit_type='extracted_schema' OR source LIKE 'extracted_%.sql'`，两条都命中 → **在线元数据审核历史列表出现重复记录**，同时污染"文件审核"历史。

### 3.2 根因
`backend/api/sql_audit.py::extract_and_audit`：
```python
results, summary, _ = audit_service.audit_file_content(...)   # ← 内部已 _save_audit_history("file", ...)
...
report_id = _save_audit_history(audit_type="extracted_schema", ...)  # ← 又存了一次
```
`audit_file_content()` 内部已经调用 `_save_audit_history("file", source, ...)` 落库一次，外层又显式落库一次，导致双写。

### 3.3 修复方案（任选其一）
1. 给 `audit_file_content()` 增加 `persist: bool = True` 形参，F1 调用时传 `persist=False`，只保留外层 `extracted_schema` 落库；或
2. F1 不再显式二次落库，改为读取 `audit_file_content` 的返回自行取 id；或
3. 最小改动：`/extracted-reports` 列表过滤改为仅 `audit_type='extracted_schema'`（去掉 `OR source LIKE`），并让 `audit_file_content` 对 F1 场景不落 `file` 行。推荐 1，根因上消除双写。

## 4. ISSUE-V12-03【低】运维工具箱 /run 空壳 + NameError

- `POST /api/v1/toolkit/run` 仅向 `tool_runs` 插入一行 `status='RUNNING'` 便返回"任务已提交调度"，**此后没有任何执行、状态永远停在 RUNNING、`finished_at` 永为 NULL**（实测 `tr_d24043100bd2` 提交后一直 RUNNING）。功能名不副实。
- `tool_bridge_service.py:41` `update_run_status` 用了未定义的 `error_message`（形参名为 `error_msg`）→ 一旦被调用即 `NameError`；目前无处调用（`/run` 从不更新状态），属潜伏缺陷。
- `create_run_task` 的 `operator` 取自请求体 `payload.get("operator","system")` 而非认证上下文，可被伪造/记为 `system`。
- **正面**：C6 未落地为"Web 端 SSH 远程执行 shell"，无 RCE 风险（我在设计评审中标注的高危面未出现），这点是对的。
- **建议**：本版若不交付真实调度，应隐藏/下线 `/run` 入口，或补齐执行链；顺手修 NameError；operator 改用鉴权用户名。

## 5. ISSUE-V12-04 / 05【低】双 UI 未交付 + 死代码

- **V12-04**：`frontend/dist/` 不存在，`/` 与 `/?ui=v1` 均回退到 v1 单页（正常可用，118KB）。`frontend_v2/` 仅有 `package.json` + 2 个组件（MonacoEditor.vue / ToolConsoleDrawer.vue），**并非可运行的应用**。而升级手册第二节把"Vite + Vue 3 渐进式双 UI 架构""Monaco Editor 智能高亮"列为已交付特性——**过度宣称**。建议：要么构建并随包交付 `dist/`，要么在手册中标注"v2 前端为预研，未启用"。
- **V12-05**：`backend/connectors/` 下 5 个文件（base_connector / connection_pool / metadata_fetcher / monitordb_client / slow_query_fetcher）为死代码（F1 实际 import 的是 `backend.connectors.metadata_fetcher`——需注意：该文件**是**被 F1 引用的，其余 4 个无引用）。M 首轮 ISSUE-01 已提，G 未清。建议核对后清理无引用者。

> 更正：经核对，`backend/connectors/metadata_fetcher.py` **被 `sql_audit.py` 的 F1 引用**（`from backend.connectors.metadata_fetcher import MetadataFetcher`），并非死代码；其余 `base_connector.py / connection_pool.py / monitordb_client.py / slow_query_fetcher.py` 无引用，可清理。
