## 5. 关键发现 (P0 关注)

### 5.1 🚨 G 1.2.0.4 错误码改动被 Agent A 静默回滚

**G 1.2.0.4 commit `21806e9`** (release(v1.2.0.4)):
```python
-        raise HTTPException(status_code=500, detail=str(e))
+        raise HTTPException(status_code=400, detail=str(e))
```

**Agent A commit `d2b00ca`** (P0/P1 性能优化, 1.2.0.4 之后):
```python
-        raise HTTPException(status_code=400, detail=str(e))
+        raise HTTPException(status_code=500, detail=str(e))
```

**实际现状**: 用户**感受**仍是 400, 因为 `monitor_probe()` 失败的前置校验仍走 400。但**`svc.run_daily` 内部异常** (如 DB 死锁、SQL 错误) 现在会回 500, 不再是 400。

**建议**: G 决策是按 G 设计 (统一 400) 还是按 Agent A 改回 500 (统一 500)。当前混用状态**不利于 API 一致性**。

### 5.2 G 1.2.0.4 perf test 是 in-process mock, 不覆盖 HTTP 层

`tests/test_v2_daily_inspect_perf.py`:
```python
mock_pool = MagicMock()
mock_pool._monitor_execute.return_value = []
res1 = run_daily(mock_pool, connection_id="test_perf_conn", inspect_date="2026-07-23")
```

**问题**:
- 直接调 `svc.run_daily`, 完全绕开 HTTP 层 (`asyncio.to_thread` 调度逻辑没测)
- Mock pool 返回空数据, 性能提升主要来自 mock 兜底而非真实计算

**建议**: G 补一个 HTTP 层 perf 集成测试, 用真实 monitordb (在 monitordb 可用环境)。

## 6. 测试环境局限性 (诚实声明)

**当前测试环境的限制**:

1. **无 monitordb**: 测试环境只有 MySQL 8.0 (127.0.0.1:13306), 没有 monitordb 端口 15001
   - 影响: daily inspect HTTP **成功路径**不可达 (monitordb 不可用直接 400)
   - 应对: 改用 in-process 调用 `svc.run_daily` + MagicMock 验证 service 层逻辑

2. **In-process mock 性能测试的局限**:
   - mock 池返回空数据, cache miss 耗时很短 (实测 25.9ms), 真实环境应该更慢
   - 加速比 55.8x 是 in-process 数字, 真实 HTTP 加速比可能不同

3. **未能验证的项目**:
   - HTTP 层 asyncio.to_thread 在真实 monitordb 下的并发表现
   - daily inspect 真实多节点采集 (mock 默认 2 节点)
   - 真实 cache miss 性能

**建议**: G 在有 monitordb 的环境补一轮 HTTP 层 SIT, 验证真实业务流下的性能。
