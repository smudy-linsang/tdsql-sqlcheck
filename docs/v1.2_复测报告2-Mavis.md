## 2. BUG-RBAC-01 修复详细验证 (核心)

### 2.1 修复方式 (G 的 diff)

```python
# backend/services/auth_service.py:206-210
+_ADMIN_AUDITOR_ONLY_PREFIXES = (
+    "/api/v1/admin/operation-logs",
+)

# backend/services/auth_service.py:277-279
+if any(path.startswith(p) for p in _ADMIN_AUDITOR_ONLY_PREFIXES):
+    return role in ("admin", "auditor")

# 关键: 这条规则放在 if role == "dba": return True 之前
# 所以 dba 走到这里时已经被这条规则拦下,根本到不了下面的 dba 一级放行
```

```python
# backend/services/database.py:1409-1413
+# 存量库订正: sys-auditlog 仅 admin 与 auditor 可见, dba 与 developer 改为 0
+conn.cursor().execute("""
+    UPDATE role_permissions SET visible=0
+    WHERE menu_key='sys-auditlog' AND role_id IN ('dba', 'developer')
+""")
```

### 2.4 修复正确性分析

G 的修复方式我认为是**正确的**,且**双层防御**:
- **API 层**: `_ADMIN_AUDITOR_ONLY_PREFIXES` 在 `if role == "dba": return True` 之前,确保 dba 访问此路径时直接被拦
- **数据层**: 启动时通过 `_init_default_data` 中的 UPDATE 订正存量库数据,防止菜单层绕过
- **测试层**: 新增 dba 测试用例,保证后续回归

**唯一的疑虑** (但可接受): `_ADMIN_AUDITOR_ONLY_PREFIXES` 写成元组常量,如果将来审计相关接口扩展(如 `/api/v1/admin/audit-export`),需要记得加进元组。建议 G 后续把审计路径抽成一个公共前缀 `/api/v1/admin/audit*` 用 startswith 统一处理。

def test_rbac_dba_restrictions(rbac_env):
    client, tokens = rbac_env
    # DBA 禁止越权访问操作/审计日志
    resp = client.get("/api/v1/admin/operation-logs", headers=tokens["test_dba"])
    assert resp.status_code == 403, f"DBA 不应能看操作审计日志: {resp.status_code}"
    # DBA 禁止创建/管理用户
    resp = client.post("/api/v1/auth/users", headers=tokens["test_dba"], json={"username": "new_user", ...})
    assert resp.status_code == 403, f"DBA 不应能管理用户: {resp.status_code}"

### 6.2 ISSUE-04 `visible-menus` 未在 `is_public_path` (G 未改)

**复测验证**: 无 token 访问 `/api/v1/auth/visible-menus` 返回 401,前端实际在登录后才调用,401 不影响功能。
**风险**: 无
**建议**: 不必修,前端调用时机决定 401 是合理的

### 6.3 长期建议 (Mavis 提议,非 G 必修)

1. **`_ADMIN_AUDITOR_ONLY_PREFIXES` 抽公共前缀**: 把审计路径统一定义为 `/api/v1/admin/audit*` 模式,避免后续扩展遗漏
2. **pytest 测试顺序稳定化**: 第一次跑全量时偶发 27 fail,但单跑/重跑都过,说明测试间存在状态污染,建议加 `--randomly-seed` 或显式 fixture teardown 保证可重复
3. **审计日志访问审计**: dba 现在被禁止访问 operation-logs,但建议在 operation-logs 接口本身加访问审计日志,记录"某用户尝试访问被拒"事件,便于安全审计
