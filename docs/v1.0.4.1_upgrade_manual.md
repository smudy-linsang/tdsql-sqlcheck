## 二、本次更新内容 (v1.0.4.0 → v1.0.4.1)

本次更新为热补丁版本，主要修复了 MonitorDB 监控库测试连通性与扫描时的易用性问题：

### 1. 实例连接测试支持 MonitorDB
* 修复前：新建或编辑实例连接时，“测试连接”按钮仅测试业务实例连接是否通畅，无法测试 MonitorDB 配置。
* 修复后：测试连接时会同时对 MonitorDB 进行连通性探测。若业务库成功但监控库连接失败，界面会显示警告徽标并提示具体错误，方便用户在保存前核对。

### 2. 慢SQL扫描异常弹窗提醒
* 修复前：当使用 `monitordb` 做扫描数据源且遇到连接报错或权限报错时，后端报错会被静默吞没，前端提示“扫描完成，抓取 0 条慢SQL”。
* 修复后：前端会对抓取接口返回的 `errors` 进行解析，如果检测到报错，会弹出明确的警告对话框显示报错详情（如 `Access denied` 或 `table not found`），并自动把该任务状态标记为 `failed`。

### 3. 动态兼容数值型 `timestramp` 时间字段
* 修复前：在 `get_cluster_slow_queries()` 查询中，直接对 `timestramp` 使用字符串形式的日期范围过滤。部分 TDSQL 版本的 `proxy_classes_analysis.timestramp` 为数值型 `bigint` Unix 时间戳，导致比对直接返回 0 行。
* 修复后：后端在扫描取数前会自动探测 `timestramp` 的实际数据类型。若为数值型，则自动探测其是“秒”级还是“毫秒”级，并动态将日期时间范围转换为对应数值参与过滤，保证不同 TDSQL 版本下均可准确拉取。

### 4. 修复任务列表“抓取数”空白的问题
* 修正了前端在扫描任务表格中把列属性错误绑定为 `fetched_count` 的 Bug，目前已修改为 `total_fetched`，能正常在列表展示抓取到的慢SQL记录数。

## 四、升级步骤（严格按顺序执行）

### 第 1 步：上传并校验发布包
将发布包 `tdsql-sqlcheck-v1.0.4.1-linux-x86_64.tar.gz` 拷贝至目标服务器的 `/tmp/dist/` 目录下：
```bash
cd /tmp/dist/
sha256sum -c tdsql-sqlcheck-v1.0.4.1-linux-x86_64.tar.gz.sha256
```
**期望输出**：`tdsql-sqlcheck-v1.0.4.1-linux-x86_64.tar.gz: OK`
*若校验失败，请立即停止部署并重新传输文件。*

### 第 2 步：解压发布包
```bash
tar -xzf tdsql-sqlcheck-v1.0.4.1-linux-x86_64.tar.gz
cd tdsql-sqlcheck-v1.0.4.1-linux-x86_64
ls -la
```
确认存在 `backend/`、`frontend/`、`deploy/`、`wheels/`、`requirements.txt`、`VERSION` 等文件和目录。

### 第 3 步：停止运行中的 v1.0.4.0 服务
```bash
sudo systemctl stop tdsql-sqlcheck
sudo systemctl status tdsql-sqlcheck
```
确保服务已停止且状态为 `inactive (dead)`。

### 第 4 步：备份旧版本
```bash
sudo cp -a /opt/tdsql-sqlcheck /opt/tdsql-sqlcheck-v1.0.4.0-backup
ls -la /opt/tdsql-sqlcheck-v1.0.4.0-backup/
```
确保备份完整，便于在升级失败时秒级回滚。

### 第 5 步：覆盖更新文件 (保留配置与临时数据)
```bash
# 复制后端核心代码
sudo cp -r backend/ /opt/tdsql-sqlcheck/

# 复制前端核心静态包
sudo cp -r frontend/ /opt/tdsql-sqlcheck/

# 复制依赖说明文件
sudo cp requirements.txt /opt/tdsql-sqlcheck/

# 复制版本标记文件
sudo cp VERSION /opt/tdsql-sqlcheck/

# 复制新版运维与验证脚本
sudo cp -r deploy/ /opt/tdsql-sqlcheck/
```
> [!IMPORTANT]
> **请勿覆盖** `/opt/tdsql-sqlcheck/.env` 文件以及 `/opt/tdsql-sqlcheck/data/` 目录，以保持原有的数据库连接设置与口令！

### 第 6 步：升级 Python 离线依赖包 (利用内置 wheels)
```bash
cd /opt/tdsql-sqlcheck
sudo ./venv/bin/pip install --no-index --find-links=/tmp/dist/tdsql-sqlcheck-v1.0.4.1-linux-x86_64/wheels/ -r requirements.txt
```
**期望输出**：`Successfully installed` 或 `already satisfied`。

### 第 7 步：启动服务并自动验证
```bash
sudo systemctl start tdsql-sqlcheck
sleep 5
sudo systemctl status tdsql-sqlcheck
```

### 第 8 步：执行部署验证脚本
```bash
cd /opt/tdsql-sqlcheck
sudo bash deploy/verify_deploy.sh
```
**期望输出**：
```text
════ 部署验证 v1.0.4.1 @ http://127.0.0.1:8000 ════
  [PASS] 健康检查 /health
  [PASS] 版本号 1.0.4.1
  [PASS] 首页可访问
  ...
════ 验证结果: PASS=7 FAIL=0 ════
部署验证全部通过 ✔
```

## 五、回滚方案 (若升级失败)

若验证脚本报错或系统运行异常，请执行以下命令回滚到原 v1.0.4.0 版本：
```bash
sudo systemctl stop tdsql-sqlcheck
sudo rm -rf /opt/tdsql-sqlcheck
sudo cp -a /opt/tdsql-sqlcheck-v1.0.4.0-backup /opt/tdsql-sqlcheck
sudo systemctl start tdsql-sqlcheck
```
