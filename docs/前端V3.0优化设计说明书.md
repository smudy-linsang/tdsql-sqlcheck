# TDSQL SQL 审核平台 — 前端 V3.0 优化设计说明书

> **文档性质**：本文档是一份**可直接执行的前端重构设计规范**，供编码智能体据此完成全部前端开发工作。
> **文档目标**：把当前功能可用但体验粗糙的前端，升级为一套**面向商业银行 DBA / 开发 / 审计人员、专业、优雅、高效、一致**的企业级管理台。
> **约束前提**：① 后端 API 已稳定，**不改后端**，前端只对接现有接口；② 必须**纯内网可用**（无外网 CDN、无外网构建依赖）；③ 保留现有认证与 RBAC 机制。
> **版本**：面向前端 V3.0；对应后端 V2.1（77 条规则 / MySQL 元数据库 / 多实例连接注册表）。

## 第 0 章 如何使用本文档（给编码智能体）

1. **先读第 1~4 章建立全局认知**，再按第 5 章逐页实现，第 3 章的设计令牌（Design Tokens）与组件规范是**全程强制遵循的基准**。
2. **不要臆造后端接口**。所有可用接口、请求/响应字段见[附录 A](#附录-a-全量-api-对接映射表)，与后端 `backend/api/*.py` 一一对应。若某页面需要的数据后端确实没有，本文档会显式标注「【后端暂无，前端降级处理】」。
3. **RBAC 前端只做「可见性 + 禁用态」，真正的鉴权在后端**。前端根据 `authState.role` 控制菜单和按钮显隐（见[附录 B](#附录-b-角色--菜单可见性矩阵)），但即使前端误放行，后端中间件仍会返回 403，前端需统一捕获并友好提示。
4. **保持纯内网**：所有第三方库继续放在 `frontend/static/vendor/`，禁止引入任何 `https://` 外链；字体、图标、样式全部本地化。
5. **交付形态**：见[第 8 章](#第-8-章-技术实现方案)。允许两种实现路径（保持单文件增强 / 升级为工程化构建），文档对两者都给出要求，**推荐工程化路径**。

## 第 1 章 现状诊断：为什么必须重做

### 1.1 现状概述

当前前端是**单个 `frontend/index.html`（约 2430 行）**，Vue 3 全局构建版 + Element Plus + ECharts，全部资源已本地化（这点很好，必须保留）。它实现了 8 个页面：审核治理概览、SQL 审核、文件审核、慢 SQL 分析、EXPLAIN 分析、TDSQL 管理、审核规则、用户管理。功能可用，但在**一致性、信息架构、专业观感、操作效率、能力完整性**五个维度都存在系统性问题。

### 1.2 十二项核心体验问题（均有代码证据）

| # | 问题 | 证据 / 量化 | 影响 |
|---|------|-------------|------|
| P1 | **设计语言不统一**：Element Plus 组件与手写原生控件混用 | 原生 `<input>` 21 个 / `<select>` 6 个 / `<textarea>` 4 个，而 `el-input` 仅 9 个、`el-select` 仅 2 个 | 同一表单里控件高度、圆角、focus 态、校验风格都不一致，显廉价 |
| P2 | **硬编码内联样式泛滥，无设计令牌** | `style="..."` 出现 **161 处**；颜色值 `#409eff/#f56c6c/#606266` 等散落全文件 | 无法统一换肤、维护成本极高、视觉不一致 |
| P3 | **完全没有响应式** | `@media` 查询 **0 处**；`.sidebar` 固定 220px、`.stat-cards` 固定 4 列、慢SQL扫描表单固定 7 列栅格 | 笔记本小屏 / 分屏下布局溢出错乱 |
| P4 | **加载态原始**：多数用文字「加载中...」 | 全站仅 1 处 `v-loading`，其余是 `<div class="empty-state">加载中...</div>` | 无骨架屏，闪烁、跳动，感知慢 |
| P5 | **分页器实现不一致** | 慢SQL记录用**手写按钮分页**（首页/上一页/数字/下一页/末页），文件报告和扫描任务用 `el-pagination` | 同一产品两套分页交互 |
| P6 | **导航是「功能堆列」而非「任务组织」** | 侧边栏 8 个 emoji 平铺，无分组、无层级、无面包屑 | 找功能靠记忆；新用户无从下手 |
| P7 | **连接模型与后端能力矛盾**：前端仍是「单一全局当前连接」 | `tdsqlStatus.connected` 单例状态贯穿全站；而后端是**多实例连接注册表**，支持数百实例并存 | 用户一次只能面对「一个已连接实例」，而银行场景要在几百个库之间频繁切换——这是最伤的效率问题 |
| P8 | **信息密度失衡**：慢SQL/扫描任务/连接用「卡片流」堆叠 | `analysis-card` 逐条竖排，无表格、无排序、无列对齐 | 几十上百条记录时无法快速扫读、对比、排序 |
| P9 | **反馈与危险操作不规范** | 删除类操作仅 4 处 `ElMessageBox.confirm`，部分删除直接执行 | 误删风险；操作结果反馈时有时无 |
| P10 | **文案与帮助信息塞进主界面** | 慢SQL扫描页把「TDSQL 架构说明」「数据源说明」「Proxy 配置」等大段说明常驻在表单下方（约 6 个说明块） | 高频操作被低频说明淹没，界面拥挤 |
| P11 | **无顶栏、无全局上下文**：没有 Header，没有面包屑，没有全局实例/项目切换器，没有全局搜索 | 布局只有「侧边栏 + 内容区」 | 用户不知道「我现在在哪、连的哪个库、属于哪个项目」 |
| P12 | **视觉观感偏「后台工具」而非「银行级平台」** | 深色侧边栏 `#304156` + emoji 图标 + 蓝色到底，无品牌感、无留白节奏、无层次阴影体系 | 与「商业银行生产系统」的专业预期不符 |

### 1.3 能力暴露缺口（最严重的产品问题）

**后端有 90+ 个 API、11 个业务域，前端只对接了约 40%。** 以下**后端已实现、前端完全没有入口**的能力，导致大量功能只能靠 `curl` 使用——这对银行运维是不可接受的：

| 后端能力 | 接口前缀 | 前端现状 | 业务价值 |
|----------|----------|----------|----------|
| **项目管理** | `/api/v1/projects` | ❌ 无页面 | 规则集/门禁绑定的载体，多团队隔离的基础 |
| **规则集管理（多租户）** | `/api/v1/rulesets` | ❌ 无（规则页只读展示 77 条） | 开发/测试/生产差异化规则，核心治理能力 |
| **质量门禁** | `/api/v1/gate/*` | ❌ 无 | strict/normal/loose 策略、发版卡点 |
| **数据保留策略** | `/api/v1/admin/retention` | ❌ 无 | 数据生命周期治理 |
| **操作审计日志** | `/api/v1/admin/operation-logs` | ❌ 无 | **审计员角色的核心场景在前端根本不存在** |
| **系统信息** | `/api/v1/admin/info` | ❌ 无 | 安全配置状态自检 |
| **定时扫描计划** | `/api/v1/tdsql/scan-schedules` | ❌ 无 | 数百实例的自动化扫描编排（否则只能手动逐个扫） |
| **调度器状态** | `/api/v1/tdsql/scheduler/status` | ❌ 无 | 运维可观测 |
| **字符集一致性检查** | `/api/v1/tdsql/check/charset` | ❌ 无 | 诊断隐性乱码/排序规则问题 |
| **大表检查 / 大表治理** | `/api/v1/tdsql/check/large-tables`、`/api/v1/bigtable/*` | ❌ 无 | L1/L2/L3 分级、分区水位、变更管控 |
| **元数据增强审核** | `/api/v1/tdsql/audit/with-metadata` | ❌ 无 | 用真实分片键精确判定分布式规则 |
| **SET 发现** | `/api/v1/tdsql/sets` | ❌ 无 | 分布式实例拓扑 |
| **监控告警** | `/api/v1/monitor/*` | ❌ 无 | 阈值告警规则与活跃告警 |
| **巡检管理** | `/api/v1/inspection/*` | ❌ 无 | 例行体检任务与结果 |
| **慢SQL状态流转** | `/api/v1/slow-queries/{id}/status` | ❌ 无 | 标记「已优化/已忽略」，闭环治理 |
| **慢SQL详情** | `/api/v1/slow-queries/{id}` | ❌ 无 | 单条深钻 |
| **设为默认连接** | `/api/v1/tdsql/connections/{id}/set-default` | ❌ 无 | 多实例默认项 |
| **PDF 报告导出** | `/api/v1/audit/report/{id}/export`、`/slow-report/{id}/export` | ❌ 无 | 单条审核/慢SQL 的正式报告 |

> **结论**：V3.0 前端不只是「美化」，而是要把这套已经很强大的后端能力**完整、有组织地呈现出来**，同时把交互体验提升到银行级专业水准。

## 第 2 章 设计理念、目标用户与设计目标

### 2.1 目标用户与核心任务流（Persona → Jobs）

平台有 4 类角色（后端 RBAC 已定义），前端体验必须围绕他们的**高频任务动线**来组织，而不是围绕「后端有哪些接口」。

| 角色 | 画像 | 高频任务（前端必须让这些动线 ≤ 3 步可达） |
|------|------|-------------------------------------------|
| **系统管理员 admin** | 平台运维负责人 | 开通/停用用户、配置数据保留、查操作审计、看系统健康 |
| **数据库管理员 dba** | 管几十上百套库 | 接入/切换实例、配置定时扫描、看慢SQL并标记处置、定制规则集与门禁、跑字符集/大表体检 |
| **开发人员 developer** | 提交 SQL 的业务开发 | 贴一段 SQL 秒级审核、看违规和修复建议、按项目规则审核、EXPLAIN 一条慢查询 |
| **审计员 auditor** | 合规审计岗 | 全局只读：查审核历史、查操作日志、看治理概览、导报告 |

### 2.2 设计原则（贯穿全文档）

1. **一致性优先（Consistency First）**：一套设计令牌、一套组件库、一套交互模式。同类操作在任何页面长得一样、行为一样。禁止再手写原生表单控件。
2. **任务导向的信息架构（Task-Oriented IA）**：导航按「业务域 + 角色」分组，不是功能平铺。用户带着任务来，3 步内完成。
3. **上下文常驻（Context Always Visible）**：顶栏永远显示「当前实例 / 当前项目 / 当前用户」，用户永远知道自己在对哪个库、哪个项目操作。
4. **数据即主角（Data First）**：列表用表格（可排序、可筛选、列对齐、密度可调），把说明性文字收进 Tooltip / 抽屉 / 帮助气泡，主界面留给数据和操作。
5. **渐进披露（Progressive Disclosure）**：概览给结论，详情按需展开（抽屉/弹窗/深钻）。复杂参数（如 processlist 轮询）默认折叠。
6. **反馈明确、危险可控（Clear Feedback, Safe by Default）**：每个操作都有加载态、成功/失败反馈；破坏性操作一律二次确认并标红。
7. **专业克制的美学（Professional & Restrained）**：银行级 = 稳重、清晰、留白、层次；少用 emoji 作正式图标，改用统一线性图标；克制用色，强调色只服务于「状态语义」。
8. **可访问与响应式（Accessible & Responsive）**：满足键盘操作、对比度、1280~1920 宽度自适应（银行常见分辨率）。

### 2.3 可衡量的设计目标

| 维度 | 现状 | V3.0 目标 |
|------|------|-----------|
| 内联样式 `style=` | 161 处 | ≤ 20 处（仅动态计算样式），其余走 token/class |
| 原生表单控件 | 31 个 | 0 个（全部 Element Plus） |
| 响应式断点 | 0 | ≥ 3（1280/1440/1920 自适应，≥1280 不横向滚动） |
| 后端能力前端覆盖率 | ~40% | ≥ 95%（除纯机器接口如 webhook） |
| 「切换实例」操作路径 | 需进「TDSQL管理」页点连接 | 顶栏切换器，任意页面 1 步 |
| 列表可排序/筛选 | 慢SQL/连接为卡片流不可排序 | 全部列表化，关键列可排序、可筛选 |
| 危险操作二次确认覆盖 | 部分 | 100% |

## 第 3 章 设计系统（Design Tokens 与组件规范）

> 本章是**全站视觉与组件的唯一真理来源**。所有页面必须引用这里定义的令牌，禁止再写魔法数值和魔法色值。实现方式见 [8.4](#84-设计令牌落地方式)。

### 3.1 色彩体系

采用「品牌主色 + 语义色 + 中性灰阶」三层结构。以稳重的**深蓝**为品牌色（银行金融感），强调色只用于状态语义。

**品牌主色（Primary）**
```
--brand-50:  #eef4ff
--brand-100: #d9e6ff
--brand-500: #2563eb   /* 主色：按钮、链接、选中、关键强调 */
--brand-600: #1d4ed8   /* 主色 hover */
--brand-700: #1e40af   /* 主色 active / 深色顶栏可选 */
```

**语义色（Semantic — 严格对应审核严重级别，全站统一）**
```
--success-500: #16a34a   /* 通过 / 已优化 / 健康 */    浅底 --success-50: #f0fdf4
--warning-500: #d97706   /* WARNING / 待处理 / 关注 */  浅底 --warning-50: #fffbeb
--danger-500:  #dc2626   /* ERROR / 拦截 / 危险操作 */  浅底 --danger-50:  #fef2f2
--info-500:    #64748b   /* INFO / 中性提示 */          浅底 --info-50:    #f8fafc
```
> **强约束**：审核严重级别 ERROR/WARNING/INFO 的配色必须全站唯一：ERROR→danger、WARNING→warning、INFO→info。规则严重级别徽标、慢SQL 级别、违规条目左边框、统计卡片数字，全部复用这三色，不得各写各的。

**中性灰阶（Neutral — 文本、边框、背景）**
```
--text-primary:   #1e293b   /* 主文本（标题、正文） */
--text-regular:   #475569   /* 次级文本 */
--text-secondary: #94a3b8   /* 辅助文本、占位、元信息 */
--border-base:    #e2e8f0   /* 常规边框、分割线 */
--border-light:   #f1f5f9   /* 浅分割线 */
--fill-base:      #f8fafc   /* 卡片内浅底、表头背景 */
--bg-page:        #f5f7fa   /* 页面背景 */
--bg-container:   #ffffff   /* 卡片/容器背景 */
```

**深色侧边栏 / 顶栏（可选深色导航）**
```
--nav-bg:        #0f1e34   /* 侧边栏底色（比现在的 #304156 更深沉稳重） */
--nav-bg-active: #1c3050   /* 选中项底色 */
--nav-text:      #b9c4d4
--nav-text-active: #ffffff
--nav-accent:    #4c8dff   /* 选中项左侧指示条/图标高亮 */
```

> **深色模式（Dark Mode）**：作为**可选增强**（第 9 章 P2 阶段）。若实现，用 `:root[data-theme=dark]` 覆盖上述令牌，Element Plus 2.7 已内置暗色变量，接入其 `dark/css-vars` 即可。

### 3.2 字体排印（Typography）

```
--font-family: "Microsoft YaHei", "PingFang SC", "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
--font-mono:   "JetBrains Mono", "Cascadia Code", Consolas, "Courier New", monospace;  /* SQL/代码/指标 */
```
字号阶梯（rem 基准 16px；建议直接用 px 值保持精确）：
```
--fs-h1: 20px / 600 / 行高28   页面标题
--fs-h2: 16px / 600 / 行高24   卡片标题、区块标题
--fs-h3: 14px / 600 / 行高22   小节标题
--fs-body: 14px / 400 / 行高22 正文
--fs-sm: 13px / 400 / 行高20   次级信息、表格内容
--fs-xs: 12px / 400 / 行高18   元信息、时间戳、辅助说明
```
> SQL 文本、指纹、执行计划、耗时/行数等**数字与代码一律用 `--font-mono`**，保证对齐与可读。

### 3.3 间距、圆角、阴影、层级

**间距标度（8px 基准）**：`--sp-1:4px --sp-2:8px --sp-3:12px --sp-4:16px --sp-5:20px --sp-6:24px --sp-8:32px`。卡片内边距统一 `--sp-5`(20px)，卡片间距 `--sp-4`(16px)，表单项间距 `--sp-4`。

**圆角**：`--radius-sm:4px`（控件）`--radius-md:8px`（卡片）`--radius-lg:12px`（弹窗/大容器）`--radius-full:999px`（徽标/头像）。

**阴影（层次感，替代现在到处一样的 `0 2px 12px`）**：
```
--shadow-card:    0 1px 3px rgba(15,23,42,.06), 0 1px 2px rgba(15,23,42,.04);
--shadow-raised:  0 4px 12px rgba(15,23,42,.08);   /* hover 卡片 */
--shadow-overlay: 0 12px 32px rgba(15,23,42,.16);  /* 弹窗/抽屉/下拉 */
```

**z-index 层级规范**：`内容 0 / 吸顶栏 100 / 下拉浮层 1000 / 抽屉 1500 / 弹窗 2000 / 全局 Loading 2500 / 消息 Toast 3000`。

### 3.4 组件规范（全站统一，强制）

> 全部基于 **Element Plus 2.7**（已本地化）。下述是**用法约定**，不是重新造轮子。

| 组件 | 规范 |
|------|------|
| **按钮 Button** | 主操作 `type=primary`；次操作默认；危险操作 `type=danger`（且危险操作必带二次确认）。一个视图内**只允许一个 primary 主按钮**。图标按钮必带 `title`/Tooltip。尺寸：页面级 `default`，表格行内 `small`，链接式操作用 `link`。 |
| **表单 Form** | 一律 `el-form` + `el-form-item`，`label-position=top`（窄栏）或 `right`（宽栏），统一 `el-input/el-select/el-date-picker/el-input-number/el-switch`。**禁止原生 input/select/textarea**。必填项显示红色星号并做前端校验（`el-form` rules）。 |
| **表格 Table** | 列表数据一律 `el-table`：斑马纹 `stripe`、关键数值列 `sortable`、超长文本列 `show-overflow-tooltip`。右侧固定「操作」列。空数据用 `el-empty`。加载用 `v-loading`（带描述）。分页统一 `el-pagination`（`layout="total, sizes, prev, pager, next, jumper"`，居右）。**全站分页只用这一种。** |
| **卡片 Card** | 统一 `el-card`（`shadow=never` + 自定义 `--shadow-card`），标题区左侧 3px 主色竖条（保留现有语义，但走 token）。 |
| **标签 Tag / 徽标** | 状态用 `el-tag`：ERROR→danger、WARNING→warning、INFO→info、通过/已优化→success、待处理→warning、已忽略→info。尺寸 `small`。 |
| **对话框 Dialog / 抽屉 Drawer** | 简单表单/确认用 `el-dialog`（≤560px）；详情深钻、长内容用 `el-drawer`（右侧滑出，宽 `40%~50%`）。慢SQL详情、连接详情、规则集编辑、扫描任务详情**用抽屉**，避免弹窗套弹窗。 |
| **消息 Message** | 操作反馈统一 `ElMessage`（success/warning/error）；全局阻断错误用 `ElNotification`；二次确认统一 `ElMessageBox.confirm`（危险操作 `type=warning`，确认按钮标红）。**保留现有 ElMessage 用法并补齐所有缺失反馈的操作**。 |
| **空状态 Empty** | 统一 `el-empty`，带一句引导性文案 + （可选）一个引导操作按钮（如「暂无连接，去新建」直接跳转）。 |
| **加载 Loading** | 首屏/区块用**骨架屏 `el-skeleton`**；表格/卡片区块用 `v-loading`（`element-loading-text` 给文案）；按钮用 `:loading`。**消灭所有「加载中...」纯文字**。 |
| **提示 Tooltip / Popover** | 帮助说明、字段解释、截断文本，一律 `el-tooltip`；较长的帮助内容用 `el-popover` 点击展开。**把主界面的大段说明块收进这里。** |
| **面包屑 Breadcrumb** | 顶栏下方 `el-breadcrumb`，反映「业务域 / 当前页 / 详情」层级。 |
| **步骤 Steps** | 多步流程（如首次接入实例引导、扫描配置）可用 `el-steps`。 |

### 3.5 图标体系

- 用 **Element Plus Icons（已本地化 `element-plus-icons.iife.min.js`）** 作为**唯一图标来源**，全站线性风格统一。
- **导航与正式功能区不再用 emoji**（🔍📋🐌 等），改用线性图标（如 Odometer/Document/DataAnalysis/Coin/Setting/User/List/Monitor 等）。emoji 仅可用于「非正式的空状态插画」或「报告类装饰」。
- 图标 + 文字组合时保持 `--sp-2`(8px) 间距、图标 16px、垂直居中。

│          │                                                          │

- **顶栏 TopBar（新增，核心）**：左侧平台标识；中部两个全局切换器（实例、项目）；右侧告警铃铛（红点数）、主题切换（可选）、用户菜单（头像 + 姓名 + 角色 tag，下拉含「修改口令 / 退出」）。

### 4.3 全局实例切换器（TopBar — 最重要的改造）

这是解决问题 P7 的核心。把「单一全局当前连接」升级为「**全局实例选择器 + 每页数据随选中实例联动**」：

- 顶栏放一个 `el-select`（带搜索 `filterable`），选项来自 `GET /api/v1/tdsql/connections`（返回 `id/name/host/port/database/is_default/active` 等）。选项显示：`● 连接名 (host:port/db)`，`●` 用绿/灰点表示 `active`（是否已建立活跃连接）。
- 选中一个实例即成为「**当前工作实例**」，存入前端全局状态（`store.currentConnectionId`）与 `localStorage`，**所有需要目标库的页面**（慢SQL扫描、EXPLAIN、数据库体检、大表、元数据增强审核）自动带上 `connection_id` 参数，无需每页再选。
- 若选中的实例尚未 `active`，切换时前端调用 `POST /api/v1/tdsql/connections/{id}/connect` 建连（带 loading 与结果提示）；失败则明确提示（连接超时/认证失败等，后端已返回中文错误）。
- 切换器旁显示当前实例的连接状态点（绿=已连/灰=未连/红=连接失败），替代现在侧边栏底部那个模糊的「已连接/未连接」。
- **默认实例**：首次加载时若无历史选择，自动选 `is_default=true` 的实例。支持在实例管理页「设为默认」（`POST /connections/{id}/set-default`，当前前端缺失）。

> 这样，DBA 在「几百个库」之间切换只需顶栏一步，且切换后当前页数据自动刷新——直接消灭 P7。

### 4.4 全局项目切换器（TopBar）

- 顶栏第二个 `el-select`，选项来自 `GET /api/v1/projects`。选中「当前项目」存入全局状态。
- 影响：即时审核 / 文件审核默认带上 `project_id`（触发该项目绑定的**规则集覆盖 + 质量门禁评估**，后端 `audit/sql`、`audit/file` 已支持 `project_id`）。
- 允许选「（不绑定项目）」= 用默认规则、不评估门禁。
- 无项目管理权限的角色（developer/auditor）只读选择；admin/dba 可在「项目管理」页维护。

### 4.5 角色可见性

菜单与操作按钮按 `authState.role` 显隐，详见[附录 B](#附录-b-角色--菜单可见性矩阵)。原则：**能看见的都能点，点了没权限的由后端 403 兜底并前端友好提示**；但为体验，尽量不给用户展示他无权使用的入口。

## 第 5 章 逐页设计规范

> 每页给出：目标、布局、组件、状态、对接 API、以及相对现状的关键改进。字段名与[附录 A](#附录-a-全量-api-对接映射表)一致。

### 5.1 登录页

- **目标**：稳重专业的第一印象，替代当前居中小白盒。
- **布局**：左右分栏（宽屏）——左侧品牌区（深蓝渐变背景 `--nav-bg`→`--brand-700`，平台名、一句话定位「银行级 SQL 质量管控与慢查询治理平台」、版本号、底部备案/版权位），右侧登录卡（居中，宽 400px）。窄屏时品牌区收起为顶部条。
- **组件**：`el-form`（用户名/口令，`el-input` 带前缀图标 User/Lock，口令 `show-password`），主按钮全宽「登 录」（`:loading`），回车提交。错误信息用 `el-alert`（type=error）而非现在的裸文字。
- **首次登录**：若 `must_change_password=true`，登录后**强制弹出修改口令对话框**（当前逻辑已有，保留并用 `el-dialog` 承接，不可关闭直到改完）。
- **对接**：`POST /api/v1/auth/login` → 存 token；`GET /api/v1/auth/me` 恢复会话。

### 5.2 工作台 / 治理概览（Dashboard）

- **目标**：一屏给出「今天平台干了什么、有什么风险要处理」，是登录后的默认落地页。
- **对接**：`GET /api/v1/dashboard/summary`（返回 `audit.{today_count,today_passed,today_failed,today_pass_rate,today_violations,today_errors,today_warnings}`、`slow_queries.{pending,critical_count,optimized,top3_time[]}`、`recent_audits[]`、`rules.{total,by_category}`）、`GET /api/v1/dashboard/audit-trend?days=7`、`GET /api/v1/dashboard/rule-stats`。
- **布局（栅格自适应）**：
  1. **KPI 卡片行**（4 张，`el-row :gutter` + 响应式 `:span`）：今日审核量 / 今日通过率 / 今日发现违规 / 待处理慢SQL。每张卡：大数字（`--font-mono`，语义色）+ 副标题（分解数据）+ 迷你趋势/环形（可选 ECharts sparkline）+ 点击可跳转对应明细页（如「待处理慢SQL」→ 慢SQL记录页并预置 `status=pending`）。
  2. **图表行**（左右各半，响应式下堆叠）：左「近7天审核趋势」ECharts 折线（通过/拦截双线）；右「高频违规规则 TOP5」条形榜（rule_id + 严重级别 tag + 命中次数），点击某规则跳转到「审核规则库」并高亮该规则。
  3. **明细行**（左右各半）：左「Top3 高耗时慢SQL」（列表，点击进详情抽屉）；右「今日审核记录」（列表，通过/未通过 tag + 类型 + 计数 + 时间，点击进详情）。
- **改进点**：KPI 卡可点击下钻；图表加载用 `el-skeleton`；空数据用 `el-empty` 而非纯文字；ECharts 主题色改用设计令牌语义色。
- **角色差异**：auditor 同样可见（只读）；developer 可只看与审核相关的卡片。

### 5.3 即时审核（单条 SQL）

- **目标**：开发者最高频入口，追求「贴上就出结果」的顺滑。
- **布局**：上「SQL 输入区」下「结果区」，左右可选「审核选项侧栏」。
  - **输入区**：用**代码编辑器体验的 `el-input type=textarea`**（等宽字体、行高、`autosize` min 8 行）。顶部工具条：语言提示、`el-select` 选择「审核项目」（默认取顶栏当前项目）、开关「启用元数据增强」（开启则调 `audit/with-metadata` 并带当前实例 `connection_id`；关闭走 `audit/sql`）、示例下拉（SELECT/CREATE/UPDATE/DELETE 示例）、清空。
  - **主按钮**：「开始审核」（`Ctrl/Cmd+Enter` 快捷键触发）。
- **结果区**：
  - 顶部**结论横幅**（`el-alert` 或自定义 banner）：通过=success，未通过=error，显示 SQL 类型、违规计数（ERROR/WARNING/INFO 分别计数用对应 tag）。
  - 若绑定了项目并评估了门禁，展示**门禁结果**（`gate_result`：passed / blocked_by / detail），未通过门禁用醒目红色条。
  - **违规列表**：每条违规卡片——左侧严重级别色条 + `[规则ID]` + 级别 tag + message + 折叠的「💡 修复建议」。规则ID 可点击 → 弹 `el-popover` 显示该规则完整规范（来自规则库）。
  - **导出**：提供「导出报告」按钮（若该次审核已落库有 `report_id`，走 `GET /api/v1/audit/report/{id}/export` 下载 PDF）。
- **对接**：`POST /api/v1/audit/sql`（body: `{sql, project_id, }`）或 `POST /api/v1/tdsql/audit/with-metadata`（body: `{sql, connection_id}`）。
- **改进点**：元数据增强、项目门禁、规则详情弹窗、快捷键、报告导出——均为当前缺失。

### 5.4 文件审核

- **目标**：批量审核 SQL 文件 / MyBatis XML，并留存报告。
- **布局**：`el-tabs` 两页签（保留现有「文件上传 / 审核报告」结构，但组件化）。
  - **上传页**：`el-upload`（拖拽 `drag`，`accept=.sql,.xml`，限制大小并提示），替代手写拖拽区。上传后展示汇总条（总数/通过/未通过/通过率，用 `el-descriptions` 或统计条）+ 逐条结果（可折叠 `el-collapse`，每条含 SQL 原文查看抽屉、违规列表）。绑定顶栏项目 → 触发项目规则集与门禁。
  - **报告页**：`el-table`（文件名、SQL数、通过/失败、通过率 tag、创建人、时间、操作「下载HTML报告」），`el-pagination`。对接 `GET /api/v1/audit/file-reports`、`GET /api/v1/audit/file-reports/{id}/html`。
- **改进点**：`el-upload` 替代手写拖拽；结果用 `el-collapse` 收纳；SQL 原文用抽屉；报告列表用标准表格。

### 5.5 慢 SQL 治理（拆成 3 个子页面 + 详情抽屉）

当前「慢SQL分析」把扫描任务、手动分析、记录塞在一个页的三个 tab 里，且塞满说明文字。V3.0 拆分并瘦身：

#### 5.5.1 扫描任务（子页）
- **目标**：对**当前实例**发起一次慢SQL抓取，并管理历史任务。
- **布局**：
  - **顶部操作条**：一个「+ 新建扫描」主按钮 → 打开**配置抽屉**（而非把 7 列表单铺在页面上）。抽屉内：数据源（digest/processlist 单选卡片，各带一句说明 Tooltip）、时间窗口（`el-date-picker` 范围选择，datetime）、耗时阈值（`el-input-number`）、抓取条数（`el-input-number`）；processlist 模式下**渐进展开**轮询参数（时长/间隔）。当前实例信息只读展示在抽屉顶部。「所有大段架构说明」收进抽屉里的一个 `el-collapse`「关于数据源」折叠块 + 字段级 Tooltip。
  - **任务列表**：`el-table`（任务名、数据源、库、抓取条数、时间窗口、创建人、创建时间、操作）。支持多选 → 批量删除（`ElMessageBox.confirm`）；行操作「查看慢SQL」（跳记录页并预置 `scan_task_id`）、「跨SET分析」（打开分析抽屉）、「删除」。分页 `el-pagination`。
  - **跨SET对比分析**：从卡片区改为**点击任务后在抽屉/弹窗展示**（SET 分布、热点 SET、跨SET共现、优化建议）。对接 `GET /api/v1/slow-queries/cross-set-analysis?scan_task_id=`。
- **对接**：`POST /api/v1/tdsql/slow-queries/fetch`、`GET /slow-queries/scan-tasks`、`DELETE /slow-queries/scan-tasks/{id}`、`GET /tdsql/proxy-config`（Proxy 慢日志配置收进抽屉的说明区，不再常驻主界面）。
- **改进点**：表单入抽屉、说明入折叠/Tooltip、卡片流→表格、跨SET分析按需展示。

#### 5.5.2 慢SQL 记录（子页）
- **目标**：多维筛选、排序、扫读，并对每条慢SQL做处置闭环。
- **布局**：
  - **筛选栏**：用 `el-form inline` + `el-select/el-input`（库、SET、扫描任务、级别、状态、关键词）替代现在的原生 select 栏；「筛选/重置」按钮；右侧「清理无任务关联记录」（危险操作二次确认）。
  - **数据表**：`el-table`（严重级别 tag、SQL指纹（等宽截断+Tooltip全文）、平均耗时（sortable，`--font-mono`）、执行次数（sortable）、扫描行数、库、SET、状态 tag、末次执行时间、操作）。默认按平均耗时降序。
  - **行操作**：「详情」→ 详情抽屉；「标记状态」→ 下拉（待处理/已优化/已忽略，调 `PUT /slow-queries/{id}/status`）；「导出报告」（`GET /api/v1/audit/slow-report/{id}/export` PDF）。
  - **分页**：`el-pagination`（替代手写分页）。
- **慢SQL 详情抽屉**：对接 `GET /api/v1/slow-queries/{id}`。展示：完整指纹/SQL（等宽、可复制）、执行统计（次数/平均/最大/锁等待/扫描/返回）、执行者（user/host）、首末次执行时间、`analyses[]`（问题类型/证据/根因/建议/优化SQL，每条一张分析卡）、状态流转操作、「去 EXPLAIN 分析这条」快捷入口（带 SQL 跳 EXPLAIN 页）。

## 第 6 章 交互、反馈与状态规范

### 6.1 加载态
- **首屏/整页**：`el-skeleton`（骨架屏），结构贴近真实内容（KPI 卡骨架、表格骨架）。
- **区块/表格**：`v-loading` + `element-loading-text`（如「正在扫描慢SQL，请稍候…」）。
- **按钮**：`:loading`，禁用重复提交。
- **长任务**（processlist 轮询扫描，可能几十秒）：按钮 loading + 提示预计时长；有条件时展示进度或「扫描中…已用 Ns」。
- **强制**：删除所有「加载中...」纯文字。

### 6.2 空状态
- 一律 `el-empty`，文案具体 + 引导操作。例：实例列表空 → 「还没有接入任何 TDSQL 实例」+ 按钮「+ 新建连接」；慢SQL记录空 → 「当前筛选无记录，去发起一次扫描」+ 跳转。

### 6.3 错误处理
- **业务错误**（4xx）：`ElMessage.error(后端 detail)`，后端已中文化（如「数据库用户名或密码错误」）。
- **权限错误**（403）：统一拦截，`ElMessage.warning('当前角色无权执行该操作')`，不弹裸 403。
- **认证失效**（401）：清 token + 跳登录（`apiFetch` 已有 `onUnauthorized`，保留）。
- **网络/5xx**：`ElNotification.error('服务暂时不可用，请稍后重试')` + 保留 request-id（响应头 `X-Request-ID`）便于排查，可在通知里附上。
- **全局兜底**：封装 `apiFetch`，统一解析 `{code,message,detail}`，页面只处理业务分支。

### 6.4 表单校验
- 用 `el-form` rules 做**前端即时校验**：必填、格式（host/port/口令强度规则前端预校验，与后端一致：≥8 位、含大小写/数字/特殊字符三类）。
- 提交前校验不通过时聚焦第一个错误项。

### 6.5 确认与危险操作
- 删除连接、删除项目、删除规则集、删除用户、批量删除扫描任务、清理无关联记录、立即清理数据、断开连接 —— **全部 `ElMessageBox.confirm`**，`type=warning`，确认按钮 `type=danger`，文案写清后果与数量（如「将删除 3 个扫描任务及其下慢SQL记录，不可恢复」）。

### 6.6 快捷键（提效）
- 即时审核：`Ctrl/Cmd + Enter` 提交审核。
- 全局：`/` 聚焦搜索（若实现全局搜索）；`Esc` 关闭抽屉/弹窗。

### 6.7 长文本与代码
- SQL/指纹/执行计划：等宽字体、可横向滚动容器、一键复制（`ElMessage` 提示已复制）、超长在表格内截断 + Tooltip 全文 + 详情抽屉看完整。

## 第 8 章 技术实现方案

### 8.1 两条实现路径（推荐路径 B）

**路径 A（低风险增量）**：保持单文件 `frontend/index.html` + Vue 全局构建，但**必须**：抽出全站 CSS 到 `frontend/static/css/app.css`（含设计令牌 `:root{--brand-500:...}`）、抽出 JS 到 `frontend/static/js/` 多个模块（用原生 ES Module，`<script type="module">`），按页面拆分组件为 `Vue.defineComponent` 局部注册。适合快速交付、纯内网零构建。

**路径 B（推荐，工程化）**：升级为 **Vite + Vue 3 SFC + Element Plus + Pinia + Vue Router** 工程，**本地构建产物**输出到 `frontend/static/`，由 FastAPI 继续托管（`main.py` 已挂 `/static` 与根路由，产物仍是纯静态、纯内网）。收益：组件化、类型友好、设计令牌与主题可控、可维护性强。
- **纯内网约束**：依赖在有网环境 `npm ci` 安装并 `npm run build` 出静态产物，产物不含任何运行时外链；提交构建后的 `frontend/static/` 到仓库。开发者无需在内网 `npm install`。
- 若选路径 B，路由用 `createWebHashHistory` 或让 FastAPI 对未知路径回落 `index.html`（避免刷新 404）。

> **无论哪条路径，都必须落地第 3 章设计令牌、第 4 章布局骨架、第 5 章页面规范、第 6 章交互规范。**

### 8.3 状态管理（Pinia store）
- `useAuthStore`：token、当前用户、role、权限判断 `can(action)`。
- `useConnectionStore`：`connections[]`、`currentConnectionId`、`currentConnection`、`ensureConnected()`（切换时建连）；持久化到 localStorage。**全站需要目标库的请求从这里取 `connection_id`。**
- `useProjectStore`：`projects[]`、`currentProjectId`；审核请求从这里取 `project_id`。
- `useAlertStore`：活跃告警数（顶栏铃铛）。

### 8.4 设计令牌落地方式
- 所有令牌写在 `styles/tokens.css` 的 `:root{}`（暗色在 `:root[data-theme=dark]{}`）。
- **覆盖 Element Plus 主色**：设置 `--el-color-primary: var(--brand-500)` 及其 `light-3/5/7/9`、`dark-2` 派生（Element Plus 2.7 支持 CSS 变量覆写），使所有 el- 组件主色统一为品牌蓝。语义色同理映射到 `--el-color-success/warning/danger/info`。
- 组件里**只允许引用 `var(--...)`**，禁止硬编码色值/间距（Code Review 卡点）。

### 8.5 API 封装约定
- `request.js` 暴露 `get/post/put/del`，自动：注入 `Authorization: Bearer`、`Content-Type: application/json`、拼接 query、解析响应、统一错误分支（401/403/4xx/5xx 见 6.3）。
- 需要目标库的接口，封装层自动从 `useConnectionStore` 注入 `connection_id`（也允许显式覆盖）。
- 保留现有 `API_BASE=''` 同源相对路径策略（反向代理友好）。

### 8.6 图表
- 继续用本地化 ECharts；封装 `<TrendChart>`、`<Sparkline>` 组件，主题色取设计令牌语义色；`resize` 监听容器（响应式）。

2. 全局布局骨架：TopBar（实例切换器 + 项目切换器 + 用户菜单 + 告警铃铛）、分组侧边栏（可折叠）、面包屑。
3. API 封装层 + Pinia 四个 store + 401/403 统一处理。
4. 登录页重构。

**P1 — 存量页面规范化 + 连接模型改造（体验主要收益）**
5. 实例管理重构为「多实例表格 + 抽屉表单 + 设为默认/SET拓扑」，与顶栏切换器联动。
6. 慢SQL 治理拆 3 页：扫描任务（表单入抽屉、说明入折叠）、慢SQL记录（表格+排序+筛选+状态流转+详情抽屉+PDF）、扫描计划（全新）。
7. 即时审核（元数据增强 + 项目门禁 + 规则详情 + 快捷键）、文件审核（el-upload + collapse）、EXPLAIN（el-table + 单元格高亮）规范化。
8. Dashboard 重构（KPI 可下钻 + 骨架屏 + 图表令牌化）。
9. 系统管理整合（用户 + 数据保留 + 操作审计 + 系统信息）。

**P2 — 补齐治理能力（能力完整性）**
10. 项目管理、规则集、质量门禁（可并入项目详情）。
11. 数据库体检、大表治理。
12. 监控告警、巡检管理。

**P3 — 增强（可选）**
13. 深色模式、全局搜索、更多快捷键、可访问性打磨。

### 9.2 验收清单（Definition of Done）

- [ ] 全站 0 个原生 `<input>/<select>/<textarea>`（全部 Element Plus）。
- [ ] 内联 `style=` ≤ 20 处（仅动态计算）；色值/间距全走 `var(--...)`。
- [ ] ≥1280 宽度无横向滚动；1280/1440/1920 三档布局正常；侧边栏可折叠。
- [ ] 顶栏实例切换器可用：切换后当前页数据随之刷新；未连实例切换时自动建连并提示。
- [ ] 顶栏项目切换器可用：审核请求带 `project_id` 并展示规则集覆盖 + 门禁结果。
- [ ] 后端能力覆盖率 ≥95%：附录 A 列出的页面全部落地并联通对应接口。
- [ ] 所有列表用 `el-table`（可排序/筛选/分页统一 `el-pagination`）；所有加载态用骨架屏/`v-loading`（无「加载中...」文字）；所有空态用 `el-empty` 带引导。
- [ ] 所有破坏性操作有 `ElMessageBox.confirm` 二次确认，文案写明后果。
- [ ] 401 跳登录、403 友好提示、5xx 通知兜底，全部生效。
- [ ] 角色可见性符合附录 B；developer/auditor 看不到无权入口。
- [ ] 纯内网可运行：无任何外网请求（构建产物自查，Network 面板 0 外链）。
- [ ] 审计员用一个账号能完成：看概览、查审核历史、查操作审计日志、导出报告——全程无需 curl。
- [ ] DBA 用一个账号能完成：顶栏切实例 → 发起扫描 → 看慢SQL列表 → 标记已优化 → 配置定时扫描 → 定制规则集——全程 UI 内闭环。
