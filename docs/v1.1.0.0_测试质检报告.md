# TDSQL数据库SQL审核工具 v1.1.0.0 测试质检报告

> **报告类型**：送测质检验收报告（Smoke / SIT / UAT 全量执行）
> **被测版本**：v1.1.0.0（`git main` @ commit 8c8a7ad，`VERSION=1.1.0.0`，`/health` 返回 `1.1.0.0`）
> **测试依据**：`docs/v1.1.0.0_test_cases.md`（G智能体编写）
> **测试日期**：2026-07-21
> **测试环境**：本地 MariaDB 10.11（模拟 TDSQL 业务库 + `tdsqlpcloud_monitor` 监控库，端口 13306，`performance_schema` 开启）＋ uvicorn 后端（`AUTH_ENABLED=true`）＋ Chromium/Playwright 真实浏览器端到端驱动
> **测试方式**：后端接口逐条实测（curl/HTTP）＋ 前端浏览器真实操作（登录、点击、截图）＋ 元数据库直查核对＋源码根因定位

## 一、结论摘要

本轮对 v1.1.0.0 的 4 条冒烟、9 条 SIT、3 个 UAT 场景全部实测。**冒烟测试全部通过，系统基本可用**；但在 SIT / UAT 阶段发现 **8 项缺陷，其中 3 项 High（含 2 项安全/越权相关）**，直接导致 SIT-03、SIT-08 与 UAT 场景三无法按测试用例预期闭环。

**核心风险（必须整改）**：

1. **深度诊断 9 个子权限的"接口级"访问控制对内置角色（admin/dba/developer/auditor）完全失效**——菜单在前端隐藏了，但后端 API 不拦截，存在越权访问。这恰好是 v1.1.0.0 的主打新特性，却只做了"看不见"没做"进不去"。
2. **首次登录强制改密形同虚设**——弹窗可被 ESC / 关闭按钮 / 取消 / 点遮罩关掉，关闭后 token 依旧有效；后端也不校验 `must_change_password`，业务接口照常调用成功。
3. **即时 SQL 审核不支持多条 SQL，多语句时静默判"全部通过"（假阴性）**——对一款质量门禁工具，这是危险的漏放。

## 三、集成测试结果（SIT）

### SIT-01 废弃菜单彻底清除 —— ✅ 通过
- `visible-menus` 返回不含 `health-check`。
- 元数据库直查：`SELECT COUNT(*) FROM role_permissions WHERE menu_key='health-check'` → **0**。
- `role_permissions` 中 `menu_key` distinct 集合已无任何 `health-check` 残留。

### SIT-02 深度诊断子模块权限矩阵 —— ✅ 通过
- 权限矩阵页表头（浏览器实测 `uat_s1_perm_matrix`）：**已无"深度诊断"独立大列作为唯一入口**，且新增 9 列子权限齐全：
  `深度诊断-集群巡检 / 日常巡检与对比报告 / 索引体检 / 结构比对 / 应急诊断 / SQL分析 / 网关日志分析 / PDF报告与大屏 / 运维工具箱`。
- 4 个内置角色在 9 列下均有初始默认值（默认全 1）。
- 备注：矩阵中仍保留一列名为"深度诊断"的父级开关（与 9 个子列并列），非缺陷，但含义上与"大列已消失"表述略有出入，可留意。

### SIT-03 子菜单权限隔离生效 —— ❌ **不通过（BUG-01）**
- 步骤1-2（前端隐藏）**通过**：去掉 developer 的 `网关日志分析`+`应急诊断` 后保存成功；`role_dev` 登录进入"深度诊断"，顶部仅剩 7 个子标签（`结构比对/应急诊断/运维工具箱` 已按场景一进一步去除，实测标签为 `集群巡检/日常巡检与对比报告/索引体检/SQL分析/PDF报告与大屏`），网关/应急正确隐藏。
- 步骤3-4（**接口级 403**）**失败**：
  - developer 去掉 `deep-diag-gateway` 后，`GET /api/v1/gateway-log/reports` → **200**（应为 403）。
  - 补充验证：dba 去掉 `deep-diag-gateway` 后，`GET /api/v1/gateway-log/reports` 同样 → **200**。
  - （`POST /emergency/run` 返回 403 属"developer 本就不允许写"，非子权限拦截，不能作为通过依据。）
- **判定**：v1.1.0.0 的子权限"接口级隔离"对内置角色未生效，仅前端隐藏，存在越权风险。详见 BUG-01。

### SIT-04 日常巡检手动采集3天限制 —— ✅ 通过
- 前端浏览器实测：选择 `7-18 ~ 7-21`（4天）点"手动采集巡检数据"→ 弹警告 **"单次手动采集时间跨度不能超过 3 天……"**，请求被前端拦截未下发（截图 `uat_s2_daily_4day_warning`）。
- 选择 `7-18 ~ 7-20`（3天）→ 顺利触发，依次采集 3 天并自动比对、渲染折线图。
- 说明：限制为**前端拦截**（`app.js` `runDailyInspect` 中 `dates.length > 3` 判断），与用例"不向后端发送"一致；后端 `daily-inspect/run` 为单日接口，无跨度概念，符合设计。

### SIT-05 多日巡检趋势比对逻辑 —— ✅ 通过
- `GET /daily-inspect/compare?date1=2026-07-18&date2=2026-07-20`：`instance_diffs` 仅呈现 18 与 20 两天起止对比，含 `old_val/new_val/diff/pct_change/significant(⚠️)`。
- `GET /daily-inspect/trend?date_from=18&date_to=20`：折线数据含 **18、19、20 三天连续**曲线（浏览器 canvas 渲染成功，截图 `uat_s2_daily_after_collect`）。
- 阈值滑块联动实测：`threshold_multiplier` 0.5→1.0→3.0→10.0 时 ⚠️ 数量 34→31→20→6 动态递减；前端 5.0 时 ⚠️ 由 21 降至 13。符合预期。

### SIT-06 SQL分析平均耗时单位修正 —— ✅ 通过
- 浏览器实测"SQL调用量多维分析"表头为 **"平均耗时(ms)"**（截图 `uat_sit06_sqlstats`）。
- 数值由秒自动换算为毫秒（前端 `(row.avg_seconds*1000).toFixed(2)`）：`1.2345s→1234.50`、`0.005s→5.00`、`0.4213s→421.30` 均正确，保留两位小数。

### SIT-07 连接按钮状态切换同步 —— ⚠️ **部分通过（BUG-05 / BUG-07 / BUG-08）**
- **核心通过**：点某实例"连接"→ 弹"已连接"消息框，同时左侧"状态"列同步切换为绿点"已连接"（浏览器实测 `uat_s2_instances_after`，状态列由 `row.active` 驱动，无延迟）。
- **不符点1（BUG-07 Low）**：连接过程中按钮**未进入 loading 态**（按钮无 `:loading` 绑定，`connectInstance` 也不置 loading），与用例"按钮进入 loading 状态"不符。
- **不符点2（BUG-05 Medium）**：若先编辑实例配置（如补 monitordb 信息）再点"连接"，返回的是**旧连接池**（配置未刷新），需先"断开"再"连接"才生效。详见 BUG-05。
- **不符点3（BUG-08 Low）**：`disconnect` 只清内存池，不重置 DB `status`，API 返回 `status='connected'` 与 `active=false` 自相矛盾。

### SIT-08 新增用户强制修改密码 —— ❌ **不通过（BUG-02）**
- 步骤"必须改密"**部分通过**：新建用户 `must_change_password=1`，登录后前端弹出"首次登录请修改口令"弹窗（截图 `uat_s3_force_pwd_dialog`）；改密成功后旧口令 401、新口令 200，改密链路本身正常。
- **失败点（背景遮罩锁定 / 无法绕过）**：
  - 弹窗按 **ESC 即关闭**（`dialog still visible after Escape: False`），关闭后可正常进入"工作台"操作主功能（截图 `uat_s3_force_pwd_bypassed`）。弹窗还带"取消"按钮、右上关闭 X，均可关闭。
  - 后端 `middleware.py` 只校验 `status=='active'`，**不校验 `must_change_password`**：用 `must_change_password=1` 的 token 直接 `POST /api/v1/audit/sql` → **200 返回审核结果**。
- **判定**：强制改密可被绕过，不满足用例"未成功修改密码前无法通过任何方式绕过弹窗去操作主功能"。详见 BUG-02。

### SIT-09 离职冻结与状态切换 —— ✅ 通过
- 冻结（`status=disabled`）后 `dev_tester` 登录 → **401 "账户已被禁用"**。
- 加强验证：冻结后即便持**先前已签发的 token**，`GET /auth/me` 与业务接口均 **401 "账户不存在或已禁用"**（中间件实时校验，冻结即时生效，优于用例要求）。
- 重新启用后可正常登录。

### 场景三：Developer SQL上线审核闭环 —— ❌ 不通过

2. **多库模式（BUG-06）**：用例要求"点击多库模式开关"，但即时审核页**只有"元数据增强"开关，无"多库模式"**，此步骤无法执行。
3. **规则库 119 条**：进入"审核规则库"实测页面显示 **"共 119 条"**，8 大分类（命名/DDL/DML/索引/分布式/安全/性能/事务）齐全——**通过**（截图 `uat_s3_rules`）；"上线检查"对本地实例执行返回 12 项检查折叠面板——通过（截图 `uat_s3_schema_check`）。

## 五、缺陷详情（复现步骤 / 根因 / 证据 / 建议）

### BUG-01 【High】深度诊断子权限接口级鉴权对内置角色失效（越权）
- **关联用例**：SIT-03 步骤4 / 场景一。
- **现象**：将 developer（或 dba）的某深度诊断子权限（如"网关日志分析"）在权限矩阵置为不可见后，前端菜单/标签正确隐藏，但直接请求该子模块 API 仍返回 200，未按用例返回 403。
- **复现**：
  1. admin 登录，`PUT /api/v1/auth/role-permissions/developer`，body `{"permissions":{"deep-diag-gateway":0}}`。
  2. developer 账户登录取 token，`GET /api/v1/gateway-log/reports`。
  3. 期望 403，实际 **200**。dba 同理复现。
- **根因**：`backend/services/auth_service.py:300` `check_permission()` 中，第二级 `role_permissions` 菜单可见性校验被包在 `if role not in _BUILTIN_ROLES:` 内——**仅自定义角色**才做子权限校验；内置角色（admin/dba/developer/auditor）走到末尾直接 `return True`（`auth_service.py:312`），完全跳过 `_PATH_TO_MENU`（已含 9 个子模块→menu_key 映射）的二级校验。
- **影响**：v1.1.0.0 主打的"子模块权限颗粒化"仅是前端视觉隐藏，服务端未强制。任意持有效 token 的内置角色用户可绕过前端直接调用被"禁用"的子模块接口，构成越权。银行级 RBAC 场景属严重问题。
- **建议**：对内置角色（admin 除外）同样执行 `_PATH_TO_MENU` 二级校验——当请求路径命中某子模块前缀时，以 `get_visible_menus(role)` 判定 `menu_key` 是否可见，不可见即 403。注意保留 admin 全通过、以及各角色原有的读写粒度。

### BUG-02 【High】首次登录强制改密可被绕过
- **关联用例**：SIT-08。
- **现象**：`must_change_password=1` 用户登录后弹出的改密弹窗可被 ESC / 关闭X / 取消 / 点击遮罩关闭；关闭后 token 有效可操作主功能；后端亦不拦截。
- **复现**：
  1. admin 新建用户（`must_change_password=1`），该用户登录。
  2. 弹窗出现后按 ESC → 弹窗消失，点击左侧菜单可正常进入各页面。
  3. 或直接用该 token `POST /api/v1/audit/sql`（body `{"sql":"SELECT * FROM t1"}`）→ **200 返回审核结果**。
- **根因**：
  - 前端 `frontend/index.html:1294` 的改密 `el-dialog` 未设置 `:close-on-press-escape="false"` / `:close-on-click-modal="false"` / `:show-close="false"`，且 footer 含"取消"按钮；强制改密与自助改密复用同一弹窗，无"强制不可关"分支。
  - 后端 `backend/middleware.py:90-99` 仅校验 `user.status=='active'` 与 RBAC，**未校验 `must_change_password`**。
- **影响**：初始弱口令用户可长期不改密、以初始口令持续使用系统，违背强制改密的安全初衷。
- **建议**：
  1. 前端：强制场景下弹窗设为不可关闭（禁用 ESC/遮罩/关闭X、隐藏"取消"），未改密前锁定路由与操作。
  2. 后端（关键，防绕过）：中间件对 `must_change_password=1` 的用户，除放行 `/auth/change-password`、`/auth/logout`、`/auth/me` 外，其余业务接口一律 403，强制服务端闭环。

### BUG-03 【High】即时SQL审核不支持多条SQL，多语句静默漏报
- **关联用例**：UAT场景三 步骤1。
- **现象**：即时审核（`POST /api/v1/audit/sql`）对含多条语句的输入只解析一条，其余语句违规被漏报；两条语句时甚至返回 0 violations、`sql_type=UNKNOWN`，判"全部通过"。
- **复现**：
  - `{"sql":"SELECT * FROM t_order; UPDATE t_user SET a=1;"}` → `type=UNKNOWN, violations=[]`（**假阴性**）。
  - 单条 `SELECT * FROM t_order WHERE ...` → 正确命中 R012；单条无WHERE `UPDATE` → 命中 R013/R014/R070。
  - 前端场景三三语句输入亦仅显示 1 项 R001（截图 `uat_s3_audit_result`）。
- **根因**：`backend/services/audit_service.py:audit_single_sql` → `checker.audit_sql(sql)` → `backend/engine/parser.py:112` 使用 `sqlglot.parse_one(sql_clean)`，只解析首条语句；多语句时解析结果非预期类型（UNKNOWN），DML/DDL 规则因 `sql_type/is_create_table` 判空而整体跳过。即时审核页文案又是"粘贴SQL语句"，用户极易一次贴多条。
- **影响**：对一款"质量门禁/SQL审核"工具，多语句静默判过是危险的漏放，可能放行本应拦截的高危变更。
- **建议**：即时审核入口按分号拆分为多条（可复用已有 `split_sql_statements`），逐条 `audit_sql` 后汇总违规（带语句序号/行号）；或至少在检测到多语句时给出明确提示并引导用户逐条/走文件审核。避免"静默返回空违规"。

### BUG-04 【Medium】导出HTML对比大屏依赖公网CDN，纯内网不可用
- **关联用例**：UAT场景二 步骤4。
- **现象**：导出的对比大屏 HTML 通过 `<script src="https://cdn.jsdelivr.net/npm/chart.js">` 加载图表库，内网无外网时折线图无法渲染。
- **根因**：`backend/services/daily_inspect_service.py:746` 硬编码公网 CDN。而系统主站静态资源已全部内网化（`frontend/index.html` 引用 `/static/vendor/*`，含 echarts.min.js）。此导出报告与"纯内网部署"要求不一致。
- **影响**：目标环境（银河麒麟内网，`v1.1.0.0_upgrade_manual.md` 明确 10.243.x 内网）中，导出报告图表空白，UAT 场景二第 4 步无法达成。
- **建议**：将 Chart.js 改为内联/本地 `/static/vendor/` 引用（或复用主站已有的 echarts 生成图表），确保离线可用。

### BUG-05 【Medium】编辑已连接实例配置后不生效（返回旧连接池）
- **关联用例**：UAT场景二 步骤1 / SIT-07。
- **现象**：对已建立活跃连接的实例执行 `PUT /connections/{id}`（如补 monitordb 主机/端口/库）后，再点"连接"或做 monitor-probe，仍使用**旧配置**；须先"断开"再"连接"方生效。
- **复现**：
  1. 连接实例 `4330ff92`（活跃池已建立，旧 monitor_port=15001）。
  2. `PUT /connections/4330ff92` 更新 `monitor_port=13306` 等。
  3. `POST /connections/4330ff92/monitor-probe` → `ok=false, monitor_port=15001`（旧配置）。
  4. `POST /disconnect?connection_id=4330ff92` 后再 `connect` → `ok=true, monitor_port=13306`（新配置生效）。
- **根因**：`backend/services/connection_registry.py` `get()`（≈L97-145）命中 `self._pools[conn_id]` 即返回缓存池、不比对配置；`save_connection()` 更新 DB 后也不淘汰对应活跃池。
- **影响**：DBA 修改实例（尤其 monitordb 接入信息）后"测试连接"仍报旧结果，易误判、影响巡检/慢SQL采数。
- **建议**：`save_connection` 更新已存在连接时，若该 `conn_id` 存在活跃池则 `disconnect(conn_id)` 淘汰，令下次 `get()` 按新配置重建；或在 `get()` 内比对配置指纹不一致时重建。

### BUG-06 【Medium】即时审核页缺少"多库模式开关"
- **关联用例**：UAT场景三 步骤2。
- **现象**：用例要求"点击多库模式开关，一次性针对同一连接下多个数据库并发审核"，但即时审核页仅有"元数据增强"开关，无"多库模式"，前后端均无对应实现（`grep 多库/multiDb` 无命中）。
- **判定**：功能缺失或测试用例与实现规格不符。请 G智能体确认：是遗漏实现，还是用例需修订（该"多库"能力实为"上线检查"对全部非系统库的检查）。
- **建议**：二选一——补齐即时审核的多库并发能力并加开关；或修订用例，将"多库"步骤指向"上线检查"模块。

### BUG-07 【Low】连接按钮无 loading 态
- **关联用例**：SIT-07 步骤1。
- **根因**：`frontend/index.html:507` 连接按钮无 `:loading` 绑定；`app.js` `connectInstance` 不置 loading，仅 await 后弹 toast。
- **影响**：连接慢时无过渡反馈，用户可能重复点击。建议为 `connectInstance` 增加 per-row loading 态并绑定按钮 `:loading`。

### BUG-08 【Low】断开连接不重置 DB status，API 状态字段失真
- **关联用例**：SIT-07。
- **现象**：`disconnect` 后 `GET /connections` 该实例 `status` 仍为 `connected`，而 `active=false`，二者矛盾。
- **根因**：`backend/services/connection_registry.py:disconnect`（≈L146）仅 `pop` 内存池并 `close_all`，不 `UPDATE tdsql_connections SET status='disconnected'`。UI 用 `active` 展示故界面无碍，但 API `status` 字段误导下游/排障。
- **建议**：`disconnect` 时同步将该连接 DB `status` 置为 `disconnected`（`_mark_connected` 的反向操作）。
