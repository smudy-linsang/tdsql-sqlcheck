# 大表治理模块 —— TDSQL 分区表大小采集缺陷 修复方案

> **提出方**：开发智能体反馈"连 15005 分布式实例 proxy 采集不到分区表大小"
> **诊断/方案**：质检设计智能体
> **日期**：2026-07-12
> **状态**：根因已定位并本地验证；真机(15005)因白名单未放行本环境出口 IP 未能联调，附真机验证脚本
> **交付对象**：编码智能体（照本方案整改）

---

## 一、需求回顾

大表治理需采集 **容量超过 1GB** 的大表，覆盖三种情况：

| 编号 | 情况 | 判定 |
|------|------|------|
| ① | 单表（非分区）超过 1GB | 表 DATA+INDEX ≥ 1GB |
| ② | 分区表**单个分区**超过 1GB | 任一分区 DATA+INDEX ≥ 1GB |
| ③ | 分区表**合并后**超过 1GB | 各分区 DATA+INDEX 之和 ≥ 1GB |

---

## 二、根因（已定位 + 证据）

### 2.1 现状代码：只查 `information_schema.TABLES`

采集入口链路：
- 前端 `collectBigtable()`（`frontend/static/js/app.js`）→ `GET /api/v1/tdsql/check/large-tables`
- 端点 `backend/api/tdsql_manage.py:379` → `pool.check_large_tables()`
- 实现 `backend/services/tdsql_connector.py:664`（连接池版，**分布式实例走这里**）与 `:1022`（单连接兼容版），两处 SQL 完全一致：

```sql
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = %s AND TABLE_TYPE = 'BASE TABLE'
  AND (DATA_LENGTH + INDEX_LENGTH) >= %s        -- %s = 1GB
ORDER BY (DATA_LENGTH + INDEX_LENGTH) DESC
```

全后端 `grep PARTITIONS` 在采集逻辑中**零命中** —— 从未查过 `information_schema.PARTITIONS`。

### 2.2 TDSQL proxy 对分区表在 TABLES 里只给"壳值"

用户在 15005 上对分区表 `tdsql_check.big_audit_trail` 实测：

| 数据源 | 结果 | 换算 |
|--------|------|------|
| `PARTITIONS`（逐分区，query 1） | p0–p7 各 ~1.3GB | 真实 |
| `PARTITIONS`（SUM 聚合，query 2） | total_size 11,138,596,864 | **10.37 GB（真实）** |
| `TABLES`（query 3） | DATA_LENGTH=131072, INDEX_LENGTH=393216, TABLE_ROWS=0 | **512 KB（壳值）** |

即：TDSQL proxy 的 `information_schema.TABLES` 对分区表只返回一个约 512KB 的元数据壳值，且 `TABLE_ROWS=0`。真实物理大小只存在于 `information_schema.PARTITIONS`。

### 2.3 因果链

现状 SQL 用 `TABLES.(DATA_LENGTH+INDEX_LENGTH) >= 1GB` 过滤：
- 分区表 `524288 >= 1073741824` → **False** → 被过滤掉 → **永远采集不到**
- 兜底条件 `OR TABLE_ROWS >= 300万` 也救不了（`TABLE_ROWS=0`）

**结论**：情况 ② ③ 完全无法实现（代码没查 PARTITIONS），且分区表连情况①口径也被壳值挡死。当前仅情况①对**非分区普通表**有效。

### 2.4 为什么一直没被发现

在标准 MySQL/MariaDB 上，`information_schema.TABLES` 对分区表返回的是**正确聚合值**。本地 MariaDB 复现（见 §6）：同一张分区表在 TABLES 里显示 10.06MB（正确）。**该缺陷仅在 TDSQL 分布式 proxy 上暴露**——若开发在普通 MySQL 上自测则表现正常，这正是它漏网的原因。

## 三、修复设计

### 3.1 核心：数据源切换为 `information_schema.PARTITIONS`（统一源）

关键事实（本地已验证，见 §6）：`information_schema.PARTITIONS` 是**覆盖全部表的统一源**——
- 分区表：每分区一行，`PARTITION_NAME` 非空，各行为真实分区大小；
- 非分区表：恰好一行，`PARTITION_NAME = NULL`，该行为整表真实大小。

因此按 `(TABLE_SCHEMA, TABLE_NAME)` 分组聚合，一条 SQL 同时覆盖三种情况：
- `SUM(DATA_LENGTH+INDEX_LENGTH)` = 合并总大小（覆盖①③；非分区表 SUM 即自身）
- `MAX(DATA_LENGTH+INDEX_LENGTH)` = 最大单分区（用于标注②）
- `SUM(PARTITION_NAME IS NOT NULL)` = 分区数（0 表示非分区表）

**判定采集与否只需 `HAVING SUM(...) >= 1GB`**：因各分区非负，"合并 < 1GB" 必然"每个分区 < 1GB"，故 `SUM >= 1GB` 对三种情况都是充要条件；`MAX` 仅用于分类标注触发原因。

### 3.2 扫描范围：全部业务库（不再只扫默认库）

现状 `WHERE TABLE_SCHEMA = %s`（`database or self.config.database`）只扫连接默认库；分布式连接常无默认库 → `TABLE_SCHEMA=''` → 一张都查不到。改为：**排除系统库，扫全实例业务库**；保留可选 `database` 精确过滤（必须以参数 `%s` 传入，禁止字符串拼接，防注入）。

### 3.3 分级与触发类型

保留 L1/L2/L3 分级；新增 `trigger_type`（单表超标/单分区超标/合并超标）、`max_partition_gb`、`partition_count`、`is_partitioned` 供展示与后续分区水位分析。

## 四、精确代码改动（照抄级）

> ⚠️ **【致命陷阱·务必先读】改对方法！** 本文件里有**两个**同名 `check_large_tables`：
> - `TDSQLConnectionPool.check_large_tables`（**L664**，`class TDSQLConnectionPool`）—— **这才是 API/分布式实例实际调用的方法**（`_get_pool` → `registry.get()` 返回的就是 `TDSQLConnectionPool`）。
> - `TDSQLConnector.check_large_tables`（L1022，`class TDSQLConnector` 单连接兼容类）—— **API 从不调用它**。
>
> **只改 L1022 而不改 L664 = 修复完全不生效**（已发生：commit `1ea986e` 就只改了 L1022，L664 原样查 TABLES，故 15005 上分区表照旧采集不到）。**两处都要改，且 L664 是关键。**

### 4.1 后端 SQL —— `backend/services/tdsql_connector.py:664`（连接池版，**分布式实例实际走这里，必改**）

将 `check_large_tables` 整体替换为：

```python
    def check_large_tables(self, database: str = None, threshold_gb: float = 1.0) -> list[dict]:
        """检查大表（双源取大 GREATEST(PARTITIONS聚合, TABLES值)，兼容 TDSQL 分区表壳值）

        情况①单表>阈值 ②分区表单分区>阈值 ③分区表合并后>阈值 全覆盖。
        information_schema.PARTITIONS 覆盖全部表：分区表逐分区一行；非分区表以
        PARTITION_NAME=NULL 单行出现（已在 15005 实测确认：t_branch/t_dict/t_product 均在此出现）。
        按 (库,表) 聚合得合并大小(SUM part_bytes)与最大单分区(MAX max_part_bytes)；
        再与 TABLES 的整表大小取 GREATEST——本次 bug 根因正是"某个源对分区表返回壳值(512KB)"，
        双源取大可使任一源返回壳值时仍以另一源的真实值为准，彻底免疫此类意外。
        默认扫全部业务库；database 非空时仅扫该库(参数化传入，防注入)。
        """
        threshold_bytes = int(threshold_gb * 1024 * 1024 * 1024)
        sys_dbs = ("__tencentdb__", "information_schema", "mysql", "performance_schema",
                   "query_rewrite", "sys", "sysdb", "test", "xa")
        placeholders = ",".join(["%s"] * len(sys_dbs))
        db_filter = " AND TABLE_SCHEMA = %s" if database else ""

        sql = f"""
            SELECT
                agg.schema_name, agg.table_name, agg.is_partitioned, agg.partition_count,
                agg.rows_count,
                ROUND(GREATEST(agg.part_bytes, COALESCE(t.tab_bytes,0))/1024/1024/1024, 2) AS size_gb,
                ROUND(agg.max_part_bytes/1024/1024/1024, 2) AS max_partition_gb,
                ROUND(agg.part_data/1024/1024, 2)  AS data_mb,
                ROUND(agg.part_index/1024/1024, 2) AS index_mb,
                CASE
                  WHEN GREATEST(agg.part_bytes, COALESCE(t.tab_bytes,0)) >= 50*1024*1024*1024
                       OR agg.rows_count >= 200000000 THEN 'L3 特大表'
                  WHEN GREATEST(agg.part_bytes, COALESCE(t.tab_bytes,0)) >= 10*1024*1024*1024
                       OR agg.rows_count >= 30000000  THEN 'L2 重点大表'
                  ELSE 'L1 一般大表'
                END AS level,
                CASE
                  WHEN agg.is_partitioned = 0 THEN '单表超标'
                  WHEN agg.max_part_bytes >= %s THEN '单分区超标'
                  ELSE '合并超标'
                END AS trigger_type
            FROM (
                SELECT
                    TABLE_SCHEMA AS schema_name, TABLE_NAME AS table_name,
                    CASE WHEN SUM(PARTITION_NAME IS NOT NULL) > 0 THEN 1 ELSE 0 END AS is_partitioned,
                    SUM(PARTITION_NAME IS NOT NULL)                       AS partition_count,
                    SUM(COALESCE(TABLE_ROWS,0))                           AS rows_count,
                    SUM(COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0)) AS part_bytes,
                    MAX(COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0)) AS max_part_bytes,
                    SUM(COALESCE(DATA_LENGTH,0))                          AS part_data,
                    SUM(COALESCE(INDEX_LENGTH,0))                         AS part_index
                FROM information_schema.PARTITIONS
                WHERE TABLE_SCHEMA NOT IN ({placeholders})
                  AND TABLE_NAME NOT LIKE '%%_tdsql_subp_auto_%%'{db_filter}
                GROUP BY TABLE_SCHEMA, TABLE_NAME
            ) agg
            LEFT JOIN (
                SELECT TABLE_SCHEMA, TABLE_NAME,
                       COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0) AS tab_bytes
                FROM information_schema.TABLES WHERE TABLE_TYPE='BASE TABLE'
            ) t ON t.TABLE_SCHEMA=agg.schema_name AND t.TABLE_NAME=agg.table_name
            WHERE GREATEST(agg.part_bytes, COALESCE(t.tab_bytes,0)) >= %s
            ORDER BY size_gb DESC
        """
        params = [threshold_bytes, *sys_dbs]
        if database:
            params.append(database)
        params.append(threshold_bytes)
        return self._execute(sql, tuple(params))
```

> **参数顺序**（SQL 中 `%s` 自上而下）：① trigger_type 的阈值 → ② 系统库列表(9个) → ③ 可选 database → ④ 外层 WHERE 阈值。`params` 已严格按此拼装，改动时勿打乱。
> **口径说明**：`size_gb/level/WHERE` 用 `GREATEST(分区合并, 整表)` 双源取大；`max_partition_gb/trigger_type` 用 PARTITIONS 的 MAX（分区表逐分区真实值，已实测可靠）；`data_mb/index_mb` 为展示用，取自 PARTITIONS 聚合。

### 4.2 后端 SQL —— `backend/services/tdsql_connector.py:1022`（单连接兼容版）

### 4.4 前端采集 —— `frontend/static/js/app.js` `collectBigtable()`

现状把分区信息写死（`is_partitioned:false, partition_count:0`），且 schema 取单库 `cd.database`。改为逐行取后端真实字段：
```javascript
    const collectBigtable=async()=>{
      if(!currentConnectionId.value){ElementPlus.ElMessage.warning('请先选择实例');return}
      bigtableCollecting.value=true;
      try{
        const cr=await apiFetch(`${API_BASE}/api/v1/tdsql/check/large-tables?connection_id=${currentConnectionId.value}`);
        if(!cr.ok){const d=await cr.json();ElementPlus.ElMessage.error(d.detail||'采集失败');return}
        const cd=await cr.json();
        const tables=(cd.tables||[]).map(t=>({
          schema:t.schema_name||'',
          table:t.table_name,
          size_gb:t.size_gb,
          rows:t.rows_count,
          is_partitioned:!!t.is_partitioned,
          partition_count:t.partition_count||0,
          shard_key:''
        }));
        if(!tables.length){ElementPlus.ElMessage.info('未发现大表');return}
        const sr=await apiFetch(`${API_BASE}/api/v1/bigtable/inventory/${currentConnectionId.value}`,{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify(tables)});
        if(sr.ok){ElementPlus.ElMessage.success(`已采集 ${tables.length} 张大表`);loadBigtable()}
        else{const d=await sr.json();ElementPlus.ElMessage.error(d.detail||'保存失败')}
      }catch(e){ElementPlus.ElMessage.error('采集失败: '+e.message)}finally{bigtableCollecting.value=false}
    };
```
> 说明：`scan_big_tables`（`bigtable_engine.py`）读取的正是 `schema/table/size_gb/rows/is_partitioned/partition_count/shard_key` 这些键，字段名对齐即可正确入库。`bigtable_inventory` 表已有 `is_partitioned/partition_count` 列，无需改表结构即可持久化分区信息。

> **状态：已确认（用户 DBeaver 实测）。** 结论：采用 §4.1 的 **GREATEST 双源取大方案**，无需 Variant B。

真机（119.45.220.89:15005/tdsql_check）实测结果：

| 验证项 | 实测 | 结论 |
|--------|------|------|
| 分区表 `big_audit_trail` 在 PARTITIONS | p0–p7 各 ~1.3GB，SUM=10.37GB | PARTITIONS 对分区表返回**真实值** ✅ |

### （历史备选）Variant B（UNION 双源，一般不需要）

```sql
-- 分区表：从 PARTITIONS 聚合
SELECT TABLE_SCHEMA AS schema_name, TABLE_NAME AS table_name,
       1 AS is_partitioned, SUM(PARTITION_NAME IS NOT NULL) AS partition_count,
       SUM(COALESCE(TABLE_ROWS,0)) AS rows_count,
       ROUND(SUM(COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0))/1024/1024/1024,2) AS size_gb,
       ROUND(MAX(COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0))/1024/1024/1024,2) AS max_partition_gb
FROM information_schema.PARTITIONS
WHERE PARTITION_NAME IS NOT NULL AND TABLE_SCHEMA NOT IN (...系统库...)
GROUP BY TABLE_SCHEMA, TABLE_NAME
HAVING SUM(COALESCE(DATA_LENGTH,0)+COALESCE(INDEX_LENGTH,0)) >= %s
UNION ALL
-- 非分区表：从 TABLES（排除掉在 PARTITIONS 里有真实分区的表）
SELECT t.TABLE_SCHEMA, t.TABLE_NAME, 0, 0, COALESCE(t.TABLE_ROWS,0),
       ROUND((COALESCE(t.DATA_LENGTH,0)+COALESCE(t.INDEX_LENGTH,0))/1024/1024/1024,2),
       ROUND((COALESCE(t.DATA_LENGTH,0)+COALESCE(t.INDEX_LENGTH,0))/1024/1024/1024,2)
FROM information_schema.TABLES t
WHERE t.TABLE_TYPE='BASE TABLE' AND t.TABLE_SCHEMA NOT IN (...系统库...)
  AND (COALESCE(t.DATA_LENGTH,0)+COALESCE(t.INDEX_LENGTH,0)) >= %s
  AND NOT EXISTS (SELECT 1 FROM information_schema.PARTITIONS p
                  WHERE p.TABLE_SCHEMA=t.TABLE_SCHEMA AND p.TABLE_NAME=t.TABLE_NAME
                    AND p.PARTITION_NAME IS NOT NULL)
ORDER BY size_gb DESC;
```

## 六、本地验证结果（已完成）

在本地 MariaDB 建 1 张 4 分区表 `big_audit_trail`、1 张非分区大表 `plain_big`、1 张小表 `small_tbl`：

**原始 PARTITIONS 行**（证明统一源）：
```
big_audit_trail p0/p1/p2/p3  各 2637824   ← 分区表逐分区
plain_big       NULL         4734976      ← 非分区表单行(PARTITION_NAME=NULL)真实大小
small_tbl       NULL         16384
```

**修复 SQL 输出**（阈值 4MB 演示逻辑，生产=1GB）：
```
表               is_part part_cnt total_mb max_part_mb trigger_type
big_audit_trail    1       4       10.06     2.52       合并超标      ← 情况③
plain_big          0       0        4.52      4.52       单表超标      ← 情况①
（small_tbl 16KB 被 HAVING 正确过滤）
```
**阈值降到 2.4MB** 时 `big_audit_trail` 的 `trigger_type` 正确变为 **单分区超标**（max 2.52≥2.4）→ 情况② 标注正确。

**对照**：本地 `TABLES` 对 `big_audit_trail` 返回 10.06MB（**正确聚合**）——印证 §2.4「缺陷仅在 TDSQL proxy 暴露」，本地无法复现壳值，故 §5 真机验证不可省。
