## 一、文档目的

本文档基于对北京农商银行SQL审核平台架构思路、TDSQL官方四份规范文档的深度学习，结合当前项目（V0.4.0）的实际能力，进行系统化对标分析，识别差距与不足，并据此编写下一版本（V1.0）的完整需求规格说明书。

**定位说明：** 本项目专门针对TDSQL数据库，不做异构数据库支持。北京农商银行平台覆盖多种异构数据库的思路不作为本项目的对标方向，对标重点聚焦于其SQL全生命周期管理、双引擎分析、DevOps集成、质量门禁等架构理念。

## 二、对标分析

### 2.1 对标北京农商银行SQL审核平台架构

北京农商银行SQL审核平台的五层架构为：统一管理数据源层、多模式SQL采集层、双引擎优化分析层、SQL全生命周期应用场景层、微服务基础功能层。

| 架构维度 | 北京农商银行能力 | 当前项目能力 | 差距评估 |
|---------|---------------|------------|---------|
| **数据源管理** | 异构数据库统一管理（商用、开源、信创多种类型） | 专注TDSQL单类型数据库 | **不做对标**：本项目明确定位于TDSQL专用审核工具，不支持异构数据库。北京农商银行因信创替代过渡期需管理多种数据库，本项目无此需求 |
| **SQL采集层** | 文件采集、IDE插件采集、JDBC连库采集、分布式慢日志/TOP SQL接口采集 | 文件采集（含MyBatis XML）、GitLab Webhook采集、TDSQL慢日志/PROCESSLIST采集 | **中等差距**：缺少IDE插件采集和JDBC连库采集 |
| **分析引擎** | SQL解析引擎 + 专家规则引擎（双引擎） | sqlglot解析引擎 + 22条规则引擎 | **中等差距**：双引擎架构已有雏形，但规则覆盖面不足，缺少专家知识库 |
| **全生命周期** | 开发→测试→准生产→生产 全链路覆盖 | 主要覆盖开发阶段（静态审核）+ 生产阶段（慢SQL抓取） | **重大差距**：缺少测试阶段执行计划审核、准生产阶段质量门禁、生产阶段实时监控告警 |
| **DevOps集成** | IDEA插件、MyBatis自动分析、CI/CD流水线集成 | GitLab MR Webhook、MyBatis XML解析 | **中等差距**：有基础GitLab集成，但缺少IDE插件和CI/CD流水线深度集成 |
| **质量门禁** | 24个系统发版强制审核 | 无质量门禁机制 | **重大差距**：缺少发版审核阻断机制 |
| **平台架构** | 微服务架构，横向扩展，负载均衡 | 单体FastAPI应用 | **中等差距**：当前规模下单体可接受，但需考虑未来扩展 |
| **管理规模** | 90+系统纳入管理 | 单实例部署 | **中等差距**：多实例TDSQL连接已有，但缺少多系统/多团队管理 |
| **数据分布监控** | 智能算法+全链路追踪，数据分布均衡管理 | 无 | **重大差距**：缺少分片数据分布均衡性监控 |
| **SQL指纹去重** | 慢SQL fingerprint分析 | 无指纹去重 | **中等差距**：慢SQL分析缺少指纹归并 |

#### 4.2.2 慢SQL分析增强（FR-SLOW）

**FR-SLOW-001：分布式EXPLAIN分析**

- 解析EXPLAIN结果中的分区信息（partitions列），检测分区裁剪是否生效
- 检测WHERE条件是否包含分片键，未包含时标记为"全分片广播"风险
- 检测跨SET JOIN场景，标记为性能风险
- 支持EXPLAIN FORMAT=TREE解析，展示Proxy下推情况
- 提供分布式执行计划可视化（分片路由图）

**FR-SLOW-002：死锁与锁等待分析**

- 执行`SHOW ENGINE INNODB STATUS\G`解析死锁信息
- 执行`SELECT * FROM sys.innodb_lock_waits`获取锁等待关系
- 执行`SELECT * FROM performance_schema.metadata_locks WHERE LOCK_STATUS='PENDING'`获取MDL锁等待
- 自动识别死锁涉及的两条SQL、锁定的资源和加锁顺序
- 提供死锁规避建议（统一加锁顺序、缩短事务、乐观锁）

**FR-SLOW-003：长事务排查**

- 执行`SELECT * FROM information_schema.innodb_trx WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 60`获取长事务
- 解析事务SQL文本、锁定行数、修改行数
- 按运行时间排序，标记超阈值事务（>60s告警，>300s紧急）
- 提供长事务终止建议（KILL连接ID）

**FR-SLOW-004：字符集与排序规则一致性诊断**

- **SQL-1：** 查看实例/数据库默认字符集与排序规则
- **SQL-2：** 统计指定库下字符集/排序规则种类（>1即告警）
- **SQL-3：** 定位与库默认字符集不一致的表
- **SQL-4：** 定位字段collation与表collation不一致的列
- **SQL-5：** 跨表关联字段字符集/排序规则一致性检查
- **SQL-6：** 生成统一字符集修复SQL
- 自动执行上述诊断SQL并生成一致性报告
- 对不一致项标记风险级别并给出修复SQL

**FR-SLOW-005：字段类型一致性诊断**

- 执行跨表同名字段类型检查SQL
- 识别类型不一致的关联字段对（如t1.user_id bigint vs t2.user_id varchar）
- 标记为JOIN索引失效隐患
- 提供字段类型统一建议

**FR-SLOW-006：SQL指纹分析**

- 对慢SQL进行fingerprint归并（去除具体值，保留SQL结构）
- 按指纹维度统计：执行次数、总耗时、平均耗时、最大耗时、扫描行数
- 支持四维TopN排序：总耗时占比、执行频次、单次最大耗时、扫描行数/返回行数比
- 慢SQL指纹趋势图（按天/小时）

**FR-SLOW-007：索引分析与推荐**

- 执行`SELECT * FROM sys.schema_unused_indexes`检测未使用索引
- 执行`SELECT * FROM sys.schema_redundant_indexes`检测冗余索引
- 基于EXPLAIN结果分析索引使用情况（possible_keys vs key vs key_len）
- 基于查询字段和WHERE条件推荐覆盖索引
- 基于列区分度推荐复合索引字段顺序
- 提供索引创建/删除建议SQL

**FR-SLOW-008：SQL改写建议**

- SELECT * → 指定字段列表
- 深分页 LIMIT offset,N → 延迟关联 / 游标分页
- 相关子查询 → JOIN改写
- OR条件 → UNION ALL改写
- WHERE列上函数 → 范围条件改写
- IN列表过大 → 临时表JOIN建议
- 提供改写前后的SQL对比和预期效果说明

**FR-SLOW-009：统计信息检查**

- 检查`information_schema_stats_expiry`参数值
- 检测热点表的统计信息最后更新时间
- 对大批量变更后的表推荐执行ANALYZE TABLE
- 检测表数据量变化是否超过10%阈值

**FR-SLOW-010：优化效果验证**

- 记录优化前的EXPLAIN结果（type/key/rows/Extra）

  ```
  tdsql-sqlcheck audit --file ./migration.sql --format json
  tdsql-sqlcheck audit --file ./migration.sql --gate strict
  ```
- 支持Jenkins/GitLab CI/GitHub Actions集成
- 审核结果输出格式：JSON / SARIF / HTML / PDF
- 退出码：0=通过，1=门禁未通过，2=系统错误

**FR-LIFECYCLE-003：审核历史与趋势**

- SQLite持久化所有审核记录（已有audit_history表）

  - 按项目/系统维度的违规趋势图
