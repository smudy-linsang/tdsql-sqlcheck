# TDSQL SQL审核平台 V2.0 银行级改造设计说明书

**版本：** V2.0
**日期：** 2026-07-03
**状态：** 已实施

## 一、改造背景与目标

### 1.1 背景

V1.0 是一个功能完整的"单实例工具"：核心引擎（77条规则、慢SQL六维分析）质量良好，
但连接模型、存储、安全三个地基按单人单库使用设计，无法支撑商业银行规模化生产上线。

### 1.2 目标场景

- 部署环境：**纯内网**（无外网出口，无法访问CDN/外部镜像源）
- 服务规模：**数百套**开发、测试、生产环境 TDSQL 数据库
- 合规要求：等保测评、行内审计——所有操作可追溯到人，敏感数据不落地
- 组织形态：多团队多角色（管理员/DBA/开发/审计），权限隔离

### 1.3 V1.0 差距评估结论（改造依据）

| 优先级 | 差距 | V1.0现状 | V2.0处置 |
|--------|------|----------|----------|
| P0 | 全局单连接模型 | `tdsql_manage._pool` 全局变量，多用户互相顶掉且旧池泄漏 | ✅ 连接注册表（§3.1） |
| P0 | 无认证/授权/操作主体 | 全部API裸奔，CORS `*`，审计日志无用户 | ✅ 认证+RBAC+审计（§3.2） |
| P0 | 前端CDN依赖 | unpkg.com加载Vue/ElementPlus/ECharts，内网白屏 | ✅ 资产本地化（§3.3） |
| P0 | 敏感数据落地 | 慢SQL原文入库（含身份证/卡号字面量）；连接密码明文JSON | ✅ 入库脱敏+加密存储（§3.4） |
| P0 | 密钥硬编码 | 加密密钥由源码内固定种子派生 | ✅ 密钥管理（§3.5） |
| P1 | 调度无法编排多实例 | 仅环境变量单实例定时拉取；多副本重复执行 | ✅ 扫描计划+leader租约（§3.6） |
| P1 | 目标库无压力保护 | processlist轮询占线程60s，扫描无并发限制 | ✅ 双重限流（§3.7） |
| P1 | 数据无生命周期 | slow_queries/audit_history只进不出 | ✅ 保留策略+清理（§3.8） |
| P1 | 规则配置全局 | 规则启停全局生效，无法按项目/环境差异化 | ✅ 规则集多租户（§3.9） |
| P1 | 可观测性为零 | 仅文本日志 | ✅ /metrics+请求ID（§3.10） |
| P2 | Webhook可裸奔 | 未配secret跳过校验 | ✅ 默认拒绝（§3.11） |
| P2 | 文档/版本漂移 | README停留22条规则；0.4.0/1.0.0混用；双SQLite路径 | ✅ 全部修正 |

## 三、核心设计

### 3.1 多实例连接注册表（services/connection_registry.py）

替代 V1.0 全局单连接：

- **注册表**：`connection_id → _PoolEntry(pool, last_used)`，`threading.RLock` 保护
- **生命周期**：注册时验证 `SELECT 1`；替换同ID时关闭旧池（杜绝泄漏）；
  空闲超时回收（默认30分钟）；超过上限 LRU 淘汰（默认256实例）
- **持久化**：连接配置存 `tdsql_connections` 表，密码 Fernet 加密；
  `adhoc` 特殊ID保留 V1.0 即席连接语义（向后兼容）
- **路由**：全部查询类 API 增加 `connection_id` 参数；缺省回退 adhoc → 默认连接自动建连

### 3.2 认证与RBAC（services/auth_service.py + middleware.py）

详见《安全与权限设计说明书-V2.0》。要点：

- 口令 PBKDF2-HMAC-SHA256（240k轮）；令牌 HMAC-SHA256 自包含签名（默认8小时）
- 四角色权限矩阵（admin/dba/developer/auditor），中间件按 `(role, method, path)` 判定
- 登录连续失败锁定；账户禁用即时生效（每请求校验用户状态，30秒缓存）
- 操作审计：变更请求自动写 operation_logs（操作人/IP/UA/路径/状态码）
- 引导：空库首次启动创建 admin（口令来自 `ADMIN_INITIAL_PASSWORD` 或随机打印一次）

### 3.3 前端内网化（frontend/static/vendor/）

- Vue 3.4 / Element Plus 2.7 / ECharts 5.5 / 图标 / 中文语言包全部落库到
  `frontend/static/vendor/`（约2.6MB），由 FastAPI StaticFiles 服务
- `API_BASE` 由硬编码 `http://localhost:8000` 改为同源相对路径（支持反向代理）
- 新增登录页、当前用户区（角色徽标/修改口令/退出）、用户管理页（仅admin可见）
- 统一 `apiFetch` 封装：自动附加 Bearer 令牌，401 自动回到登录页

### 3.4 敏感数据治理

- **慢SQL入库脱敏**（`DATA_MASKING_ENABLED`，默认开）：入库前用 FingerprintEngine
  将字面量归一化为 `?`（分析在原文上执行，不影响诊断质量）
- **连接密码**：从明文 JSON 文件迁移到 SQLite 加密列 `password_encrypted`
- **接口脱敏**：连接列表永不返回密码（`***`）

### 3.5 密钥管理（services/security_service.py）

密钥优先级：`TDSQL_ENCRYPTION_KEY`/`AUTH_SECRET_KEY` 环境变量（KMS注入）→
`data/*.key` 密钥文件（首启自动生成随机密钥，chmod 600）。
V1.0 硬编码种子密钥仅保留**解密兼容**（历史数据迁移），不再用于任何加密。
`.gitignore` 排除全部密钥文件。

### 3.6 调度改造（services/scheduler.py）

- **leader租约**：`scheduler_lease` 表单行租约（120秒TTL，持有者续期/过期抢占），
  多副本部署时仅 leader 执行调度任务
- **扫描计划**：`scan_schedules` 表按连接配置每日扫描（源/时间/条数/阈值），
  调度器每分钟检查到期计划，经连接注册表自动建连执行
- **数据清理**：每日 03:30 执行保留策略清理
- V1.0 环境变量单实例拉取保留（向后兼容）

### 3.7 扫描限流

`registry.scan_slot(conn_id)` 双重信号量：
单连接并发（默认2，保护目标库）+ 全局并发（默认8，保护本服务）。
超限返回 HTTP 429，不排队阻塞。

### 3.8 数据保留（services/retention_service.py）

`retention_policies` 表按表配置保留天数（≥7天防误配），默认策略：
slow_queries/scan_tasks/fingerprint_stats 180天，audit_history/operation_logs/gate_audit_logs 365天，
alerts 90天。清理后执行 WAL checkpoint 回收空间。支持 API 手动触发。

### 3.9 规则集多租户（services/ruleset_service.py）

- `rule_sets` + `rule_set_items`（rule_id / enabled / severity_override）
- 项目经 `projects.rule_set_id` 绑定规则集；审核链路：
  `project_id → 规则集覆盖 → RuleChecker.audit_sql(rule_overrides=...)`
- 内置 `default` 规则集受保护（不可删改条目）；被项目引用的规则集禁止删除

### 3.10 可观测性

- `/metrics` Prometheus 文本格式（自研零依赖）：请求量/耗时、审核量、违规分布、
  扫描任务、登录成败、RBAC拒绝、活跃连接数、版本信息；路径归组防标签基数爆炸
- `X-Request-ID` 透传/生成，访问日志含用户/耗时/状态码

### 3.11 Webhook安全

未配置 `GITLAB_WEBHOOK_SECRET` 时默认**拒绝** webhook（V1.0为放行），
仅 `GITLAB_WEBHOOK_ALLOW_INSECURE=true`（开发/测试）时放行。

## 四、数据库设计变更（27张表）

V1.0 的 21 张表保留，新增：

| 表 | 用途 |
|----|------|
| T22 users | 用户（口令哈希/盐/角色/状态/锁定） |
| T23 rule_sets | 规则集 |
| T24 rule_set_items | 规则集条目（启停/级别覆盖） |
| T25 scan_schedules | 按连接扫描计划 |
| T26 retention_policies | 数据保留策略 |
| T27 scheduler_lease | 调度器leader租约 |

增量迁移：`audit_history.created_by`（审计主体）、`slow_queries.connection_id` 已有列复用。
全部迁移幂等（`IF NOT EXISTS` / `_add_column_if_not_exists`），V1.0 数据库可直接原地升级。

## 五、存储演进路线（说明）

V2.0 仍采用嵌入式 SQLite（WAL），并已完成：路径统一、busy_timeout、保留清理。
**单副本限制**：SQLite 决定应用当前为单实例部署（调度租约已为多副本做好准备）。
迁移集中式 MySQL/PostgreSQL 的准备工作已落实：

1. 数据访问集中在 `services/database.py`（裸SQL，表结构避免SQLite特有类型）
2. 业务层不直接拼接 SQLite 方言（除 `datetime('now')` 类函数，迁移时统一替换）
3. 迁移步骤：导出27表DDL→转换默认值函数→pymysql连接工厂替换`_get_connection`→
   参数占位符 `?`→`%s` 适配层→数据迁移脚本

该迁移为独立后续工程（估算2-3人周），不阻塞当前上线（单副本+定期备份data目录满足初期SLA）。
