## 零、先说结论：为什么你看着还是"空空荡荡"

**根因只有一个，但它让几乎所有新页面的表格都渲染成空的：后端返回的数据被包了一层 `{code, message, data, timestamp}` 信封，而前端的加载函数从来没有拆开这层信封去取 `data`。**

我用后端真实的数据模型跑了一遍证明给你看：

```
后端 /api/v1/projects 实际返回:
  {"code":0,"message":"success","data":[{"project_id":"p1","project_name":"核心支付"},...],"timestamp":""}

前端代码写的是:  projects.value = d.items || d || []
  d.items 不存在 → 取到 d 本身（是个对象，不是数组）
  → el-table 的 :data 收到一个对象而不是数组 → 什么都不显示 → 空表
```

**编码智能体把 12 个加载函数全部写成了 `d.items || d || []` 这个"万能猜测"写法，但没有一个新接口是这个格式。** 换句话说：**页面的架子搭好了、接口也调了，但返回的数据在最后一步被丢掉了，所以每张表都是空的。**

> 这说明一个流程问题：**编码智能体自始至终没有真正打开这些页面、连着真实数据看一眼。** 只要它把"项目管理"页打开过一次、数据库里哪怕有 1 个项目，就会立刻看到空表，5 秒就能发现这个 bug。请务必要求它这轮**逐页联调、亲眼确认每页有数据**，而不是"代码写完 = 完成"。

## 二、【高优先级】字段名对不上导致列为空 / 颜色错

### 2.1 数据库体检-大表检查（模板第 639–644 行 + 函数 1062 行）
- 后端 `GET /api/v1/tdsql/check/large-tables` 返回：
  `{"database":..,"threshold_gb":..,"total":..,"tables":[{TABLE_NAME, size_gb, TABLE_ROWS, data_mb, index_mb, level}]}`
  其中 `level` 的值是 **`"L3 特大表"` / `"L2 重点大表"` / `"L1 一般大表"` / `"一般表"`**（带中文，不是纯 `"L3"`）。
- ❌ 现在模板：`:data="healthResult.items||healthResult"`，列 `prop="table_name"`、`prop="rows"`，`row.level==='L3'`
- ✅ 改成：
  - `:data="healthResult.tables||[]"`
  - 列 `prop="TABLE_NAME"`（大写）、`prop="TABLE_ROWS"`（大写）、`prop="size_gb"`（这个对）
  - 级别标签判断改为包含匹配：`row.level&&row.level.startsWith('L3')?'danger':row.level&&row.level.startsWith('L2')?'warning':'info'`

### 2.2 数据库体检-字符集（模板第 633–638 行）
- 后端 `GET /api/v1/tdsql/check/charset` 返回（**无 data 信封，平铺**）：
  ```

   column_mismatches:[{TABLE_NAME,COLUMN_NAME,col_collation,tbl_collation}],
   cross_table_mismatches:[{COLUMN_NAME,tbl_a,col_a,tbl_b,col_b}],

  ```
- ❌ 现在模板读 `healthResult.summary`（不存在）和 `healthResult.issues`（不存在），所以字符集检查点了以后**只有一条 alert、下面什么都没有**。
- ✅ 改成：
  - 顶部 alert 用 `is_consistent`：一致→success「字符集一致」，不一致→warning「发现字符集不一致」
  - 用 `el-table :data="healthResult.column_mismatches"` 展示列级不一致：列 `TABLE_NAME` / `COLUMN_NAME` / `col_collation` / `tbl_collation`
  - 再加一张 `el-table :data="healthResult.cross_table_mismatches"` 展示跨表同名字段不一致：`COLUMN_NAME` / `tbl_a` / `col_a` / `tbl_b` / `col_b`

### 2.3 大表治理页字段（模板第 654–661 行）

## 三、【高优先级】页面太浅：只做了"只读列表"，设计要求的增删改没做

第 1 轮把空壳填成了表格，但**多数只做了"读"，设计说明书要求的"增/改/删"操作被跳过了**，导致：① 功能不完整；② 很多表本来就没数据（因为没有任何入口去创建数据），点开就是空的，加重了"空空荡荡"的观感。逐页补：

| 页面 | 现状 | 必须补（后端接口已就绪） |
|---|---|---|
| **大表治理** | 只 `GET inventory`，**首次进来清单是空的** | 加「采集/刷新清单」按钮 → `POST /api/v1/bigtable/inventory/{conn}`（先采集才有数据，否则永远空表）。设计 §5.12 |
| **巡检管理** | 只读列表 | 加「+ 新建巡检」（选实例+类型 → `POST /api/v1/inspection/tasks`）；行「查看结果」抽屉 → `GET /inspection/tasks/{id}` 展示巡检结果。设计 §5.13 |
| **规则集** | 只读列表 | 加「+ 新建规则集 / 编辑 / 删除」（规则项可勾选）→ `POST/PUT/DELETE /api/v1/rulesets`。设计 §5.11 |
| **监控告警** | 规则 tab 只读 | 加「+ 新建/编辑告警规则」→ `POST /api/v1/monitor/rules`。设计 §5.12 |
| **质量门禁** | 只能套预设策略 | 加自定义「最大 ERROR / WARNING 数」编辑并保存 → `POST /api/v1/gate/rules`。设计 §5.10 |
| **数据保留** | 只读 + 立即清理 | 加「新增/编辑保留策略」→ `PUT /api/v1/admin/retention`。设计 §5.9 |
| **项目管理** | 创建仅名称/实例/描述 | 列表显示的是 `rule_set_id`/`gate_rule_id` 原始 ID，无绑定入口。补：创建/编辑时可选规则集与门禁；列表把 ID 显示为名称。设计 §5.11 |

## 六、【必做】傻瓜式复验步骤（这轮必须亲自走一遍）

整改后**不要**只看代码，必须真正跑起来逐页确认。步骤：

1. **起后端**：配置元数据库（MySQL）后 `uvicorn backend.main:app --host 0.0.0.0 --port 8000`。
2. **验证响应格式**（改代码前先看一眼真实返回，杜绝再猜错）：
   ```bash
   # 登录拿 token
   TOKEN=$(curl -s -XPOST localhost:8000/api/v1/auth/login -H 'Content-Type: application/json' \
     -d '{"username":"admin","password":"<你的admin口令>"}' | python3 -c "import sys,json;print(json.load(sys.stdin)['token'])")
   # 逐个接口看真实结构（重点看最外层是 data 还是 rulesets 还是 policies）
   curl -s localhost:8000/api/v1/projects   -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
   curl -s localhost:8000/api/v1/rulesets   -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
   curl -s localhost:8000/api/v1/monitor/alerts -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
   curl -s localhost:8000/api/v1/admin/retention -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
   ```
