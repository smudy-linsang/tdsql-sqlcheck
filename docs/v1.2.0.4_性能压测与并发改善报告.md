## 🎯 二、核心发现与结论

1. **彻底解除单线程 Single Event Loop 串行锁死天花板**：
   - **优化前**：由于将带有同步数据库 IO 的路由函数误写为 `async def`，FastAPI 强行将其在 Uvicorn 单事件循环主线程上排队执行。由于每次查询需消耗 10ms 往返时间，**系统全局吞吐量被死死卡在 ~95 req/s 天花板**，无法利用多核与并发能力。
   - **优化后**：将 122 个数据库交互路由改为 `def` 后，FastAPI 自动将阻塞查询调度至底层 Worker 线程池并发处理。并发数到达 50 时，**吞吐量飙升至 1431.9 req/s（提升 15.1 倍）**。

2. **多用户高并发下延迟大幅平稳**：
   - 线程池完美重叠了远程 DB 的网络往返等待时间（RTT Overlapping）。在真实端点与真实元数据库交互场景下，吞吐量由 9.6 req/s **实测提升 4 倍至 38.4 req/s**（并发 50 下延迟从 5098ms 剧降至 1229ms），高并发场景下响应体验获得根本性改善！

3. **连接池容量与按需探活优化**：
   - 元数据库默认连接池大小由 10 提升至 **60**（由 `SQLCHECK_DB_POOL_SIZE` 控制），完美匹配 Uvicorn 并发并发上限；
   - 探活机制优化为**按需探活**（连接空闲超过 10s 才触发 `ping()`），热连接复用无额外 RTT 消耗。
