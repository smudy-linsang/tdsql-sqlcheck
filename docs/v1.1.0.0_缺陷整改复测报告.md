## 二、逐项复测详情

### BUG-01 【High】深度诊断子权限接口级鉴权失效（越权）—— ✅ 已修复（两轮）

- **整改过程**：
  - 首轮 `64210c8`：将二级校验的 `if role not in _BUILTIN_ROLES` 改为 `if role != "admin"`，令内置角色也走 `role_permissions` 二级校验。**8/9 子模块生效，但 `deep-diag-gateway` 仍越权**——复测发现 `_PATH_TO_MENU` 中 `/api/v1/gate`（质量门禁）是 `/api/v1/gateway-log` 的前缀且排序在前，`for ... if path.startswith(prefix): return` 首个命中即返回，网关鉴权被"质量门禁"权限顶替。
  - 二次 `3fb8644`：二级校验改为**按前缀长度降序排序 + 边界匹配**（`path == prefix or path.startswith(prefix + "/")`），双重防线消除前缀碰撞。
- **复测证据（重启后端加载新代码）**：
  - 单元级 `check_permission` 路径匹配：9 个子模块全部映射到正确 menu_key；`gate` 自身 `/api/v1/gate/strategies` 仍正确映射到 `gate`。
  - 接口级（developer 去掉 `deep-diag-gateway`）：`GET /api/v1/gateway-log/reports` → **403**（首轮为 200）。
  - 接口级（developer 去掉 `deep-diag-toolkit`）：`GET /api/v1/toolkit/scripts` → **403**。
  - **无误伤**：未撤销的 `GET /api/v1/gate/strategies`（质量门禁）→ **200**；未撤销的 `GET /api/v1/ppt-report/dashboard` → 200。
  - dba 去掉网关：`GET /gateway-log/reports` → 403，同时 `/gate/strategies` → 200。
- **关键点**：修复既堵住越权，又未过度拦截质量门禁（前缀碰撞修复最易翻车处），两边实测均正确。

### BUG-02 【High】首次登录强制改密可被绕过 —— ✅ 已修复
- **整改**：
  - 后端 `middleware.py`：对 `must_change_password=1` 用户，除 `/auth/change-password`、`/auth/logout`、`/auth/me` 外，其余接口一律 403。
  - 前端 `index.html`/`app.js`：强制场景 `pwdDialog.forced=true`，弹窗 `:close-on-press-escape="false"`、`:close-on-click-modal="false"`、`:show-close="false"`，并移除"取消"按钮；`checkSession` 恢复会话时也会重新拉起强制弹窗。
- **复测证据**：
  - 后端：未改密用户 token `POST /api/v1/audit/sql` → **403 "首次登录必须修改口令后才能访问业务接口"**；`GET /auth/me`、`POST /auth/change-password` → 200；改密后业务接口恢复 200。
  - 前端（浏览器）：强制弹窗**无关闭X、无取消按钮**，按 ESC、点击遮罩均**无法关闭**。

### BUG-03 【High】即时SQL审核多语句静默漏报 —— ✅ 已修复
- **整改**：`audit_service.audit_single_sql` 用 `split_sql_statements` 拆分，逐条 `audit_sql`，违规消息加 `[第N条语句]` 前缀，`sql_type=BATCH`。
- **复测证据**：`SELECT *; UPDATE ...(无WHERE)` → 返回 5 条违规（R012/R051/R013/R014/R070）、`passed:false`，每条带 `[第N条语句]` 前缀；3 语句（含违规 DDL）→ 检出 14 条违规（含 R001 命名等）。首轮的"0 违规判通过"假阴性已消除。

### BUG-04 【Medium】导出HTML大屏依赖公网CDN —— ✅ 已修复
- **整改**：`daily_inspect_service.py` 将 `https://cdn.jsdelivr.net/npm/chart.js` 替换为本地 `/static/vendor/echarts.min.js`，图表逻辑由 Chart.js 改写为 ECharts。
- **复测证据**：导出报告 HTML **无任何外网 URL**（`grep https?://` 无命中）；引用本地 `/static/vendor/echarts.min.js`（该资源 `GET` → 200 可服务）；浏览器打开报告离线渲染出 ECharts 折线图（canvas 存在）。

### BUG-05 【Medium】编辑已连接实例配置后旧池不刷新 —— ✅ 已修复
- **整改**：`connection_registry.save_connection` 在更新已存在连接后调用 `self.disconnect(conn_id)` 淘汰活跃旧池。
- **复测证据**：连接实例后把 monitor_port 改为 9999，**无需手动断开**，再连接 + monitor-probe 即反映新配置（`ok=False, port=9999`），证明旧池已被淘汰、按新配置重建。

### BUG-06 【Medium】即时审核页缺少"多库模式开关" —— ✅ 已修复（用例修订）
- **整改**：按建议修订 `docs/v1.1.0.0_test_cases.md` 场景三步骤2，将"多库模式"指向真正承载多库审核能力的"上线检查"功能（选实例、库过滤留空、执行检查并导出报告），并同步校正了导航路径（"SQL审核 > 即时审核"、"SQL审核 > 审核规则库"）。
- **复测证据**：`POST /api/v1/inspection/schema-check` 对目标实例（不指定库）返回 12 项检查，覆盖非系统库，与修订后用例一致。

### BUG-07 【Low】连接按钮无 loading 态 —— ✅ 已修复
- **整改**：连接按钮加 `:loading="row.connecting"`，`connectInstance` 中 `row.connecting=true/false`。
- **复测证据**：`index.html` 已含 `:loading="row.connecting"` 绑定，`app.js` 中置位逻辑齐全；浏览器实测连接流程正常（未连接→已连接、弹"已连接"提示，无回归）。（对本地实例连接极快，spinner 呈现时间短，属正常。）

### BUG-08 【Low】断开连接不重置 DB status —— ✅ 已修复
- **整改**：`connection_registry.disconnect` 在清内存池的同时 `UPDATE tdsql_connections SET status='disconnected'`。
- **复测证据**：连接后 DB `status=connected`；断开后直查 DB `status=disconnected`，字段与实际一致。
