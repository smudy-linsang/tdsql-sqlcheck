## 一、V1.0.3 更新概要

### 1.1 新增功能

| 功能 | 说明 | 影响范围 |
|------|------|---------|
| **数据库上线检查模块** | 替代 tdsql_12.sh 脚本，12项Schema检查集成到Web界面 | 新增菜单"实例与体检→上线检查" |
| **HTML报告导出** | 上线检查结果支持导出独立HTML报告 | 新增导出按钮 |
| **实例管理编辑功能** | 已添加实例支持修改全部连接信息 | 实例管理操作列新增"编辑"按钮 |
| **项目删除/停用分离** | 删除为物理删除（不可恢复），新增停用/启用切换 | 项目管理操作列新增"停用/启用"按钮 |

### 1.2 交互优化

| 优化项 | 说明 |
|--------|------|
| 全局选择器移除 | 顶栏不再显示全局实例/项目选择框，各功能页面内置独立选择器 |
| 顶栏用户菜单右对齐 | 用户头像/角色菜单推至顶栏最右侧 |
| 扫描任务实例选择 | 新建扫描任务时在抽屉内选择目标实例 |
| 耗时阈值单位修正 | 标签从"ms"改为"秒"（后端实际使用秒为单位） |

### 1.3 安全修复

| 修复项 | 说明 |
|--------|------|
| SQL注入防护 | 上线检查的 database_filter 参数增加白名单校验 |
| XSS防护 | HTML报告导出的动态内容全部经过 html.escape 转义 |
| 权限细化 | developer角色可执行上线检查，auditor角色隐藏该菜单 |

### 1.4 版本号变更

全部版本标识统一升级为 **1.0.3**：
- `VERSION` 文件
- `backend/config.py` APP_VERSION
- `frontend/index.html` 标题和登录页
- `deploy/` 全部脚本和配置

## 二、更新部署步骤（从 v1.0.2 升级）

### 2.1 传输发布包到目标机

```bash
# 在外网打包机上（已完成）
# 发布包: dist/tdsql-sqlcheck-v1.0.3-linux-x86_64.tar.gz (约24MB)
# 校验和: dist/tdsql-sqlcheck-v1.0.3-linux-x86_64.tar.gz.sha256

# 将 dist/ 目录通过U盘/SCP/文件共享拷贝至目标机 /tmp/
scp -r dist/ user@target-server:/tmp/
```

### 2.2 校验与解压

```bash
# 在目标机上执行
cd /tmp/dist

# 1. SHA256校验
sha256sum -c tdsql-sqlcheck-v1.0.3-linux-x86_64.tar.gz.sha256
# 期望输出: tdsql-sqlcheck-v1.0.3-linux-x86_64.tar.gz: OK

# 2. 解压
tar -xzf tdsql-sqlcheck-v1.0.3-linux-x86_64.tar.gz
cd tdsql-sqlcheck-v1.0.3-linux-x86_64
```

### 2.3 停服 → 备份 → 更新 → 启服

```bash
# 1. 停止当前服务
sudo systemctl stop tdsql-sqlcheck

# 2. 备份当前版本（整个安装目录）
sudo cp -a /opt/tdsql-sqlcheck /opt/tdsql-sqlcheck-v1.0.2-backup

# 3. 更新代码（覆盖式更新，保留 .env 和 data/ 目录）
sudo cp -r backend/ /opt/tdsql-sqlcheck/
sudo cp -r frontend/ /opt/tdsql-sqlcheck/
sudo cp requirements.txt /opt/tdsql-sqlcheck/
sudo cp VERSION /opt/tdsql-sqlcheck/

# 4. 更新部署脚本（可选，供后续运维使用）
sudo cp -r deploy/ /opt/tdsql-sqlcheck/

# 5. 离线安装新增/更新的Python依赖（如果有新增依赖）
cd /opt/tdsql-sqlcheck
sudo /opt/tdsql-sqlcheck/venv/bin/pip install --no-index --find-links=wheels/ -r requirements.txt

# 6. 启动服务
sudo systemctl start tdsql-sqlcheck

# 7. 等待5秒后验证
sleep 5
```

### 2.4 部署验证

```bash
# 执行部署验证脚本
cd /opt/tdsql-sqlcheck
sudo bash deploy/verify_deploy.sh

# 或手动验证
# 1. 健康检查（版本号应为1.0.3）
curl -s http://127.0.0.1:8000/health | python3 -m json.tool

# 2. 前端可访问
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8000/
# 期望: 200

# 3. 静态资源正常
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8000/static/js/app.js
# 期望: 200

# 4. 登录测试（替换实际密码）
curl -s -X POST http://127.0.0.1:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"实际密码"}' | python3 -m json.tool
```

### 2.5 回滚方案

如果更新后出现问题，可快速回滚：

```bash
# 停服 → 恢复备份 → 启服
sudo systemctl stop tdsql-sqlcheck
sudo rm -rf /opt/tdsql-sqlcheck
sudo cp -a /opt/tdsql-sqlcheck-v1.0.2-backup /opt/tdsql-sqlcheck
sudo systemctl start tdsql-sqlcheck
```

## 三、新增功能验证清单

### 3.1 上线检查模块

1. 使用 admin 或 dba 账号登录
2. 侧边栏"实例与体检" → 应看到"上线检查"子菜单
3. 选择实例 → 点击"执行12项检查" → 等待完成
4. 查看KPI统计卡片（问题总数/ERROR/WARNING/INFO/通过项）
5. 点击"导出报告" → 浏览器下载 HTML 报告文件
6. 打开 HTML 报告确认内容完整

### 3.2 实例编辑功能

1. 进入"实例管理"
2. 某实例行操作列应显示"编辑"按钮（蓝色链接）
3. 点击"编辑" → 抽屉打开，表单预填充该实例信息
4. 修改连接名称 → 点击"更新" → 提示"连接已更新"
5. 列表刷新显示新名称

### 3.3 项目管理

1. 进入"平台治理→项目管理"
2. 项目行操作列应显示"停用/启用"和"删除"两个按钮
3. 点击"停用" → 状态变为灰色"停用"，按钮变为"启用"
4. 点击"删除" → 弹出确认框"删除后不可恢复" → 确认后记录消失

### 3.4 权限验证

| 角色 | 上线检查菜单 | 执行检查 |
|------|-------------|---------|
| admin | ✅ 可见 | ✅ 可执行(200) |
| dba | ✅ 可见 | ✅ 可执行(200) |
| developer | ✅ 可见 | ✅ 可执行(200) |
| auditor | ❌ 隐藏 | ❌ 不可执行(403) |

## 四、已知限制与注意事项

1. **数据库无需变更**：v1.0.3 未引入破坏性表结构变更，`ensure_db` 增量兼容设计，元数据库无需降级或迁移
2. **配置文件兼容**：v1.0.2 的 `.env` 文件完全兼容 v1.0.3，无需修改
3. **wheels 目录**：发布包内含全部 Linux x86_64 wheels，目标机无需外网
4. **auditor 角色**：上线检查功能对 auditor 隐藏（只读角色不可执行POST操作）
5. **旧版本文档**：发布包中 docs/ 目录下的文档文件名仍为 v1.0.2 后缀，内容适用
