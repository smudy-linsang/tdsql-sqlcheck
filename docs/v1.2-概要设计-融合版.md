## 1. 对 M 稿的评审结论

### 1.1 一句话总评

> **M 稿工程质量高、"照图施工"到位、兼容性意识强（0 破坏），但它 6 项里有 5 项是"内部重构 / 工程卫生"，只有 1 项（RBAC 矩阵测试）直接降低业务风险；而恰恰缺了"让审核更强、更贴近真实业务"的能力——也就是用户这次真正想要的东西。**

M 稿把 v1.2 定义成一次"架构整理"，这在系统刚经历 8 个缺陷整改、进入稳定期时，方向可以理解；但对"团队计划开展系统改进和升级"这个诉求来说，**纯重构的投入产出比偏低，且把最高风险的两项（前端重写、SSH 工具桥接）放进了同一个版本**。

### 1.2 逐项评审

| M 项 | 主题 | 我的结论 | 价值 | 风险 | 建议 |
|---|---|---|---|---|---|
| **C1** | 数据库 schema 文件化（1500 行 DDL → 版本化迁移框架） | **缓做/降级** | 低（内部可维护性） | **中高**（在已稳定跑通的 `ensure_db()` 上重造迁移框架，启动期迁移一旦出错影响全实例） | v1.2 不做完整迁移框架；只做"DDL 抽成只读 `.sql` 资源 + 校验"，不改变现有建表执行路径 |
| **C2** | parser 拆分（pre_parser 从 parser 抽出） | **可选/顺带** | 低（纯代码整洁） | 低 | 不单列为版本目标；谁改 parser 时顺手拆，别为拆而拆 |
| **C3** | connector 拆分（71KB → 5 模块） | **部分采纳（作为 F1 的使能项）** | 中（可维护性 + 为 F1 提供 MetadataFetcher） | 中 | **只抽 `MetadataFetcher`**（表/索引/视图/触发器…采集），其余 4 类不强拆；借 F1 落地 |
| **C4** | 前端 Vite 工程化（单页 → Vue3 SFC 重写 3K 行） | **移出 v1.2** | 中（长期可维护性） | **高**（重写 + 引入 Node/Vite 构建链，纯内网部署新增构建依赖；灰度双前端维护成本翻倍） | 立项单独的"前端工程化专项"，不塞进业务版本；v1.2 只在现有单页上加 F1 的页面 |
| **C5** | RBAC 矩阵单测（800 行，4 角色 × 全接口 × 9 子权限） | **强烈采纳，列为 v1.2 必做** | **高**（直接守护刚出过越权缺陷 BUG-01 的 RBAC） | 低（仅新增测试） | 保留 M 稿设计，纳入 v1.2；补 BUG-01 前缀碰撞回归 |
| **C6** | tdsql-toolkit 桥接（Web → SSH 远程跑 .sh） | **移出 v1.2 / 重新设计** | 中（运维便利） | **高**（银行级系统里让 Web 应用 SSH 远程执行 shell = 高危攻击面，凭据管理 + RCE 风险） | v1.2 不做 SSH 远程执行；若要做，改为"平台内置只读采集 + 结果回传"的拉模式，另立项做安全评审 |

### 1.3 关键判断：M 稿缺了"业务价值轴"

M 稿的 6 项改完之后，**产品对用户的能力一点没变**——还是审同样的 SQL、出同样的报告。而用户这次明确提出的痛点是业务侧的：

> "SIT/UAT 上线前初始化数据库，是开发和测试在做，他们反复改表结构；我们问他们要建表语句文件去扫描，**拿到的文件总不是最新或最终的**……我想让系统能**直接把当前数据库的元数据全部拉出来做成文件，然后用规则审核**，把库里真实的情况和问题审出来。"

这正是 v1.2 最该做的事，也是 M 稿完全没有覆盖的方向。**所以我建议把 v1.2 的重心从"整理代码"调整为"补齐这项真实业务能力"，重构项只保留能顺带支撑它的最小集合（C3 的 MetadataFetcher）+ 低风险高收益的 C5。**

## 2. 我建议的 v1.2 路线图

### 2.1 一张图看懂调整

```
M 稿 v1.2 (6 项全是工程整理)              我建议的 v1.2 (业务价值优先)
┌─────────────────────────┐             ┌──────────────────────────────┐
│ C1 schema文件化   [中高险]│  ──缓做──▶  │ F1 实况库反向审核  ★旗舰新特性 │  ← 用户诉求
│ C2 parser拆分     [低值] │  ──顺带──▶  │    (拉真实元数据→DDL→规则审核) │
│ C3 connector拆分  [中]   │  ──收窄──▶  │ C3' MetadataFetcher(F1 使能)  │
│ C4 前端Vite重写   [高险] │  ──移出──▶  │ C5  RBAC矩阵测试   (守护越权)  │  ← M稿唯一高价值项，保留
│ C5 RBAC矩阵测试   [高值] │  ──保留──▶  │ F2  Schema漂移对比(F1 顺延)   │  ← F1 自然延伸
│ C6 toolkit SSH桥接[高险] │  ──移出──▶  │ (C1/C2/C4/C6 移入后续专项)    │
└─────────────────────────┘             └──────────────────────────────┘
```

### 2.2 v1.2 范围（建议）

| 编号 | 名称 | 类型 | 价值 | 说明 |
|---|---|---|---|---|
| **F1** | 实况库反向审核（Live Schema Reverse-Audit） | 新特性（旗舰） | ★★★ | 从活库反向抽取全量元数据 → 规范化 DDL 快照文件 → 跑 119 条规则 + 12 项 Schema 检查，审出真实库的问题 |
| **F2** | Schema 漂移对比（Schema Drift Diff） | 新特性（F1 延伸） | ★★ | 两份快照对比（同库不同时间 / SIT vs UAT），发现结构漂移；是 F1 落地后的低成本增值 |
| **C3'** | MetadataFetcher 抽取（connector 收窄拆分） | 重构（使能项） | ★ | 只把元数据采集从 71KB 巨类抽成独立模块，供 F1 复用，公共 API 不变 |
| **C5** | RBAC 权限矩阵单测 | 测试加固 | ★★ | 采纳 M 稿设计，守护刚整改的越权面，纳入 CI |

> C1/C2/C4/C6 不废弃，移入 v1.3+ 的"工程化专项"与"运维工具专项"，各自单独做安全/可行性评审。

### 2.3 为什么这样排

- **F1 是"用真实换手工"的范式升级**：从"问人要文件（永远滞后）"变成"以活库为唯一事实源"。这不是锦上添花，是把审核对象从"可能过时的输入"换成"当下真实状态"，直接命中用户痛点。
- **F1 高度复用现有资产**：119 条规则引擎、`checker.audit_file()` 多语句审核、`SchemaInspector` 12 项检查、`SHOW CREATE TABLE` 采集——**F1 的 70% 是编排已有能力，真正新写的只有"全量对象反向抽取 + 规范化"**，投入产出比极高、风险低。
- **C5 是刚出过越权缺陷后的必要保险**：BUG-01（网关子权限前缀碰撞越权）暴露了 RBAC 缺少矩阵化验证，C5 用 ~800 行把它固化为回归网，低成本高收益。
- **把高风险项挪出去**：前端重写、SSH 桥接都不该和一个要交付真实业务价值的版本绑在一起，各自风险独立、评审独立。

## 3. F1 · 实况库反向审核 —— 概要设计

### 3.1 要解决的问题（业务视角）

```
现状（痛点）：                              F1（目标）：
开发/测试在 SIT/UAT 库反复改表结构           系统直接连到 SIT/UAT 活库
        │                                          │
   我们问他们要建表语句 .sql                  一键反向抽取全量元数据
        │  （文件常常不是最新/最终版）                │  表/列/索引/约束/视图/触发器/存储过程/事件/分片键
   人工把文件丢进"文件审核"                    自动规范化成一份确定性 DDL 快照文件
        │                                          │
   审出来的是"文件里的问题"                     用 119 条规则 + 12 项 Schema 检查审
   ≠ 库里真实的问题                            审出的就是"库里真实的问题"，并留存可追溯快照
```

一句话：**让审核对象从"人给的、可能过时的 DDL 文件"，变成"从活库直接反向工程出来的、当下真实的 DDL 快照"。**

### 3.2 能力总览

F1 = 三段流水线 + 两个衍生能力：

```
 ①采集(Extract)            ②规范化(Normalize)          ③审核(Audit)
┌───────────────┐        ┌────────────────────┐      ┌─────────────────────┐
│ 连活库,枚举对象 │        │ 拼装确定性 DDL 快照   │      │ 快照 → checker.      │
│ 逐对象SHOW CREATE│ ──▶   │ 排序/去易变量/脱敏    │ ──▶  │ audit_file(119规则)  │
│ 表/视图/触发器/  │        │ 落盘 .sql + 入库快照表 │      │ + SchemaInspector12项│
│ 存储过程/事件/分片│        └────────────────────┘      │ → 违规报告(按对象归并)│
└───────────────┘                  │                   └─────────────────────┘
                                    ▼
                        ┌──────────────────────┐   ┌──────────────────────┐
                        │ 衍生A: 快照下载(.sql)  │   │ 衍生B: 漂移对比(F2)    │
                        │ 交付/归档/离线复用     │   │ 快照↔快照 结构差异     │
                        └──────────────────────┘   └──────────────────────┘
```

### 3.3 覆盖的元数据对象

| 对象 | 采集源 | 反向 DDL 方式 | 备注 |
|---|---|---|---|
| 表（列/主键/约束/引擎/字符集） | `information_schema.TABLES/COLUMNS` | `SHOW CREATE TABLE` | 索引、外键、PK 均内嵌在建表 DDL 中，一并覆盖 |
| 索引 | 同上（内嵌） | 建表 DDL 内 | 复用现有 `get_indexes` 交叉校验 |
| 视图 | `information_schema.VIEWS` | `SHOW CREATE VIEW` | 记录依赖关系 |
| 触发器 | `information_schema.TRIGGERS` | `SHOW CREATE TRIGGER` | TDSQL 分布式实例可能受限 → 优雅降级并标注 |
| 存储过程/函数 | `information_schema.ROUTINES` | `SHOW CREATE PROCEDURE/FUNCTION` | 同上 |
| 事件 | `information_schema.EVENTS` | `SHOW CREATE EVENT` | 同上 |
| TDSQL 分片键 | `information_schema.TDSQL_SHARDING_RULES` + 建表 DDL 解析 | 注入到表 DDL 注释/元信息 | 复用现有分片键解析逻辑 |

> **对象类型可选**：默认采集"表+视图"，触发器/过程/函数/事件按需勾选。系统库（`information_schema/mysql/performance_schema/sys/__tencentdb__/...`）默认排除，口径与现有 `_BIGTABLE_SYS_DBS`、`SchemaInspector` 一致。

### 3.4 三个关键设计点（决定这套东西好不好用）

**① 快照必须是"确定性"的（diff-stable）**
反向工程出来的 DDL 如果每次顺序、`AUTO_INCREMENT` 计数、易变注释都在变，快照就没法对比、没法归档。所以规范化阶段要：
- 对象**稳定排序**（库 → 类型 → 名称 字典序）；
- **剥离易变量**：`AUTO_INCREMENT=NNN`、统计性注释、时间戳；
- 统一大小写/反引号/换行；
- 输出带 `sha256` 指纹 —— 指纹相同即结构相同，秒判"库有没有变过"。

**② 规则需要"实况档位"校准（很重要，容易被忽略）**
119 条规则最初是审"开发手写 DDL"的。`SHOW CREATE` 反出来的是 MySQL 规范化 DDL（一定有反引号、一定显式 ENGINE/charset），于是一批"作者期"规则（如"必须显式指定引擎"）会**恒过**，噪声大。因此 F1 提供一个内置规则集档位 **`live_schema`（实况库档）**，聚焦对"既存结构"真正有意义的规则：命名规范、缺主键、冗余/缺失索引、超大字段、字符集不一致、缺注释、分片键相关、危险类型等；作者期专属规则自动降噪。此档位复用现有"规则集多租户"机制实现，不改规则本体。

**③ 面向数百实例要能异步 + 限流**
大库对象多、`SHOW CREATE` 是逐对象调用，必须走后台任务（复用现有 `scan_tasks` 模式）：分批采集、进度上报、超时兜底、单实例并发信号量，避免独占连接（正好呼应 v1.1 的"3 天采集限制"那类防护思路）。

### 3.5 与现有能力的关系（复用 vs 新增）

| 能力 | 现状 | F1 如何用 |
|---|---|---|
| `SHOW CREATE TABLE` + 分片键解析 | 已有（`get_table_metadata`/`_extract_shard_key`） | **复用**，扩展到视图/触发器/过程/事件 |
| `checker.audit_file(content)` 按 `;` 拆分跑 119 规则 | 已有（文件审核用） | **直接复用**为③审核阶段的引擎 |
| `SchemaInspector` 12 项 information_schema 检查（C01~C12） | 已有（"上线检查"用） | **并入** F1 报告，作为"元数据级"补充检查 |
| 规则集多租户（启停/级别覆盖） | 已有 | **复用**实现 `live_schema` 档位 |
| 审核历史 `audit_history` | 已有 | 新增来源 `source='live_schema'` |
| 后台任务/并发信号量 | 已有（scan_tasks） | **复用**采集任务编排 |
| 连接注册表（多实例池） | 已有 | **复用**选实例 |

> **净新增只有**：全量对象反向抽取器 `MetadataExtractor`、快照规范化器、快照存储表 `schema_snapshots`、一组 `/live-schema/*` API、一个前端页面。约 70% 是编排、30% 是新代码——这是把它排为旗舰的底气。

### 3.6 接口概览（详细签名见详细设计）

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/v1/live-schema/snapshot` | 发起元数据采集（异步任务），入参：连接、库列表、对象类型、脱敏开关 |
| GET | `/api/v1/live-schema/snapshots` | 快照列表（按实例/库/时间） |
| GET | `/api/v1/live-schema/snapshots/{id}` | 快照详情（对象计数、指纹、状态） |
| GET | `/api/v1/live-schema/snapshots/{id}/download` | 下载规范化 DDL 快照 `.sql` |
| POST | `/api/v1/live-schema/snapshots/{id}/audit` | 对快照跑规则审核（可指定规则集，默认 `live_schema` 档） |
| GET | `/api/v1/live-schema/snapshots/{id}/diff?base={id2}` | 两份快照结构漂移对比（F2） |

RBAC：新增菜单键 `live-schema`，归入"实例与体检"或"上线检查"分组；纳入 C5 权限矩阵测试。

### 3.7 数据模型（新增一张表）

`schema_snapshots`（快照台账）：`id / connection_id / database_name / object_types / object_counts(json) / sha256 / file_path / size_bytes / status / masked / created_by / created_at`。快照 DDL 正文落盘 `data/snapshots/`，表内只存元信息与指纹——大文件不入库，与现有"报告落盘"一致。

### 3.8 前端（现有单页内新增，不引入构建链）

在"实例与体检 / 上线检查"下新增页 **实况库审核**：选实例 + 库 + 对象类型 → 「采集真实元数据快照」→ 展示对象计数/指纹/耗时 →「下载快照」/「对快照执行规则审核」→ 违规报告（按对象归并、可导出 HTML）；再挂一个「与历史快照对比」入口（F2）。全部用现有 Vue3-CDN 单页 + Element Plus 组件实现，**不引入 Vite/Node**（与 2.2 保持一致）。

## 4. 兼容性与风险

| 维度 | 兼容/风险 | 处置 |
|---|---|---|
| 现有 API / 响应体 | 100% 不变（仅新增 `/live-schema/*`） | 新增路由，不改老路由 |
| 数据表 | 仅新增 `schema_snapshots` | 走现有 `ensure_db()` 建表路径，不引入迁移框架（呼应 C1 降级结论） |
| 规则本体 | 0 改动 | `live_schema` 档位用规则集覆盖实现 |
| 连接安全 | 只读 `SHOW CREATE`/`information_schema` | 全程只读；采集账号最小权限（SELECT + SHOW VIEW） |
| 敏感信息 | 视图/过程体可能含业务逻辑；快照文件落盘 | 提供脱敏开关（默认对过程/函数体可选脱敏）；快照文件纳入数据保留策略 |
| TDSQL 分布式限制 | 触发器/过程在分布式实例可能不可 `SHOW CREATE` | 逐对象 best-effort，失败标注"该实例不支持/无权限"，不中断整体 |
| 大库性能 | 逐对象 SHOW CREATE 慢 | 异步任务 + 分批 + 并发信号量 + 超时兜底 |
| RBAC | 新增子权限面 | 纳入 C5 矩阵测试，避免重蹈 BUG-01 |
