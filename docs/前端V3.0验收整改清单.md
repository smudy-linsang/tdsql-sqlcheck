## 一、一句话总评

设计说明书要求落地的 19 个页面里，**只有 8 个是真正实现的功能页，其余 11 个是零功能的 `el-empty` 空壳占位**——而这 11 个页面对应的后端接口**全部已经开发完成、随时可接**。提交信息号称"19 页面"，与实际严重不符。存量页面的骨架（TopBar / 分组导航 / 令牌 / Element Plus 组件化）搭得不错，但"全局实例切换器 / 项目切换器"这两个核心卖点基本是**摆设**（切换不刷新数据、项目列表根本没加载）。这就是"活干得粗"的根源：**搭了架子，没通电**。

### 🔴 P0 — 阻断级（验收一票否决）

#### P0-01 十一个页面是纯空壳 `el-empty` 占位，零功能；且后端接口全部已就绪

这是本次交付最核心的问题。以下页面**只有一个标题 + 一句"暂无数据"**，没有任何数据加载、没有表格、没有表单、没有交互。**关键是：它们的后端 API 早已实现并可用**，编码智能体只是没有接：

| 页面 | 位置 | 后端接口（已就绪，直接接） |
|---|---|---|
| 数据库体检 `health-check` | index.html:608–613 | `GET /api/v1/tdsql/check/charset`、`GET /api/v1/tdsql/check/large-tables` |
| 大表治理 `bigtable` | index.html:616–621 | `GET /api/v1/bigtable/inventory/{conn_id}`、`POST /inventory/{conn_id}`、`GET /report/{conn_id}`、`GET /classify/{table}` |

| 规则集 `rulesets` | index.html:629–631 | `GET/POST /api/v1/rulesets`、`GET/PUT/DELETE /rulesets/{id}` |
| 质量门禁 `gate` | index.html:634–636 | `GET /api/v1/quality-gate/rules/{project_id}`、`POST /rules`、`POST /strategy/{project_id}`、`GET /strategies` |
| 监控告警 `monitor` | index.html:639–641 | `GET /api/v1/monitor/alerts`、`POST /alerts/{id}/acknowledge`、`GET/POST /monitor/rules`、`POST /evaluate` |
| 巡检管理 `inspection` | index.html:644–646 | `GET/POST /api/v1/inspection/tasks`、`GET /tasks/{id}`、`POST /tasks/{id}/status`、`POST /tasks/{id}/results` |
| 扫描计划 `slow-schedule` | index.html:518–529 | `GET/POST /api/v1/tdsql/scan-schedules`、`PUT/DELETE /scan-schedules/{id}` |
| 数据保留 `sys-retention` | index.html:690 | `GET/PUT /api/v1/admin/retention`、`POST /retention/run` |
| 操作审计 `sys-auditlog` | index.html:692 | `GET /api/v1/admin/operation-logs` |
| 系统信息 `sys-info` | index.html:694 | `GET /api/v1/admin/info` |

**期望**：按设计说明书 §5.9–§5.13 及附录 A，每个页面落地真实的表格 / 表单 / 抽屉 / 详情，并联通上述接口。空态才用 `el-empty`，而不是把 `el-empty` 当"完工"。

**修复指引**：
- 扫描计划：`el-table` 列出计划（名称、cron 时刻、条数、启用开关），`el-switch` 加 `@change` 调 `PUT /scan-schedules/{id}`；"+ 新建计划"抽屉；页面 `watch` 里补 `loadScanSchedules()`。
- 数据库体检：选实例 → 触发 charset / large-tables 检查 → 分类展示结果卡片。
- 项目管理 / 规则集 / 门禁：标准 CRUD（表格 + 抽屉表单 + 二次确认）。
- 监控告警：告警列表 + 确认按钮 + 告警规则配置；同时驱动顶栏铃铛（见 P1-07）。
- 巡检管理：任务表格 + 新建巡检 + 结果抽屉。
- 数据保留 / 操作审计 / 系统信息：分别对接 retention / operation-logs / info 接口。

### 🟠 P1 — 严重（已"实现"部分里的真 Bug / 硬指标不达标）

#### P1-02 顶栏项目切换器永远是空的（项目门禁全链路失效）
- **位置**：`projects` 声明于 index.html:732；顶栏下拉 index.html:157–159；即时审核绑定项目 index.html:294–296。
- **现状**：全代码**没有任何 `loadProjects()`**，`loadAll()`（index.html:844）也不加载项目。`projects` 永远是空数组。
- **后果**：顶栏"选择项目"永远空白；即时审核 / 文件审核的"绑定项目"永远空白 → 规则集覆盖、项目门禁这条设计主线**完全无法触发**。
- **修复**：新增 `loadProjects()`（`GET /api/v1/projects`），在 `loadAll()` 与登录后调用。

#### P1-03 顶栏项目切换 `onProjectSwitch` 是空函数
- **位置**：index.html:805 `const onProjectSwitch=()=>{};`
- **现状**：切换项目不做任何事——不刷新数据、不改变审核上下文。
- **期望**（§9.2 第 5 条）：切换项目后，审核请求带 `project_id`，并展示该项目的规则集覆盖 + 门禁结果。
- **修复**：切换后刷新受项目影响的页面数据，并把 `currentProjectId` 贯穿到审核/列表请求。

#### P1-04 顶栏实例切换后，页面数据不刷新（陈旧数据）
- **位置**：index.html:804 `onConnectionSwitch`。
- **现状**：只调 `/connect` 建连并提示，**不重新加载** dashboard / 慢SQL列表 / 扫描任务。
- **后果**：切换实例后，屏幕上仍是上一个实例的数据，误导性极强。违反 §9.2 第 4 条。
- **修复**：连接成功后调用 `loadAll()`（或当前页对应的 loader）刷新。

#### P1-05 "全局实例上下文"名存实亡——列表/概览请求不带 connection_id
- **位置**：`loadDashboard`(806)、`loadSlowList`(815)、`loadScanTasks`(822) 均不传 `connection_id`。
- **现状**：只有"发起扫描"(826)、"EXPLAIN"(829) 用到了 `currentConnectionId`；概览、慢SQL列表、扫描任务列表都是全局不分实例。
- **后果**：所谓"全局实例切换器"对绝大多数页面只是装饰。这是设计核心卖点的落空。
- **修复**：明确各接口是否支持按实例过滤；支持的一律带 `connection_id`，与顶栏切换器联动。

#### P1-06 CRITICAL 级慢SQL 被渲染成绿色"成功"标签（严重误导）
- **位置**：`sevTagType` index.html:792：`s==='ERROR'?'danger':s==='WARNING'?'warning':s==='INFO'?'info':'success'`。
- **现状**：慢SQL 用的级别是 **CRITICAL / WARNING / INFO**（见筛选项 index.html:464），而 `sevTagType('CRITICAL')` 落入 `else` → 返回 `'success'`（**绿色**）。
- **后果**：慢SQL记录列表、慢SQL详情、概览"高频违规 TOP5" 中，**最严重的 CRITICAL 显示成绿色成功标签**，风险被视觉淡化。
- **修复**：`sevTagType` 增加 `CRITICAL/ERROR → 'danger'` 映射，覆盖两套级别体系。

#### P1-07 顶栏告警铃铛 `activeAlerts` 永远为 0
- **位置**：`activeAlerts` 声明 index.html:734；铃铛 index.html:162。
- **现状**：无任何 loader 拉 `/api/v1/monitor/alerts`，红点永远不亮。
- **修复**：随监控告警页一并接入，登录后/定时刷新活跃告警数。

#### P1-08 扫描计划 `el-switch` 无 `@change`，且页面数据不加载
- **位置**：index.html:526 `<el-switch :model-value="row.enabled">`（无 change 处理）；`scanSchedules` 声明 780、页面 518，但 `watch`(846) 切到 `slow-schedule` 不加载。
- **现状**：即便有数据也切不动启用状态；实际因不加载而永远空。后端 `/scan-schedules` CRUD 已就绪却完全未接。
- **修复**：见 P0-01 扫描计划条目。

#### P1-09 即时审核"绑定项目"复用了全局 `currentProjectId`
- **位置**：index.html:294 `<el-select v-model="currentProjectId" placeholder="绑定项目">`。
- **现状**：单次审核选项目会**污染顶栏的全局项目上下文**，两者应解耦。
- **修复**：即时审核用独立的局部变量（如 `auditProjectId`），或明确"以顶栏项目为准"并去掉页面内下拉。

#### P1-10 内联 `style=` 80 处，超验收上限（≤20）4 倍，且大量硬编码色值
- **位置**：全文；典型硬编码色值 index.html:152/153（`#16a34a`/`#94a3b8`）、576（`#16a34a`/`#94a3b8`）、808（图表 `#16a34a`/`#dc2626`）。
- **期望**（§9.2 第 2 条）：内联 style ≤ 20（仅动态计算），色值/间距全走 `var(--...)` 设计令牌。
- **修复**：把重复布局样式沉到 CSS 类；色值改用语义令牌（`var(--success-500)` / `var(--danger-500)` / `var(--text-secondary)`）；图表色也用令牌。

#### P1-11 零骨架屏 `el-skeleton`（设计要求首屏/整页骨架）
- **位置**：全站 0 处 `el-skeleton`，仅 6 处 `v-loading`。
- **期望**（§4.2/§9.2 第 7 条）：首屏 Dashboard、整页区块用 `el-skeleton` 骨架屏，结构贴近真实内容。
- **修复**：Dashboard KPI 卡 / 表格首屏用 `el-skeleton`；其余区块 `v-loading` 可保留。

#### P1-12 残留"加载中…"纯文字（且是永久假 loading）
- **位置**：index.html:694 系统信息页 `<el-empty description="加载中...">`。
- **现状**：既违反"消灭'加载中…'文字"，本质又是空壳页（永远停在"加载中"）。
- **修复**：随 P0-01 系统信息页接入真实数据后移除。

#### P1-13 无全局 403 / 5xx 兜底，且多处 `catch` 静默吞错
- **位置**：`apiFetch`(720) 只处理 401；`loadDashboard`(806)、`loadRules`(814)、`loadSlowList`(815) 等大量 `catch(e){}` 空吞。
- **期望**（§9.2 第 9 条）：401 跳登录（已做）、403 友好提示、5xx 通知兜底。
- **修复**：在 `apiFetch` 统一处理 403（ElMessage 友好提示）、5xx（通知兜底）；业务 `catch` 不再静默，至少给用户可见反馈。

## 四、整改优先级建议

1. **先补 P0-01 的 11 个空壳页**（后端全就绪，是纯前端接线活）——这是验收通过的前提。优先级：扫描计划、监控告警、项目管理、规则集、门禁（业务主线）> 数据库体检、大表治理、巡检 > 数据保留、操作审计、系统信息。
2. **再修 P1 的功能性 Bug**——尤其 P1-02/03/04（切换器通电）、P1-06（CRITICAL 变绿）这类"错得离谱"的，投入小、观感提升大。
3. **P1-10/11/12/13 硬指标**——令牌化、骨架屏、错误兜底，收尾达标。
4. **P2/P3** 随手清理（删调试脚本、拆代码、校 RBAC）。

## 五、复验方式

整改后请对照本清单第二节 11 项逐条自查，并确保：
- 以 admin / dba / developer / auditor 四个角色分别登录，走通附录 B 的可见性；
- 顶栏切实例后当前页数据确实变化；顶栏选项目后审核带上 `project_id`；
- 11 个原空壳页均能加载真实数据、完成各自核心操作；
- 复跑 `_test_file_report.py` 之类脚本前先移出仓库。
