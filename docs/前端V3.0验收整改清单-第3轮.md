## 二、仍需整改：2 个"写操作"接口契约 bug（照抄即可）

这两个都是同一类错误——**前端请求的格式和后端接口约定对不上**，导致点击后必然报 422 失败。和前几轮一样，根子还是**没有把这个操作真正点一遍**。

### 🔴 BUG-1：新建巡检必失败（`createInspection`，app.js 第 235 行）

- **现象**：巡检管理页点「+ 新建巡检」→ 填实例和类型 → 提交，必定失败（HTTP 422）。
- **原因**：后端 `POST /api/v1/inspection/tasks` 的签名是
  `async def create_task(connection_id: str, inspection_type: str)` —— 这两个是 **query 查询参数**（没有 Pydantic body 模型）。
  而前端发的是 **JSON body**、URL 上没带参数 → 后端认为必填的 query 参数缺失 → 422。
- **现在（错）**：
  ```js
  const resp=await apiFetch(`${API_BASE}/api/v1/inspection/tasks`,{method:'POST',
    headers:{'Content-Type':'application/json'},body:JSON.stringify(inspectionDialog.form)});
  ```
- **改成（对）**——把参数放到 query string：
  ```js
  const q=new URLSearchParams({connection_id:inspectionDialog.form.connection_id,
                               inspection_type:inspectionDialog.form.inspection_type});
  const resp=await apiFetch(`${API_BASE}/api/v1/inspection/tasks?${q}`,{method:'POST'});
  ```
  （或者反过来改后端：让 `create_task` 接收一个 body 模型。二选一，改前端更省事。）

### 🔴 BUG-2：扫描计划的「启用」开关一切就失败（`toggleScheduleEnabled`，app.js 第 205 行）

- **现象**：扫描计划页拨动「启用」`el-switch`，必定失败（HTTP 422），开关状态回弹。
- **原因**：后端 `PUT /api/v1/tdsql/scan-schedules/{id}` 要求一个**完整的 `ScanScheduleRequest`**，其中
  `connection_id` 是**必填**（`Field(...)`）。而前端只发了 `{enabled: 0/1}` → 缺 `connection_id` 等必填字段 → 422。
- **现在（错）**：
  ```js
  await apiFetch(`${API_BASE}/api/v1/tdsql/scan-schedules/${row.id}`,{method:'PUT',
    headers:{'Content-Type':'application/json'},body:JSON.stringify({enabled:row.enabled?0:1})});
  ```
- **改成（对）**——把整行字段一起回传，只翻转 enabled：
  ```js
  await apiFetch(`${API_BASE}/api/v1/tdsql/scan-schedules/${row.id}`,{method:'PUT',
    headers:{'Content-Type':'application/json'},body:JSON.stringify({
      connection_id:row.connection_id, source:row.source||'digest',
      cron_hour:row.cron_hour, cron_minute:row.cron_minute,
      limit_rows:row.limit_rows, min_time:row.min_time,
      enabled:!row.enabled
    })});
  ```
